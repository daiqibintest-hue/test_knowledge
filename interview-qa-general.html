第一部分 测试基础与流程

共 16 题

#1

1. 一个需求从评审到上线，你完整的测试流程是什么？

答

我把流程拆成 7 步：

  • ①需求评审（澄清边界、识别风险点）→
  • ②测试方案与计划（人力、时间、风险、准入准出标准）→
  • ③用例设计与评审（思维导图 + Xmind/TestLink，覆盖功能/接口/异常/性能/兼容）→
  • ④测试环境与数据准备（造数脚本、Mock 第三方）→
  • ⑤执行（冒烟→功能→接口自动化→回归→专项）→
  • ⑥缺陷管理（Jira 跟踪，每日站会推进）→
  • ⑦上线评估与报告（质量门禁、上线公告、灰度监控）。上线后还会做 24-48 小时的线上巡检与回归。
#2

2. 需求评审阶段，测试应该关注哪些风险？

答

六个维度：

  • 需求完整性：是否有边界条件、异常分支、空值/极值描述；
  • 业务闭环：上下游链路是否打通（比如工单状态变更是否同步账单、业务文档、通知模块）；
  • 技术可行性：接口契约是否定义、数据库表结构是否变更、是否依赖第三方；
  • 可测性：日志、监控、Mock、测试账号是否具备；
  • 影响范围：是否涉及历史数据迁移、是否影响存量用户；
  • 合规与安全：AI 产品类项目尤其关注内容合规、个人信息脱敏、权限隔离。
#3

3. 你怎么判断一个需求是否可以进入测试？

答

我们有明确的"测试准入标准"（DoR）：

  • ①PRD 终版冻结且评审通过
  • ②UI/接口文档齐全
  • ③研发提供冒烟报告，冒烟用例 100% 通过
  • ④测试环境部署完成且可访问
  • ⑤测试数据、依赖服务准备完毕
  • ⑥提测邮件包含变更范围、影响模块、回归建议。任何一项不达标，我会发"打回单"并同步项目经理。
#4

4. 什么情况下你会阻塞上线？

答

四类硬性阻塞：

  • ①P0/P1 缺陷未关闭（如金额错误、数据泄露、AI 输出违法内容）
  • ②冒烟用例不通过
  • ③性能不达标（响应时间、错误率、TPS 突破基线）
  • ④高风险变更未灰度。软性阻塞需评审：自动化通过率<90%、未达到代码覆盖率门禁等。作为负责人我有一票否决权，但会用数据和风险评估而不是个人判断。
#5

5. 测试计划一般包含哪些内容？

答

9 个要素：项目背景与目标、测试范围与不测范围、测试策略（手工/自动化/性能/AI 评估）、资源与排期（人力分配、里程碑）、风险识别与应对、测试环境与数据、准入准出标准、交付物清单（报告、用例、自动化脚本）、沟通机制。我会用一页 A3 的形式输出，让产品/研发/老板一眼看完。

#6

6. 测试报告一般包含哪些内容？

答

报告必含 8 项：版本信息、测试范围、用例执行情况（通过/失败/阻塞/跳过）、缺陷统计（按严重等级、模块、状态）、未关闭缺陷清单及风险评估、性能/AI 专项结果、遗留问题与建议、上线结论（同意/有条件同意/不同意）。AI 项目额外加：准确率、召回率、幻觉率、Prompt Injection 通过率等指标。

#7

7. 缺陷生命周期是什么？

答

New → Assigned → Open → Fixed → Verified → Closed。中间分支：Reopen（验证不通过）、Rejected（研发认定不是 Bug，需评审）、Deferred（延期处理，需产品确认）、Duplicate（重复）。每个状态都有 SLA：P0 当天关闭、P1 三天内、P2 一周内。

#8

8. 严重程度和优先级有什么区别？

答
  • 严重程度（Severity）：技术维度，描述 Bug 对系统的破坏力，分 Blocker/Critical/Major/Minor/Trivial；
  • 优先级（Priority）：业务维度，描述修复的紧迫性，分 P0/P1/P2/P3。

两者可能不一致：比如登录按钮文案错别字是 Trivial 但若是产品发布会用，优先级会被定为 P0；后台日志拼写错误是 Major 但优先级 P3。

9. 如果研发认为不是 Bug，你怎么处理？

答

四步：

  • ①拿出证据——复现步骤、截图、抓包、日志、相关需求条款
  • ②回到需求文档对比预期
  • ③拉产品三方对齐，让产品判定是不是预期行为
  • ④如果产品也认为是预期但用户体验差，我会作为"建议项"留档，下个版本评审。绝不和研发个人对抗，而是让规则说话。
#10

10. 如果产品临时改需求，你怎么保证质量？

答

首先评估变更范围（哪些用例失效、哪些新增、哪些回归需要重跑）→ 提交"变更影响评估单"，量化对测试工期的冲击 → 与产品/研发对齐：要么延期、要么砍范围、要么承担风险上线 → 如果必须上线，按风险优先级压缩用例（只跑核心路径 + 影响接口的自动化回归）→ 上线后加强监控。我会把决策邮件留档保护团队。

#11

11. RAG 问答类功能怎么设计测试用例？

答

分 5 层设计：

  1. 文档层：解析正确性（PDF/Word/扫描件）、切分效果（Chunk 大小、重叠）；
  2. 检索层：召回准确率（Top1/Top3/Top5）、关键词与向量混合检索效果；
  3. 生成层：答案忠实性、是否基于引用、是否产生幻觉；
  4. 业务层：业务场景准确性（规范条款引用、地区差异、新旧规范）；
  5. 异常层：知识库外问题、模糊/超长/恶意 Prompt、多轮上下文。

每层都有 Golden Dataset（约 500-1000 条标注样本）做回归。

#12

12. AI 分类模块怎么设计测试用例？

答
  • ①构建分层测试集：每个类别至少 100 条样本，正负样本 1:1
  • ②设计混淆矩阵评估：TP/FP/FN/TN
  • ③单独看少数类的召回率（防止"准确率高但小类全错"）
  • ④边界用例：歧义通知、双类别、空内容、表情符号
  • ⑤对抗用例：错别字、谐音、变体
  • ⑥版本回归：模型/规则更新后对比 F1 是否下降。
#13

13. 怎么避免用例重复和遗漏？

答

两套机制：

  • 避免遗漏：用需求-用例追溯矩阵（RTM），每个需求条款必须对应至少一条用例；用思维导图按"功能/接口/异常/兼容/性能/安全"六维拉网；
  • 避免重复：用例库统一管理（TestLink/禅道），评审时按"前置条件+操作步骤+断言"做去重；自动化用例做参数化，把数据驱动和用例分离。

按"模块 → 场景 → 优先级"三级目录组织。比如工单管理 / 工单创建 / P0_正常流程、工单管理 / 工单创建 / P1_异常分支。同时打标签：@smoke @regression @ai @performance @security。Pytest 用 mark，TestLink 用 Suite + Tag。每个版本按标签筛选执行。

15. 1000+ 用例中，功能、接口、AI、性能分别占多少？

大致比例：功能用例 50%（约 500 条，含 UI 和业务流程）、接口用例 30%（约 300 条，覆盖 100+ 核心接口）、AI 评估用例 15%（约 150 条，含 RAG Golden Set、分类测试集、业务文档校验集）、性能场景 5%（约 50 条，含基线、容量、稳定性）。

16. 如果一个需求对应多个接口和多个页面，你怎么保证用例覆盖完整？

答

画"需求-接口-页面"三维矩阵：横轴接口、纵轴页面，单元格放对应用例编号。任何空白单元格都要解释——是不需要覆盖还是漏了。再加业务流程图，把端到端链路标注出来，确保每条链路至少一条 E2E 用例。

#17

1. GET 和 POST 的区别是什么？

答
  • 语义：GET 幂等只读，POST 非幂等可能改服务端状态；
  • 参数位置：GET 放 URL，POST 放 Body；
  • 长度限制：GET 受 URL 长度限制（约 2KB），POST 无；
  • 缓存：GET 可缓存，POST 默认不缓存；
  • 安全：GET 参数会出现在日志和浏览器历史，敏感数据用 POST。

但 RESTful 设计中关键是语义而非技术差异。

#18

2. PUT 和 PATCH 有什么区别？

答
  • PUT：全量替换资源，请求体包含资源完整字段，幂等；
  • PATCH：增量更新，只传变更字段，非严格幂等（如增加计数器）。

比如更新工单状态，如果只改 status 字段用 PATCH；如果整个工单对象替换用 PUT。

#19

3. HTTP 状态码 200、201、204、301、302、400、401、403、404、409、500 分别代表什么？

答
  • 2xx 成功：200 OK、201 Created（创建成功）、204 No Content（成功无返回体）；
  • 3xx 重定向：301 永久重定向、302 临时重定向；
  • 4xx 客户端错误：400 参数错误、401 未认证（Token 缺失/过期）、403 已认证但无权限、404 资源不存在、409 冲突（如重复提交、版本冲突）；
  • 5xx 服务端错误：500 内部错误。

测试时要分清 401 和 403 的差异，越权问题往往体现在这里。

#20

4. Cookie、Session、Token、JWT 有什么区别？

答
  • Cookie：浏览器存储，由服务端 Set-Cookie 下发，自动随请求携带；
  • Session：服务端存储会话状态，依赖 Cookie 传递 sessionId；
  • Token：泛指身份令牌，无状态，客户端主动放入 Header；
  • JWT：Token 的一种实现，由 Header.Payload.Signature 三段组成，自包含用户信息，可被服务端无状态校验。

我们项目用 JWT + Redis 黑名单，既享受无状态又能主动注销。

#21

5. 接口鉴权一般有哪些方式？

答

五种主流方式：

  • ①Basic Auth（明文 base64，仅用于内部）
  • ②API Key（简单但不安全）
  • ③Session-Cookie（传统 Web）
  • ④Token/JWT（前后端分离主流）
  • ⑤OAuth2.0（第三方授权）。我们系统对外用 JWT，对内服务用 mTLS + API Key。
#22

6. 什么是接口幂等性？

答

同一请求执行 N 次的结果与执行 1 次相同。实现方式：

  • ①唯一索引（数据库层面）
  • ②业务幂等键（如 requestId）
  • ③乐观锁（版本号）
  • ④状态机校验。测试要点：重复提交、网络重试、超时重试、并发提交。
#23

7. 如何测试重复提交？

答

四种场景：

  • ①点击两次按钮（前端防抖测试）
  • ②用 JMeter 1 秒内发 10 个相同请求
  • ③模拟弱网下用户重试
  • ④用 Postman 重放抓包。验证维度：数据库只插入一条、返回结果一致、不产生脏数据、不重复扣费/发通知。
#24

8. 如何测试接口越权？

答

两类：

  • 水平越权：A 用户的 Token 访问 B 用户的资源。比如业务人员 A 用自己 Token 访问 GET /orders/{B 业务人员的工单 ID}，预期 403；
  • 垂直越权：低权限用户访问高权限接口。比如运营专员用 Token 访问 DELETE /admin/users/{id}，预期 403。

测试时遍历所有"含 ID 的接口"，用低权限 Token 替换 URL/Body 中的 ID 进行扫描。

#25

9. 如何测试接口限流？

答
  • ①根据限流规则（QPS/分钟/小时）设计 JMeter 场景：刚好达到阈值、超过阈值、远超阈值
  • ②验证：是否返回 429、限流 Header（X-RateLimit-Remaining）是否正确、不同用户/IP 是否独立计数、冷却时间后是否恢复
  • ③边界：刚跨秒、令牌桶是否平滑。
#26

10. 如何测试接口超时和重试？

答
  • ①用 Charles 的 Throttle Settings 延迟响应模拟服务端慢
  • ②用 JMeter 的 Timeout 设置模拟客户端超时
  • ③验证客户端是否按规则重试（次数、间隔、退避）
  • ④验证重试是否幂等不重复落库
  • ⑤验证最终失败时的兜底（toast 提示、降级数据）。
#27

11. 一个新增工单接口，你会设计哪些测试点？

答

六维设计：

  • 功能正向：必填字段、字段类型、枚举值、关联校验（业务人员/客户必须存在）；
  • 异常分支：必填缺失、字段超长、特殊字符、SQL 注入；
  • 业务规则：工单编号唯一性、状态初始值、自动分配业务人员逻辑；
  • 数据落库：主表 + 关联表 + 流水表都要校验；
  • 联动效果：是否触发通知、是否生成默认业务文档、是否记录操作日志；
  • 权限/幂等：越权创建、重复 requestId 提交。
#28

12. 一个工单状态流转接口，你会设计哪些异常场景？

答
  • ①状态机非法跳转（如"已关闭"跳"新建"）
  • ②并发流转（两个业务人员同时改）——验证乐观锁
  • ③前置条件不满足（无账单记录不能关闭）
  • ④回滚后状态正确
  • ⑤流转后下游联动（通知、业务文档、报表）是否正确
  • ⑥流转日志是否记录操作人和时间。
#29

13. 一个账单结算接口，你会怎么校验计算正确性？

答

四道关：

  • ①写 SQL 把原始金额、计费金额、调整金额、服务费分别查出来，按业务公式手工算一遍对比
  • ②边界值：金额为 0、负数、超大数（防溢出）、小数位（防精度丢失，强制 BigDecimal）
  • ③多笔合计：N 笔账单相加是否等于汇总
  • ④并发结算下账目不重不漏。我自己写过 SQL 对账脚本辅助。
#30

14. 一个通知发送接口，你会怎么验证通知是否真的发送？

答

四层验证：

  • ①接口返回成功 + 通知流水表落库
  • ②通知平台后台查询投递状态
  • ③真实手机收到内容、签名、链接（变量替换正确）
  • ④异步场景看消息队列是否消费、失败是否进死信队列。我们内部有"通知测试账号池"，可自动拉取最近收到的通知做内容断言。
#31

15. 一个 AI 问答接口，你会怎么测试返回结果？

答

分四种断言：

  • 结构断言：返回字段完整（answer、references、confidence）；
  • 业务断言：基于 Golden Dataset，命中关键概念词、引用规范条款编号是否正确；
  • 质量断言：用另一个 LLM 做评判（LLM-as-Judge），打分 4-5 分通过；
  • 安全断言：不包含敏感词、不输出违法建议、不泄露其他用户信息。

非确定性输出我用多次采样 + 一致性评估。

#32

16. 接口依赖第三方服务时，如何测试？

答

分两条线：

  • ①真实联调（覆盖正常路径和上线前回归）
  • ②Mock 服务（用 WireMock/Hoverfly，覆盖第三方异常：超时、500、限流、错误格式）。我会维护一个"第三方异常场景集"，作为接口健壮性测试的固定回归。
#33

17. 上游接口返回异常时，下游如何验证？

答

通过 Mock 上游构造异常响应（4xx、5xx、超时、错误格式、null、空数组），验证下游：

  • ①不崩溃
  • ②给出友好错误提示
  • ③不脏数据
  • ④有重试或降级策略
  • ⑤日志记录了 traceId 和上游响应。
#34

18. 接口返回字段很多时，你怎么做断言？

答

分层断言：

  • ①状态码 + 关键字段强断言
  • ②结构断言（JSONSchema 校验）

#47

1. 常用 SQL 查询有哪些？

答

日常使用 SELECT、INSERT、UPDATE、DELETE、JOIN（INNER/LEFT/RIGHT）、GROUP BY、HAVING、ORDER BY、LIMIT、子查询、UNION、CASE WHEN、聚合函数（SUM/COUNT/AVG/MAX/MIN）、窗口函数（ROW_NUMBER/RANK）。测试常用是 JOIN 多表对账 + 聚合函数对总数。

#48

2. inner join、left join、right join 有什么区别？

答
  • INNER JOIN：两表都匹配才返回；
  • LEFT JOIN：左表全保留，右表无匹配则字段为 NULL；
  • RIGHT JOIN：右表全保留。

测试时常用 LEFT JOIN 找"主表有但关联表没有"的脏数据。

#49

3. group by 和 having 怎么用？

答

GROUP BY 按字段分组聚合，HAVING 对分组后的结果过滤（WHERE 不能用聚合函数）。例：找重复工单号 SELECT order_no, COUNT(*) FROM orders GROUP BY order_no HAVING COUNT(*) > 1。

#50

4. order by 和 limit 怎么用？

答

ORDER BY field DESC/ASC 排序，LIMIT offset, count 分页。注意：分页排序字段不唯一会出现"翻页重复"，必须加唯一辅助字段（如 id）。

#51

5. count(*)、count(1)、count(字段) 有什么区别？

答
  • count(\*)：统计所有行（含 NULL）；
  • count(1)：等价于 count(\*)，性能在 InnoDB 中几乎无差异；
  • count(字段)：统计该字段非 NULL 的行数。

测试覆盖率统计要选对，不然漏算。

#52

6. 索引是什么？

答

索引是加速查询的数据结构，MySQL InnoDB 默认 B+ 树。聚簇索引（主键）数据和索引在一起，二级索引存的是主键值需要回表。代价是写性能下降和占空间。我会用 EXPLAIN 看是否走索引。

#53

7. 什么情况下索引会失效？

答

经典 7 种：

  • ①左模糊 like '%x'
  • ②字段做函数运算或类型转换
  • ③or 连接非索引字段
  • ④复合索引违反最左前缀
  • ⑤ != / not in / not exists
  • ⑥数据分布不均（MySQL 优化器判断全表扫更快）
  • ⑦字符集不一致。
#54

8. 事务的 ACID 是什么？

答
  • A 原子性：要么全成功要么全回滚；
  • C 一致性：事务前后数据满足业务规则；
  • I 隔离性：并发事务互不干扰；
  • D 持久性：提交后即使宕机也不丢。
#55

9. 脏读、不可重复读、幻读是什么？

答
  • 脏读：读到其他事务未提交的数据；
  • 不可重复读：同一事务两次读同一行结果不同（其他事务 UPDATE 了）；
  • 幻读：同一事务两次范围查询结果数不同（其他事务 INSERT/DELETE 了）。
#56

10. MySQL 隔离级别有哪些？

答

RU（读未提交，脏读全有）、RC（读已提交，解决脏读）、RR（可重复读，解决不可重复读，MySQL 默认通过 MVCC 也基本解决幻读）、Serializable（串行化，全解决但性能差）。

2. 为什么选择 Python + Requests + Pytest？

答
  • ①Python 生态丰富，团队上手快

22. yield fixture 一般用于什么场景？

答

yield 前是 setup（创建测试数据），yield 后是 teardown（清理）。比如：

    order_id = create_case()
    yield order_id
    delete_case(order_id)

保证每个用例独立干净。

3. 并发用户数、TPS、QPS、吞吐量、响应时间分别是什么？

答
  • 并发用户数：同时活动的用户数；
  • TPS：每秒事务数，业务粒度；
  • QPS：每秒查询数，接口粒度；
  • 吞吐量：单位时间处理的请求数（含 TPS/QPS）；
  • 响应时间：单次请求从发出到收到响应的时间。

4. 平均响应时间和 90 线、95 线、99 线有什么区别？

10. 如何判断系统性能瓶颈？

19. 准确率、召回率、精确率、F1 值分别是什么？

答

基于 TP/FP/FN/TN：

  • Accuracy 准确率 = (TP+TN)/(TP+TN+FP+FN)：整体对的比例；
  • Precision 精确率 = TP/(TP+FP)：预测为正中真正的比例；
  • Recall 召回率 = TP/(TP+FN)：真正为正中被预测出的比例；
  • F1 = 2\*P\*R/(P+R)：精确和召回的调和平均。

  • FN：是运营触达，模型说不是（漏报）；
  • TN：不是运营触达，模型说不是。

  • ②新模型跑一遍记录所有响应
  • ③老版本基线对比，输出 diff 报告（哪些用例从对变错、错变对、对变对但内容变化）
  • ④分析退化用例，决定是否上线
  • ⑤超过 5% 退化的版本不允许上线。
#243

28. 如何做提示词版本回归？

答

和模型回归类似。Prompt 进版本管理（Git 或专门的 Prompt 仓库），每次变更 Golden Set 全量回归，对比通过率和质量分。重要变更要灰度（A/B 测试 5% 流量）。

#244

29. 你们是否有标准答案集？

答

有 RAG Golden Set 500 条知识库问答，每条包含：

  • ①question
  • ②expected_answer（标准答案文本）
  • ③key_points（必须命中的事实点）
  • ④expected_law_refs（必须引用的规范条款）
  • ⑤forbidden_content（不能出现的内容）。
#245

30. 一条业务问题可能有多个合理答案，怎么评估？

答

不追求精确匹配，而是看：

  • ①关键事实点覆盖率（如费用计算的"基础费+服务费"两层都要提）
  • ②是否引用正确规范条款编号
  • ③是否避免明确错误（错引规范条款/编造数字）。多答案合理时只要满足以上即算通过。
#246

31. 问答准确率达标的阈值是多少？

答

核心场景（劳动、合同、债务）≥ 90%，普通场景 ≥ 85%，长尾场景 ≥ 75%。上线门槛按版本设定，每个版本至少不退化 5%。

#247

32. 准确率低时，你怎么判断是检索问题还是生成问题？

答

看链路日志：

  • ①如果检索片段中没有答案需要的事实——是检索问题（Chunk 切分、Embedding、TopK）
  • ②如果检索片段中有正确事实但答案错——是生成问题（Prompt 不严谨、模型能力、温度过高）。两类问题完全不同的修复策略。
#248

33. 有没有区分"检索失败"和"生成错误"？

答

有。日志记录 query、召回片段、最终答案三段，回归时按 Context Recall 是否 ≥ 0.8 区分：召回不足→检索问题；召回够但答案错→生成问题。我们统计两类比例分别归不同小组优化。

#249

34. 用户输入模糊问题，AI 怎么测试？

答
  • ①构造模糊 query（"这个工单 8 年前的问题现在还能申诉吗"——缺主体、业务类型、地区）
  • ②看 AI 是否反问澄清
  • ③或给出多种可能解读的回答
  • ④不能编造完整事实回答（这是幻觉）。
#250

35. 用户输入错别字，AI 怎么测试？

答

构造错别字测试集（一字、谐音、形近、漏字），如"报销单据被驳会"（驳回→驳会）。预期 AI 仍能理解并答对，但 robustness 下降在 5% 内可接受。

#251

36. 用户输入超长文本，AI 怎么测试？

答
  • ①构造 1k、5k、10k、超 context 上限的 query
  • ②超出上限时 AI 是否优雅降级（截断 + 提示）而不是报错或乱回答
  • ③长文档摘要场景要看是否完整覆盖关键点。
#252

37. 用户输入恶意 prompt，AI 怎么测试？

答

Prompt Injection 测试集 100+ 条：

  • ①"忽略以上规则"
  • ②角色扮演越狱（"你现在是 DAN"）
  • ③Unicode 混淆
  • ④嵌入 SQL/代码
  • ⑤多轮诱导。预期：拒绝执行 + 不暴露系统 prompt + 不输出违法内容。
#253

38. 用户让 AI 输出违法内容，怎么测试？

答

构造违法引导（"教我怎么逃避业务机构执行"、"帮我伪造离婚证明"），AI 必须明确拒绝并给出合规提示。同时测试委婉引导（"假设有人想..."、"在小说中..."）也要拒绝。

#254

39. 用户问知识库外的问题，AI 应该怎么回答？

答

应该明确说"我不知道"或"知识库未覆盖"，而不是强行编造。测试时构造知识库外的问题（如其他行业规范、最新业务规则），预期看到拒识响应。这是反幻觉的关键设计。

#255

40. 用户要求 AI 编造规范条款，怎么测试？

答

构造引导（"请给我一段最新的《平台规范》第 9999 条内容"），AI 必须拒绝编造而非顺从。我们维护一个真实规范条款编号库做交叉验证，输出中出现不存在的规范条款编号即记为幻觉。

#256

41. AI 回答中引用不存在的规范条款，怎么发现？

答
  • ①爬取企业规范库建立规范条款编号库
  • ②AI 输出后用正则提取所有"《xxx规范》第 N 条"
  • ③对照规范知识库校验是否存在
  • ④不存在则标红告警。我们这套校验把幻觉率从 8% 降到 1% 以下。
#257

42. AI 生成业务文档中金额、姓名、单据编号错误，怎么测试？

答
  • ①取数据库结构化字段作为 Ground Truth
  • ②生成业务文档后用正则/解析提取出对应字段
  • ③一一对比，任何不一致即报错。关键字段（金额、单据编号）一定不能让模型自由生成，必须模板填充。
#258

43. AI 多轮对话上下文丢失，怎么测试？

答
  • ①设计多轮场景测试集（5+ 轮，每轮基于前面）
  • ②在第 N 轮提一个只能通过前 N-1 轮信息推出的问题
  • ③看回答是否正确利用历史
  • ④超过 context window 时是否合理总结而非粗暴截断。
#259

44. 你有没有做过 Prompt Injection 测试？

答

做过。维护了 100+ 条 Injection 测试集，每次模型/Prompt 升级都跑。覆盖：直接覆盖系统 prompt、角色扮演越狱、编码混淆、间接注入（在文档中藏指令）、多轮诱导。通过率 > 95% 才允许上线。

#260

45. 有没有做过越狱测试？

答

做过，包括 DAN/Developer Mode/老奶奶漏洞等经典越狱手法。我们定期跟进社区最新越狱手法补充测试集。同时测试越狱后是否影响其他用户（黑客越狱后是否会泄露其他人业务单据信息）。

#261

46. 有没有测试过"忽略以上所有规则"这种输入？

答

测过，是 Prompt Injection 测试集的基础项。预期：

  • ①AI 不响应该指令
  • ②仍按原 prompt 工作
  • ③可以在日志中标记可疑请求做风控。
#262

47. 企业知识库问答是否有安全边界？

答

有：

  • ①不给具体专业建议（用"建议咨询人工客服/业务专家"兜底）
  • ②不输出违法操作教程
  • ③不引用不存在规范条款
  • ④不泄露其他业务单据信息
  • ⑤敏感问题（自杀、伤害）给出心理援助提示。
#263

48. 如果 AI 给出错误专业建议，产品如何兜底？

答

四层兜底：

  • ①Prompt 强制要求"建议咨询业务专家"
  • ②规范条款引用必须从知识库取
  • ③重要问题强制人工审核
  • ④免责声明常驻
  • ⑤上线后投诉/反馈快速通道，错误案例进入修复 Backlog。
#264

49. 什么是 RAG？

答

Retrieval-Augmented Generation，检索增强生成。流程：用户问题 → 向量化 → 在知识库检索相关片段 → 拼入 Prompt → 大模型基于上下文生成答案。优势：

  • ①减少幻觉
  • ②知识可更新
  • ③可解释（能给出引用）。
#265

50. RAG 的完整流程是什么？

答

8 个环节：

  • ①文档解析（PDF/Word→文本）
  • ②文档切分（Chunk）
  • ③Embedding 向量化
  • ④存入向量数据库
  • ⑤用户 Query 向量化
  • ⑥向量检索 Top-K
  • ⑦（可选）Rerank 重排
  • ⑧拼入 Prompt 让 LLM 生成。每个环节都有独立测试点。
#266

51. RAG 为什么能减少幻觉？

答

让模型基于外部检索的真实文档作答而非凭记忆生成。当 Prompt 明确指示"只基于以下内容回答"时，模型编造空间减少。但减少不等于消除。

#267

52. RAG 一定能解决幻觉吗？

答

不能完全。剩余幻觉来源：

  • ①检索失败时模型仍编造（兜底缺失）
  • ②检索片段不全模型自行补充
  • ③模型未严格遵守"基于上下文"指令
  • ④Prompt 设计不严。需要工程化兜底（拒识、引用校验）。
#268

53. Embedding 是什么？

答

把文本映射到高维向量空间的过程，使语义相近的文本向量距离也近。常见模型：OpenAI text-embedding-3、bge-large-zh、m3e-large。维度 768-1536 居多。质量直接决定检索召回。

#269

54. 向量数据库的作用是什么？

答

存储文档向量并支持高效相似度检索（余弦/内积/欧氏）。主流：Milvus、Pinecone、Qdrant、Weaviate、PG vector。核心能力：高维向量索引（HNSW/IVF）、亿级扩展性、混合检索。

#270

55. Chunk 是什么？

答

文档切分后的小块，作为检索的基本单位。大小常见 200-800 token。切分策略：固定长度、按句子/段落、按语义、按结构（如规范条款按"第 X 条"）。Chunk 大小直接影响召回质量。

#271

56. TopK 是什么？

答

检索时返回相似度最高的 K 个 Chunk。常见 K=3、5、10。K 太大引入噪音和成本，K 太小可能漏召。需要结合业务实验。

#272

57. Rerank 是什么？

答

重排序。先用快速 Embedding 召回 Top-N（如 N=20），再用更精的 Cross-Encoder 模型重排取 Top-K（如 K=3）。两阶段架构平衡效率和精度。常用 bge-reranker、Cohere Rerank。

#273

58. 混合检索是什么？

答

向量检索 + 关键词检索（BM25）结合。向量检索擅长语义但可能漏精确关键词（如规范条款编号），关键词检索补足。两路结果用 RRF 或加权融合。业务场景特别有效。

#274

59. 你们企业知识库问答是怎么接入 RAG 的？

答

架构：

  • ①知识库（产品手册、业务 SOP、服务协议、内部 FAQ + 历史工单）
  • ②文档解析（PDF + OCR）
  • ③切分（按规范条款结构，每条一个 Chunk）
  • ④Embedding（bge-large-zh）
  • ⑤向量库（Milvus）
  • ⑥检索（向量 + 关键词混合）
  • ⑦Rerank
  • ⑧大模型生成。
#275

60. 知识库来源是什么？

答

三大类：

  • ①官方/企业规范（官网或内部规范库同步，定期更新）
  • ②行业标准和最佳实践（由业务专家维护）
  • ③内部知识（公司工单库、业务人员手册、业务 FAQ）。
#276

61. 文档是规范条款、案例、内部制度，还是业务 FAQ？

答

全都有：

  • ①规范条款 70%
  • ②案例 15%
  • ③内部 FAQ 10%
  • ④业务人员手册 5%。不同类型用不同切分策略：规范条款按条切，案例按背景/处理过程/规范条款三段切，FAQ 按 Q&A 对切。
#277

62. 文档如何切分？

答

按文档类型差异化：

  • ①规范条款按"第 X 条"为最小单元（不可拆分）
  • ②案例按段落 + 句子边界，500 token 一段
  • ③FAQ 一个 Q&A 一个 chunk。所有 chunk 保留 50-100 token 重叠避免上下文断裂。
#278

63. Chunk 大小是多少？

答

默认 500 token。规范条款特殊处理（按条目，平均 200 token）。FAQ 视长度（短 100、长 500）。Chunk 大小经过 A/B 测试：试过 200/500/800/1200，500 在业务场景效果最好。

#279

64. 有没有重叠？

答

有，50-100 token 重叠（约 10-20% 重叠率）。规范条款之间不重叠（每条独立完整）。重叠避免在句子中间切断导致语义丢失。

#280

65. 用的什么向量数据库？

答

Milvus 2.x。原因：

  • ①开源可控
  • ②性能好（HNSW 索引 100ms 内返回）
  • ③支持标量过滤（按文档类型、权限标签筛选）
  • ④活跃社区。
#281

66. 用的什么 Embedding 模型？

答

BAAI/bge-large-zh-v1.5。原因：

  • ①中文场景表现优秀
  • ②开源可本地部署（数据合规）
  • ③1024 维平衡精度和成本。对比过 text-embedding-3（API 依赖）、m3e、acge，bge 综合最优。
#282

67. TopK 设置多少？

答

两阶段：

  • ①初筛 Top-20（向量+关键词混合）
  • ②Rerank 后 Top-5 进 Prompt。Top-5 是经过实验的最优值：Top-3 偶有漏召，Top-10 引入过多噪音。
#283

68. 有没有 Rerank？

答

有，用 bge-reranker-large。流程：初筛 Top-20 → Cross-Encoder 打分 → Top-5。Rerank 把 Context Precision 从 0.65 提到 0.85。

#284

69. RAG 测试应该测哪些层？

答

5 层：

  • ①文档解析层（PDF→文本完整性、OCR 准确性）
  • ②切分层（Chunk 边界、语义完整性）
  • ③检索层（向量召回、关键词召回、Rerank）
  • ④生成层（基于上下文、不编造、引用正确）
  • ⑤端到端业务层（典型用户问题答得对）。
#285

70. 文档解析怎么测试？

答
  • ①标准文档（结构化 PDF）解析准确率 100%
  • ②扫描件 OCR 准确率（汉字 ≥ 99%，数字 100%）
  • ③表格解析（行列保持）
  • ④公式/特殊符号
  • ⑤页眉页脚去除。准备多种格式的对照样本逐一比对。
#286

71. 文档切分怎么测试？

答
  • ①切分后总字数 ≈ 原文（允许 1% 偏差，因为重叠会增加）
  • ②规范条款不被切断（每条完整在一个 chunk）
  • ③句子边界（不能从一句话中间切）
  • ④元数据保留（chunk 关联到原文页码、章节）。
#287

72. 向量检索怎么测试？

答
  • ①构建 Query-Doc 配对集（query 100 条，每条标注应召回的标准 chunk）
  • ②跑检索看 Top-K 命中率
  • ③语义相似 query 命中相同 chunk（一致性）
  • ④负样本 query 不命中无关 chunk。
#288

73. 关键词检索怎么测试？

答
  • ①精确编号匹配（规范条款号、单据编号必命中）
  • ②同义词扩展是否生效（如"赔偿"和"补偿"）
  • ③停用词不影响
  • ④分词正确（专业术语不被错分）。
#289

74. 混合检索怎么测试？

答
  • ①单独向量、单独关键词、混合三种方式跑同一测试集对比 Recall@K
  • ②权重调整对结果影响
  • ③一个有数字的 query（如"平台规范第 1245 条"）混合检索是否优于单纯向量。
#290

75. Rerank 效果怎么测试？

答
  • ①Rerank 前后 Top-5 的 Context Precision 对比
  • ②Rerank 是否提升答案质量（关联端到端测试）
  • ③Rerank 耗时是否在 SLA 内（一般 100-300ms）
  • ④Rerank 模型版本升级前后回归。
#291

76. 召回结果正确，但答案错误，怎么定位？

答

是生成端问题。可能：

  • ①Prompt 没有约束"只基于上下文"
  • ②模型温度太高发散
  • ③模型能力不足理解上下文
  • ④Prompt 中上下文位置不对（被指令淹没）。对策：优化 Prompt、降温度、换模型、加 Few-shot。
#292

77. 召回结果错误，但答案看起来合理，怎么定位？

答

是检索失败 + 模型幻觉的双重问题。模型靠记忆补全了答案，看起来合理但不可靠。这种最危险。修复：

  • ①优化检索
  • ②强化"只基于上下文回答"指令
  • ③答案中带引用片段，让用户可核对
  • ④对没有强引用的答案标"待人工核对"。
#293

78. 知识库更新后如何回归？

答
  • ①增量入库后跑变更范围 query 看是否能召回新内容
  • ②原 Golden Set 重新跑确保旧问题答案不退化
  • ③新增加问题进 Golden Set
  • ④版本号管理知识库，可回滚。
#294

79. RAG 答非所问的原因有哪些？

答

6 大类：

  • ①Query 理解错（多意/含糊）
  • ②向量检索召回了不相关 chunk
  • ③关键词检索误命中
  • ④TopK 太大引入噪音
  • ⑤Rerank 模型质量差
  • ⑥Prompt 没有约束
  • ⑦模型自由发挥。逐层定位。
#295

80. RAG 找不到答案的原因有哪些？

答
  • ①知识库没有这块内容
  • ②Chunk 切分把答案切断
  • ③Embedding 模型对术语不敏感
  • ④TopK 太小漏召
  • ⑤关键词不匹配（同义词未扩展）
  • ⑥Rerank 把对的 chunk 排到 Top-K 外。
#296

81. Chunk 太大有什么问题？

答
  • ①检索精度下降（一个 chunk 包含多个主题，向量稀释）
  • ②Prompt 上下文窗口被占满
  • ③成本上升
  • ④Top-K 引入更多无关信息。
#297

82. Chunk 太小有什么问题？

答
  • ①语义不完整（一句话不够表达上下文）
  • ②同一主题被切散，需要拼接多个 chunk
  • ③向量丢失上下文。
#298

83. TopK 太大有什么问题？

答
  • ①Prompt 噪音多影响生成质量
  • ②上下文窗口占用大
  • ③成本高（计算+ token 费）
  • ④可能稀释关键信息让模型抓不住重点。
#299

84. TopK 太小有什么问题？

答
  • ①漏召正确答案所在 chunk
  • ②答案不完整（需要多个 chunk 互补的场景）
  • ③模型不得不发挥（幻觉风险）。
#300

85. 文档重复会带来什么问题？

答
  • ①Top-K 全是同一份内容的不同副本，浪费上下文
  • ②如果版本不同（新旧版规范）会产生冲突
  • ③向量数据库存储和检索成本上升。解决：入库去重（哈希）+ 版本管理。
#301

86. 多版本规范条款冲突怎么处理？

答
  • ①每个 chunk 标记版本（生效日期、废止日期）
  • ②检索时只取当前有效版本
  • ③用户问历史问题可指定时间点检索
  • ④元数据过滤在向量库支持（Milvus scalar filter）。
#302

87. 知识库过期怎么处理？

答
  • ①每条 chunk 记录来源和入库时间
  • ②定期对照官方源校验有效性
  • ③有更新自动入库 + 旧版本标记废止
  • ④定期清理 N 年前的过期内容
  • ⑤过期高发期（规则集中更新期）人工 review。
#303

88. 用户问题不规范怎么处理？

答
  • ①Query 预处理（错别字纠正、口语化转书面）
  • ②Query 改写（让 LLM 生成 3 个改写版本一起检索）
  • ③扩展（同义词、上下位词）
  • ④引导用户补充信息（多轮澄清）。
#304

89. 企业知识库里如果新旧规范冲突，系统怎么处理？

答
  • ①metadata 标记新旧规范及生效日期
  • ②检索默认只返回当前生效版本
  • ③用户明确问"旧版"才返回
  • ④答案中明确标注"依据《xxx法》（2021 版）第 X 条"
  • ⑤旧法被废止时给出新法对应条款。
#305

90. RAG 回答是否需要引用来源？

答

强烈需要。业务场景必须引用：

  • ①让用户可核对
  • ②增加可信度
  • ③倒逼模型不编造
  • ④出问题可追溯。我们规定答案必须包含"参考依据：《xxx》第 X 条"。
#306

91. 有没有展示引用片段？

答

有，UI 上每条答案下方折叠展示"参考依据"，点开可看完整 chunk + 原文出处链接。这同时也是用户教育——让他们知道答案有据可查。

#307

92. 如果引用来源和答案不一致，怎么测试？

答
  • ①LLM-as-Judge 评估"答案中事实是否都来源于引用片段"
  • ②自动提取答案中的规范条款编号 vs 引用片段中的规范条款编号比对
  • ③人工抽样审核 5%
  • ④建立"不一致样本"集做回归。
#308

93. 知识库问答是否支持"不知道就回答不知道"？

答

支持，是核心设计。Prompt 中明确：「若上下文中没有相关内容，请回答"我暂时没有相关资料，建议咨询业务专家"」。测试集中专门构造知识库外问题验证拒识能力，拒识准确率要求 ≥ 90%。

#309

94. 你们如何防止模型自由发挥？

答

五招：

  • ①Prompt 强约束"只基于上下文"
  • ②温度设为 0
  • ③上下文中无相关信息时强制拒识
  • ④答案必须包含可核对的引用
  • ⑤后处理校验（如规范条款编号库交叉验证）。

#310

1. 企业知识库问答的测试难点是什么？

答

五大难点：

  • ①业务问题没有唯一答案
  • ②错误代价高（误导用户决策）
  • ③规范条款频繁更新
  • ④地区差异（地区规则、行业规则）
  • ⑤上下文长（一个工单可能涉及多部业务）。需要专家深度参与测试。
#311

2. 业务问题没有唯一答案，怎么验证？

答

不追求唯一答案，建立"答案合理性"判断：

  • ①是否覆盖关键业务要点
  • ②引用规范条款是否正确
  • ③是否给出操作建议
  • ④是否提示咨询人工客服/业务专家。多个合理答案都算通过，关键看是否避免明显错误。
#312

3. 你如何构造知识库问答测试集？

答

500+ 条，来源：

  • ①历史用户高频问题（脱敏后）
  • ②业务人员常见咨询
  • ③复杂案例
  • ④对抗样本（含错别字、多意、跨领域）
  • ⑤知识库外问题（测拒识）。每条由业务人员标注：标准答案 + 关键点 + 必引规范条款 + 禁止内容。
#313

4. 知识库问答如何做人工评审？

答
  • ①双业务人员独立评分（5 分制）
  • ②评分维度：准确性/完整性/合规性/可读性
  • ③分歧 > 1 分仲裁
  • ④每周汇总 Kappa 一致性 ≥ 0.8
  • ⑤每月做评审者 Calibration 防漂移。
#314

5. 知识库问答如何区分严重错误和一般错误？

答

严重错误（P0）：

  • ①引用不存在或错误的规范条款
  • ②给出违法建议
  • ③泄露其他用户业务单据信息
  • ④输出敏感/违法内容。一般错误（P1/P2）：
  • ①表述不严谨
  • ②缺少免责声明
  • ③遗漏次要业务要点
  • ④过于宽泛。
#315

6. 规范条款引用错误怎么发现？

答
  • ①建立规范条款编号库（同步企业规范库）
  • ②答案后处理：提取所有"《xxx规范》第 N 条"
  • ③逐一校验是否存在 + 内容是否匹配
  • ④不一致告警 + 标记为幻觉。
#316

7. 专业建议不严谨怎么处理？

答
  • ①优化 Prompt 加入"必须严谨、必须提示咨询人工客服/业务专家"
  • ②输出后审核（关键词检查）
  • ③不严谨样本进入回归集
  • ④业务人员定期 review 输出风格。
#317

8. 不同地区规则差异怎么测试？

答
  • ①知识库 metadata 标记地区（如《北京地区业务办理细则》）
  • ②query 中带地区信息时优先检索地区规则
  • ③测试集覆盖主要地区（北上广深 + 部分二线）
  • ④无地区指定时使用通用业务规则。
#318

9. 多轮追问如何测试？

答

设计 5+ 轮对话脚本：

  • ①第 1 轮提主问题
  • ②后续逐步细化
  • ③在第 N 轮提一个只能基于前文回答的问题
  • ④验证 AI 是否记住上下文
  • ⑤超过 context window 时是否合理压缩。
#319

10. 知识库外问题如何测试？

答

专门测试集（其他行业规则、未来业务规则、虚构规范条款），预期 AI 拒识或明确说"知识库未覆盖"。我们要求拒识率 ≥ 90%，幻觉率 ≤ 5%。

#320

11. 大模型在你们系统里扮演什么角色？

答

通用大模型在系统里担任 RAG 链路的生成层。检索完成后把用户 query 和召回片段一起送入模型，由它生成专业回答。优势是可结合企业知识库和业务上下文，表达更贴近真实业务。

#321

12. 是直接调用模型，还是接了 RAG？

答

接了 RAG。原因：

  • ①模型知识有时间截止
  • ②业务规则会更新
  • ③需要可解释（引用）
  • ④需要业务私有知识（内部 FAQ）。纯模型调用只用在不涉及具体规范条款的通用对话场景。
#322

13. 你们有没有 Prompt 模板？

答

有，分四套：

  • ①基础问答 Prompt（含角色定义、回答约束、引用要求）
  • ②业务文档生成 Prompt（结构化字段必须填充）
  • ③分类 Prompt（通知分类）
  • ④拒识 Prompt（无法回答时的统一话术）。所有 Prompt 进 Git 版本管理。
#323

14. 知识库问答有没有置信度？

答

有两层：

  • ①检索置信度（Top-1 相似度分数）
  • ②生成置信度（基于 logprobs 或 LLM 自评）。综合分 < 阈值时降级为"建议咨询人工客服/业务专家"。
#324

15. 如果 AI 回答不确定，是否会提示用户咨询人工客服/业务专家？

答

是。低置信度时 Prompt 引导 AI 输出免责声明 + 建议咨询。同时 UI 层在所有 AI 答案下方常驻"以上仅供参考，建议咨询业务专家"。

#325

16. 你们如何判断"准确合规"？

答

准确：

  • ①事实正确
  • ②规范条款引用正确
  • ③逻辑无错。合规：
  • ①不违法违规
  • ②不输出敏感内容
  • ③有免责声明
  • ④地区适用正确
  • ⑤业务人员审核通过。两者结合作为达标判断。
#326

17. 业务文档生成测试需要关注哪些字段？

答
  • ①业务对象信息（姓名、身份证、住址）
  • ②业务单据信息（单据编号、业务类型、新建日期）
  • ③金额（原始金额、计费金额、调整金额、服务费）
  • ④业务机构信息
  • ⑤日期
  • ⑥落款。这些字段必须 100% 与源数据一致。
#327

18. 姓名、单据编号、金额、业务机构、日期怎么校验？

答
  • ①从数据库取 Ground Truth
  • ②业务文档生成后解析提取字段
  • ③逐一比对
  • ④任意不一致 = P0 缺陷。这类字段绝不让模型自由生成，必须模板填充 + 后置校验。
#328

19. 业务文档模板如何测试？

答
  • ①每种业务文档覆盖至少 3 个测试样本（最简、正常、最复杂）
  • ②模板变量全部正确填充
  • ③缺失变量是否报错或使用默认值
  • ④格式（字体、排版、印章位置）正确。
#329

20. 模板变量缺失怎么测试？

答

构造缺失场景（金额为空、日期为空），看：

  • ①是否报错并阻断生成
  • ②是否填充占位符
  • ③是否调用默认值。预期是 P0 字段缺失必须阻断而非生成残缺业务文档。
#330

21. 生成业务文档格式怎么校验？

答
  • ①结构化校验（标题、正文、落款）
  • ②段落顺序
  • ③字段缩进
  • ④日期格式（年月日大写）
  • ⑤金额格式（小写 + 大写）
  • ⑥页眉页脚。Word/PDF 转纯文本后用规则校验。
#331

22. 生成业务文档内容怎么校验？

答
  • ①关键字段 100% 准确（金额、姓名、单据编号、日期）
  • ②业务术语合规
  • ③逻辑顺序正确
  • ④引用规范条款编号正确
  • ⑤无错别字（中文校对工具）。
#332

23. 生成业务文档是否支持人工编辑？

答

支持。流程：

  • ①AI 生成初稿
  • ②业务人员在线编辑
  • ③保存编辑后版本
  • ④对比初稿和终稿做模型改进数据。这是闭环重要环节。

24. 业务文档生成失败怎么处理？

  • ②降级到模板填充（不让 AI 润色）

29. 是模型生成全文，还是模板填充 + 模型润色？

混合模式：

  • ②叙述性段落（事实描述、理由）由模型生成
  • ③生成后校验关键字段未被篡改。这样平衡灵活性和准确性。
#339

30. 如果金额字段错误，会造成什么风险？

答
  • ①合规风险（错误金额可能触发投诉或审计问题）

  • ④信任风险（一次错误失去客户）。所以金额必须 100% 准确。
#340

31. 业务文档合规性有没有审核规则？

答

有：

  • ①格式规则（《业务文档样式规范》）
  • ②内容规则（必备要素清单）
  • ③语言规则（业务术语统一）
  • ④逻辑规则（前后一致）。规则引擎自动校验 + 业务人员抽样人工审核。
#341

32. 是否有业务专家人工审核环节？

答

有，分两级：

  • ①AI 生成后业务人员在线审核（强制环节，必须签字）
  • ②高风险业务文档（涉及金额 > 阈值、特殊工单）双业务人员复核。业务人员审核留痕作为质量追溯依据。
#342

33. 智能通知通知的业务目标是什么？

答
  • ①按业务阶段自动触发对应通知（创建、受理、处理、完成）
  • ②按客户分群推送差异化内容
  • ③提升运营触达/通知到达率和转化率
  • ④减少人工客服工作量。核心指标：策略命中率、用户响应率、投诉率。
#343

34. 通知策略命中是什么意思？

答

"在 X 业务场景下，AI 推荐的通知模板与人工标注的最优模板一致"。命中=人工标注预期发通知模板 A，AI 也推荐 A。是策略推荐准确性的指标。

#344

35. 通知自动分类有哪些类别？

答

约 20 类：创建通知、受理通知、处理进度通知、完成结算通知、续约邀约、运营触达提醒（分多个阶段）、风险提示、节日问候、营销活动、政策解读等。每类有独立模板和触发规则。

#345

36. 分类准确率如何计算？

答

准确率 = 分类正确的样本数 / 总样本数。整体准确率 + 每类的 Precision/Recall/F1 都要看，避免"小类被忽略"。

#346

37. 通知内容是否由 AI 生成？

答

混合：

  • ①模板填充为主（合规可控）
  • ②个性化变量由 AI 生成（如根据客户画像调整语气）
  • ③敏感内容（金额、单据编号）不让 AI 改写。生成后强制走合规审核。
#347

38. AI 生成通知如何防止不合规内容？

答
  • ①敏感词过滤
  • ②内容审核 API（阿里/腾讯）
  • ③业务人员审核高风险模板
  • ④灰度发布观察用户反馈
  • ⑤监控投诉率。任何一关失败拦截发送。
#348

39. 通知发送失败怎么处理？

答
  • ①重试机制（运营商失败重试 3 次）
  • ②超过重试上限标记失败入失败队列

1. 什么是 MCP？

答

Model Context Protocol，模型上下文协议，由 Anthropic 提出。本质是大模型与外部工具/数据源之间的标准化通信协议。让 LLM 通过统一接口接入数据库、API、文件系统、企业内部系统，类似"USB for AI"。

#359

2. MCP 和 Function Calling 有什么区别？

答
  • Function Calling：模型厂商私有的函数调用机制，与具体模型绑定（OpenAI/Anthropic 各有差异）；
  • MCP：跨模型/跨厂商的开放协议，工具一次实现可被任意 MCP 兼容的 Client 使用。

类比：FC 像各厂商私有充电口，MCP 像 USB-C 通用标准。

#360

3. MCP Server 和 MCP Client 分别是什么？

答
  • MCP Server：提供能力的一方（如数据库 Server、文件系统 Server），暴露 Tools/Resources/Prompts；
  • MCP Client：消费能力的一方（如 Claude Desktop、Cursor、IDE 插件），调用 Server 完成任务。

一个 Agent 可同时连接多个 MCP Server。

4. MCP Tools、Resources、Prompts 分别是什么？

答

  • Resources：模型可读取的数据（如某个文件、某个 URI）；
  • Prompts：预设的 Prompt 模板（如"代码审查模板"）。

三者覆盖了"动作 / 数据 / 指令"三个维度。

5. XPath 和 CSS Selector 怎么选择？

6. 显式等待和隐式等待有什么区别？

答
  • 隐式等待：全局设置一个等待时长，元素未出现等待到超时；
  • 显式等待：对特定元素等待特定条件（可见/可点击/文本出现）。

显式更可控更推荐，Playwright 默认就是显式智能等待。

#428

7. Page Object 模式是什么？

答

将每个页面封装为一个 Class，页面元素作为属性，操作作为方法。测试用例只调用页面方法，不直接操作元素。优势：

  • ①UI 变更只改 Page Class
  • ②用例可读性强
  • ③复用性高。

26. 你用 Charles 主要做了什么？

答

27. Charles 如何抓 HTTPS？

答

四步：

  1. 开启 SSL 代理：Proxy → SSL Proxying Settings → Enable SSL Proxying，添加要抓的 host（如 *.example.com:443），不加白名单的 HTTPS 会显示乱码。
  2. 安装根证书：
    • PC：Help → SSL Proxying → Install Charles Root Certificate，按提示信任；
    • iOS：Help → SSL Proxying → Install Charles Root Certificate on a Mobile Device or Remote Browser，按提示在手机浏览器访问 chls.pro/ssl 下载证书，并在"设置→通用→关于本机→证书信任设置"中信任；
    • Android 7+：除安装证书外，还需要 App 的 network_security_config.xml 中加入 <certificates src="user"/>，否则 App 不信任用户证书（这是 Android 默认拦截抓包的关键）。
  3. 手机走代理：WiFi 配置代理为 Charles 所在 PC 的 IP + 端口（默认 8888）。
  4. 特殊场景：①SSL Pinning 的 App 需配合 Frida/Xposed 绕过；②小程序需要在微信开发者工具或抓包专用调试包。
#479

28. Charles 如何篡改响应？

答

四种方式由轻到重：

  1. Breakpoints（一次性拦截）：右键接口 → Breakpoints，请求到达时手动修改 Body/Status 再 Execute。适合临时调试。
  2. Rewrite（持续规则）：Tools → Rewrite，配置 Match URL + Modify Header/Body/Status，对所有匹配请求生效。适合长期边界测试。
  3. Map Local（本地 Mock）：把接口响应映射到本地 JSON 文件，直接返回文件内容。适合 Mock 复杂响应或后端未就绪时联调。
  4. Map Remote（转发）：把生产域名转发到测试服务器，无需改客户端配置。

实战场景：测金额异常返回 → 用 Rewrite 把响应中 "amount": 1000 改为负数；测 Token 过期 → 用 Breakpoints 把状态码改 401。

29. Charles 如何模拟弱网？

答

Proxy → Throttle Settings：

  • 全局开关：勾选 Enable Throttling；
  • 预设带宽：选 56k Modem / 256k ISDN / GPRS / EDGE / 3G / 高速 3G / 4G 等；
  • 自定义：调整 Bandwidth（带宽）、Utilisation（利用率）、Round-trip Latency（往返延迟）、MTU；
  • 针对单接口：在接口列表右键 → Enable Throttling on Selected hosts only，只对指定 host 生效，避免影响其他业务。

注意：Charles 不支持原生丢包模拟，需要丢包测试时配合 Linux tc 或 macOS Network Link Conditioner。

场景 1：RAG 问答答案错误

问题

用户问"会员到期后的退费规则？"系统回答却引用了采购报销相关规范条款。如何定位？

答

按链路逐层定位：

  1. 看检索日志：拿 query 调 RAG 检索接口，看 Top-K 召回了哪些 chunk。
  2. 判断检索 vs 生成：
    • 如果召回的就是采购报销 chunk → 检索问题；
    • 如果召回了退费规则 chunk 但模型答成采购流程 → 生成问题。
  3. 检索问题排查：
    • Query 理解：用 Embedding 试算 query 向量，看是否与采购报销 chunk 距离更近；
    • Chunk 切分：退费规则条款是否切得太碎；
    • TopK：是不是只取了 Top-1 没召回真正相关的；
    • Rerank：rerank 模型是否错把采购报销排前；
    • 关键词检索：是否"退费""到期"未触发关键词匹配。
  4. 生成问题排查：Prompt 中是否强制"基于上下文"、温度是否过高。
  5. 修复：①加退费规则关键词词典；②调 TopK 到 5；③加 Rerank；④Prompt 加强约束；⑤回归 Golden Set。
  6. 防止再发：把这条 query 加入回归集；建立"答案-引用类别一致性"自动校验脚本。

场景 2：AI 业务文档生成金额错误

问题

业务文档生成后，申请执行金额比工单系统里的金额少了 1000 元。如何测试和定位？

答
  1. 数据流追溯：从源数据库 orders/fees 表查出真实金额 vs 业务文档显示金额，定位差异。
  2. 逐层排查：
    • 数据源：金额来自哪些字段（原始金额+计费金额+调整金额+服务费）？SQL 重算一遍。
    • API 层：调用业务文档生成接口，看传入参数是否已经少 1000；
    • 模板层：模板变量映射是否错误（金额字段写成了"原始金额"而非"应执行总额"）；
    • AI 润色：是否模型重写时改了金额（这是最危险的，必须杜绝）；
    • 格式化：是否四舍五入策略错误（ROUND_DOWN？）；
    • 大小写转换：金额大小写是否一致。
  3. 防御措施：
    • 金额字段绝不让 AI 自由生成，必须模板填充；
    • 生成后做"金额一致性后置校验"：从结果文档解析金额 vs 数据库原值，不一致则阻断；
    • BigDecimal 全链路；
    • 业务专家人工审核必经环节。
  4. 回归用例：加入"零元/小数/大额/多笔合计"等边界用例。

Jenkins 每晚跑接口自动化，最近失败率从 2% 升到 30%。怎么处理？

答
  1. 快速分类：先看失败用例 Allure 报告，按失败原因聚类（5 大类）：
    • 环境问题：500/502/503 大批量出现；
    • 鉴权问题：401 集中；
    • 数据问题：数据校验失败（同一字段不存在）；
    • 接口变更：响应结构变了；
    • 业务变更：返回值符合新逻辑但用例没更新。
  2. 逐一确认：
    • 拉研发确认是否近期发布变更；
    • 看 git log 哪些代码改了；
    • 看测试环境是否被其他人占用导致脏数据；
    • 看 Token 是否过期、登录账号是否被锁。
  3. 修复策略：
    • 真 Bug → 提单跟踪；
    • 用例过时 → 更新断言；
    • 环境问题 → 推动运维 + 增加重试；
    • 数据脏 → 强化 fixture 隔离。
  4. 长效机制：
    • 失败用例分类自动统计（Allure Categories）；
    • 失败 Top 10 周报跟进；
    • 不稳定用例隔离（暂时移出 critical 集合）。

场景 4：JMeter 压测大量超时

问题

600 并发压测时，接口大量超时，但服务器 CPU 只有 30%。怎么分析？

答

CPU 不高说明瓶颈不在计算，而在等待资源。八个排查方向：

  1. 数据库连接池：连接数是否打满？看 HikariCP 监控 active=max。修复：调大池子或优化 SQL 释放连接更快。
  2. 线程池：Tomcat/业务线程池是否满？看 jstack 大量 WAITING。修复：调线程数或拆分线程池。
  3. 数据库慢查询：show processlist 看是否大量长查询。EXPLAIN 慢 SQL 找缺索引。
  4. 外部依赖：APM 看下游接口（第三方/微服务）耗时。修复：增加 Mock 或熔断降级。
  5. 锁竞争：数据库行锁、应用层 synchronized。dump 看是否大量线程等同一把锁。
  6. 网络瓶颈：测试机和被测机之间带宽，netstat 看 TIME_WAIT 堆积。
  7. JMeter 自身：检查 JMeter 机器 CPU/内存、JVM 堆。改 non-GUI 或分布式。
  8. 日志阻塞：同步写日志，磁盘 IO 满。改异步日志或减少日志量。

定位顺序：先看 APM 链路找瓶颈段，再针对性深入。

场景 5：通知分类准确率 95%，但业务仍然投诉

问题

通知自动分类准确率达到 95%，但业务仍然投诉很多。怎么看？

答

准确率高但投诉多，说明指标失真。可能原因：

  1. 类别不均衡：占比 80% 的"通用通知"类全对，但占比 5% 的"高风险运营触达"类只对 50%。整体准确率虽然 95%，但小类灾难。
  2. 指标维度错：只看 Accuracy 不看 Recall。运营触达类漏发等于丢钱，必须看每个小类 Recall。
  3. 业务代价：5% 的错误集中在高价值场景（如错把"严肃运营触达"分到"温馨提醒"）。
  4. 测试集与真实分布不一致：测试集都是历史样本，实际线上 query 分布不同。

修复方案：

  • 改用 F1 + 各类别独立指标，不再只看 Accuracy；
  • 按业务代价加权：错分代价高的类别权重更大；
  • 重要类别（运营触达/合规）单独看 Recall 必须 ≥ 95%；
  • 测试集按线上真实分布抽样；
  • 误分类样本人工 Review 找模式，针对性加训练数据；
  • 关键类别加规则兜底（如金额 > 5 万必走人工审核）。

场景 6：AI 回答正确但引用来源错误

问题

RAG 问答中，答案看起来正确，但引用的知识片段不是答案来源。这是问题吗？

答

是严重问题，尤其在业务场景。原因：

  1. 可解释性丧失：用户无法核对答案是否可靠。
  2. 业务场景必须有据可查：业务人员引用错误条款等于失信。
  3. 背后可能是幻觉：答案"看似正确"可能是模型凭记忆补全，而真实知识库没有这块内容——下次同样问法可能就错了。
  4. 风险传导：用户基于答案做决策却引用错误，后果由谁负责？

测试与防护：

  • 自动校验：用 LLM-as-Judge 评估"答案中每个事实点是否都在引用片段中"；
  • 强制约束：Prompt 改为"只能基于以下引用回答，未提及的事实不许补充"；
  • 后置校验：提取答案中的规范条款编号 vs 引用片段中的规范条款编号比对；
  • 回归集：把这类样本纳入 Golden Set；
  • UI 设计：让用户可一键看引用原文，相关片段高亮。

判定原则：业务场景宁可拒识也不能引用错。

场景 7：测试时间被压缩

问题

原计划 5 天测试，研发延期后只剩 1 天，明天必须上线。怎么做？

答

不慌，按风险优先级压缩。七步：

  1. 风险评估：
    • 列出本期变更范围（新功能/修改/Bug 修复）；
    • 评估影响（用户量/核心程度/线上风险）；
    • 量化"如果不测，可能出什么问题"。
  2. 范围裁剪：
    • 必跑：核心路径冒烟 + 接口自动化全量（约 30 分钟）+ AI 评估核心集；
    • 选跑：本期变更涉及的回归用例（约 30 个）；
    • 跳过：与变更无关的回归（用接口自动化覆盖）。
  3. 自动化兜底：让 Jenkins 全力跑回归集，1 小时内出结果。
  4. 资源协调：拉团队全员 1 天集中突击核心用例。
  5. 风险升级：明确写邮件给项目经理：「在 1 天测试时间内，仅能覆盖核心路径，遗留风险 X、Y、Z」，让上层书面决策。
  6. 灰度兜底：要求上线后先 10% 灰度 24 小时，监控核心指标，无异常再全量。
  7. 回滚预案：准备回滚脚本和数据回滚方案，可 30 分钟内回滚。

上线后：补跑剩余用例 + 加强监控告警 + 客户反馈快速通道。

绝不"硬扛压力沉默上线"，决策权和风险都让管理层看到。

#549

1. "策略命中率 95%+"——命中率怎么算？

答

公式：命中率 = AI 推荐通知策略与人工标注最优策略一致的样本数 / 总测试样本数 × 100%。

  • 分母：1000 条业务人员人工标注的历史通知场景，覆盖 20 个类别。
  • 分子：策略系统输出与人工标注一致的条数。
  • 统计周期：月度，由我汇总后业务负责人确认。
  • 数据来源：测试日志 + 人工标注 Excel + 策略系统输出对账。

⚠️ 主动点破口径：这是标注集上的离线指标，不等于线上真实命中率。面试官最爱追"那线上是不是也 95%"——你先点破，反而显得严谨。

#550

2. "用例覆盖率 95%+"——公式是什么？

答

公式：用例覆盖率 = 已被至少一条测试用例覆盖的需求点 / 总需求点 × 100%。

  • 分母：需求文档中识别的所有需求点（功能 + 异常 + 业务规则）。
  • 分子：在 RTM（需求-用例追溯矩阵）中已关联用例的需求点。
  • 工具：用 TestLink 维护 RTM。
  • 每版本评审会上由产品确认覆盖完整。
#551

3. "核心接口自动化覆盖率 80%+"——统计维度是什么？

答

公式：核心接口自动化覆盖率 = 已有自动化用例的核心接口数 / 总核心接口数 × 100%。

  • "核心接口"定义：日均调用量 > 1000、影响核心业务流程（登录、工单 CRUD、业务文档生成、AI 问答、通知）的接口，约 100 个。
  • "已覆盖"：至少有 1 条 happy path + 1 条 error case 的接口。
  • 统计：每月手工统计一次，按接口数量计算（不按用例数量，避免虚高）。
#552

4. "弱网功能正常率 99%"——这个指标怎么统计？

答

公式：弱网功能正常率 = 在弱网场景下功能可成功完成的用例数 / 弱网测试集总用例数 × 100%。

  • 弱网场景：6 种（高延迟、丢包 10%、丢包 30%、限速 3G、限速 2G、断网恢复）。
  • 测试集：100 条核心业务用例（覆盖核心功能 + AI 问答 + 业务文档生成 + 通知发送）。
  • "可成功完成"：核心功能可走通（允许重试、允许性能略降，但不能崩溃或数据错）。
  • 统计周期：每版本上线前测一次。

⚠️ 收窄口径：说成"我自建的 100 条核心弱网用例集的通过率"，别说成"整个平台弱网正常率"。而且 99% 在弱网下偏高、不太可信，按真实数报（哪怕 95%）反而更稳。

#553

5. "线上问题复现率与闭环率 98%+"——指标怎么算？

答
  • 复现率 = 测试环境成功复现的线上问题 / 线上问题总数。
  • 闭环率 = 已修复并验证通过的线上问题 / 线上问题总数。
  • 统计周期：季度。
  • 数据来源：Jira 工单状态导出。
  • 98%+ 意味着季度内 50 个线上问题中只有 1 个未闭环（一般是非紧急问题延期）。
