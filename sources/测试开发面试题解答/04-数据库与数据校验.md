## Q14：数据库读写操作模块封装

```python
class DBUtil:
    def __init__(self, cfg):
        self.pool = PooledDB(pymysql, maxconnections=5, **cfg)  # 连接池

    def query(self, sql, args=None):
        conn = self.pool.connection()
        try:
            with conn.cursor(pymysql.cursors.DictCursor) as cur:
                cur.execute(sql, args)        # 参数化，防注入
                return cur.fetchall()
        finally:
            conn.close()                      # 归还连接池

    def execute(self, sql, args=None):
        conn = self.pool.connection()
        try:
            with conn.cursor() as cur:
                rows = cur.execute(sql, args)
                conn.commit()                 # 显式提交
                return rows
        except Exception:
            conn.rollback()                   # 失败回滚
            raise
        finally:
            conn.close()
```

**1. 用 `pymysql` 还是 SQLAlchemy**：轻量校验用 `pymysql`/`DBUtils` 足够；ORM 多、模型复杂用 SQLAlchemy。测试场景多用前者。

**2. 连接信息**：放 `config`，**按环境区分**（dev/test/prod），绝不硬编码。

**3. 连接池**：用，避免每次查询都建连/断连，提升并发稳定性。

**4. SQL 失败处理**：捕获异常 → 打印 SQL 和参数 → 抛 **清晰异常**，不吞掉。

**5. 查询结果与接口响应对账**：把查库结果和接口响应的关键字段做断言比对（见 Q15）。

**6. 增删改是否自动提交**：**显式 commit**，失败 **rollback**；不要依赖隐式提交。

**7. 查询后释放连接**：`finally` 里 `close()`（连接池场景是归还），或用 `with`，绝不泄漏连接。

**8. 防 SQL 注入**：**参数化查询**（`cur.execute(sql, args)`），永远不用字符串拼接 SQL。

**9. 避免测试 SQL 误操作生产**：环境隔离 + **只读账号** + 禁止无 `where` 的 `update/delete`（代码层做防呆校验）+ 生产库不接自动化。

**10. 多环境切换**：`config` 按 `env` 取连接配置，运行时通过参数选择。

## Q15：接口返回成功但数据库没数据，如何定位

这是一道经典排查题，思路是 **从外到内、逐段确认**：

**1. 先看响应还是日志**：先确认接口 **真的** 返回成功——不只看 HTTP 200，要看 **业务码** 是否为成功（可能 200 但 `code` 非 0）。再结合应用日志。

**2. 确认请求参数正确**：是不是参数本身就没带对（比如 userId 传错），后端"成功"地处理了一个空操作。

**3. 确认是否写到别的库/表**：分库分表场景，可能写进了另一个分片，查错库了。

**4. 是否异步写入**：很多写操作走 **MQ 异步落库**，接口返回成功只代表消息发出去了，查太早自然没有 → 需要等待/轮询。

**5. 是否写缓存没落库 / 消息丢失**：先写 Redis 后异步落库，中间消息丢了。

**6. 是否事务回滚**：接口内后续步骤异常触发回滚，但响应已先返回成功（代码 bug）。

**7. 是否主从延迟**：写主库、查从库，主从同步有延迟 → 刚写完查从库查不到。

**8. 轮询等待**：针对异步，设计 **带超时的重试查询**（每隔 N 秒查一次，最多查 M 次），给异步留时间，避免误报。

**9. 业务代码问题 vs 脚本问题**：脚本查错库/查询条件写错 → **脚本问题**；参数对、时机对、库对，确实没写 → **业务 bug**。

**10. 用 traceId 定位全链路**：拿接口返回或日志里的 traceId，串起网关→服务→MQ→DB 的完整调用链，精确定位卡在哪一环。
