# 十三、30 分钟高频必问（速记版）

> 时间紧时优先准备这 10 题。每题给"一句话定调 + 关键点"，详细展开见对应章节。

---

### 1. 介绍你的接口自动化框架整体架构（详见 ch01）

**一句话**：分层 + 数据驱动——api / common / config / data / testcases / logs / report 各司其职，用例只调 api 层不碰 requests。
**关键点**：request_util 管"怎么发"，api 层管"发什么"；改一处不牵动全身（接口变只改 api+data）。

---

### 2. requests 请求模块如何封装（详见 ch01-Q2、ch11-Q33）

**一句话**：一个 `request(method, url, **kwargs)` 统一入口，内部处理超时、重试、日志、脱敏、统一返回。
**关键点**：method 透传；timeout 必传；只对网络异常重试、业务异常不重试、非幂等接口不重试；params/data/json 区别要说清。

---

### 3. token、orderId 这类动态参数如何传递（详见 ch03-Q11、ch11-Q36）

**一句话**：用上下文对象——从响应提取（`data.orderId`）存进 context，后续接口用 `${orderId}` 占位，发请求前自动替换。
**关键点**：支持嵌套提取；并发用独立 context 或加锁；字段不存在抛清晰错误。

---

### 4. pytest fixture 有哪些 scope，为什么 token 用 session（详见 ch02-Q6）

**一句话**：function/class/module/session 四级；token 用 session 是为了全局只登录一次，省时间、避免被风控。
**关键点**：前提是 token 可共享且有效期够；每条用例要换账号就改回 function。

---

### 5. YAML 数据驱动如何实现（详见 ch01-Q3、ch11-Q35）

**一句话**：YAML 存接口名/地址/方法/参数/预期，读成 list 喂给 `parametrize`，N 组数据 = N 条用例。
**关键点**：读取阶段校验必填字段并报清晰错误；`ids` + `allure.dynamic.title` 显示中文标题。

---

### 6. 断言失败后如何定位问题（详见 ch01-Q4）

**一句话**：断言信息带上期望值/实际值/字段路径/完整请求响应，并区分接口问题、数据问题、环境问题三类归因。
**关键点**：断业务码不只断 200；软断言一次看全部错误；失败 attach 到 Allure。

---

### 7. JMeter 压测中 TPS 上不去如何排查（详见 ch05-Q18）

**一句话**：分层排查——CPU 高看 GC/热点，CPU 不高响应慢看等锁/等 IO/连接池，再查慢 SQL、线程池、下游、MQ 积压。
**关键点**：CPU 没满还慢 = 等待型瓶颈；用链路追踪定位下游；TPS 拐点定容量。

---

### 8. 自动化测试如何接入 Jenkins（详见 ch06-Q20）

**一句话**：Pipeline 拉代码 → 装依赖 → pytest（按 marker）→ 生成 Allure → 失败通知；环境/标签做成构建参数。
**关键点**：凭证放 Credentials；冒烟与全量分流水线；报告归档 + 趋势。

---

### 9. 你如何理解精准测试（详见 ch07-Q23）

**一句话**：基于代码变更影响范围，从全量用例筛出最小必跑集，省回归时间。
**关键点**：不只测改动代码，还测受影响上下游；靠调用链 + 覆盖率建映射；最大难点是映射准确性，不准会漏测。

---

### 10. AI Agent 和传统自动化脚本有什么区别（详见 ch09-Q27）

**一句话**：脚本是固定流程写死 if-else；Agent 以 LLM 为大脑，能动态规划、调用工具、带记忆、根据中间结果调整下一步。
**关键点**：Agent 能感知环境（工具返回即 Observation）并自主决策；但没有自我意识、会幻觉，要做权限/步数/人工审核等边界控制。

---

## 临场提醒

- 每题 **先一句话定调，再展开 2~3 个关键点**，不要一上来堆细节。
- 多用"我在项目里是这么做的……因为……"句式。
- 不会的点直说掌握程度，不编造。
- 反复强调 **工程化、可维护性、可观测性、风险控制**。
