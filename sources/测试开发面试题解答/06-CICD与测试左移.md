# 六、CI/CD 与测试左移（Q20–Q22）

## Q20：接口自动化如何接入 Jenkins

**1. Jenkins 执行哪些命令**（一个 Pipeline 的典型阶段）：

```bash
git pull                                    # 拉代码
pip install -r requirements.txt             # 装依赖
pytest -m "${MARKER}" --alluredir=./report/allure-results   # 跑用例
allure generate ./report/allure-results -o ./report/html --clean   # 生成报告
```

**2. Python 依赖安装**：`requirements.txt` 锁版本，`pip install -r`。

**3. 是否用虚拟环境**：用 `venv` 或直接用 Docker 镜像，保证环境干净、可复现。

**4. 测试环境参数传入**：用 Jenkins **构建参数**（`env=test`）通过命令行传给 pytest（`--env=test`），框架按参数选 config。

**5. 只跑 smoke**：`pytest -m smoke`，把 MARKER 做成构建参数。

**6. 全量回归**：定时触发（每天凌晨）跑全量，或合并到主干时触发。

**7. Allure 报告生成**：装 Allure Jenkins Plugin，构建后自动生成趋势报告。

**8. 失败通知**：构建后步骤接 **邮件 / 钉钉 / 飞书 / 企业微信** webhook，把失败数和报告链接推群。

**9. 定时任务**：用 Jenkins 的定时构建（`H 2 * * *` 这种 cron）。

**10. 节点管理**：master 调度 + 多 agent 执行，按标签把任务分到合适节点。

**11. 历史报告保存**：归档构建产物 + Allure 趋势图，保留历史对比。

**12. 凭证管理**：账号/密钥放 **Jenkins Credentials**，用变量引用，绝不明文写进 Jenkinsfile 或代码。

## Q21：如何设计 CI/CD 中的质量门禁

质量门禁 = 在流水线关键节点设"卡口"，不达标就 **阻断**。

**1. 各阶段跑什么**：

- 提交/PR：**冒烟 + 单元测试**（快，几分钟出结果）。
- 合并到主干：**接口回归**。
- 发布前：**全量回归 + 性能基线**。

**2. 冒烟失败是否阻断发布**：**阻断**。冒烟挂说明核心链路不通，没有继续的意义。

**3. 自动化通过率门禁**：设阈值（如 ≥ 98%），低于则阻断；核心用例要求 100%。

**4. 严重缺陷是否必须为 0**：是。**阻塞级/严重级缺陷必须清零** 才能发布。

**5. 代码覆盖率门禁**：可设增量覆盖率门槛（如新增代码覆盖率 ≥ 70%），防止只加代码不加测试。

**6. 性能指标劣化是否阻断**：关键接口 TPS/响应时间相比基线劣化超过阈值（如 P95 上涨 20%）应阻断或告警。

**7. 安全扫描高危处理**：SAST/依赖扫描发现 **高危漏洞** 阻断，中低危记录跟踪。

**8. 门禁与发布效率平衡**：门禁太严会拖慢交付，太松形同虚设。做法：快检查（冒烟）放前面快速反馈，重检查（全量/性能）异步或定时跑。

**9. 强门禁 vs 弱门禁**：

- **强门禁**：不达标直接阻断（冒烟、严重缺陷、高危漏洞）。
- **弱门禁**：只告警不阻断（覆盖率轻微下降、非关键用例失败），给人工判断空间。

**10. 误报导致流水线阻塞**：要有 **快速止血通道**——标记已知 flaky 用例、人工复核、临时豁免（带审批和记录），同时排查误报根因，而不是直接关掉门禁。

## Q22：测试左移如何真正落地

测试左移 = 把测试活动 **前移到需求/设计/编码阶段**，而不是等开发完才介入。

**1. 需求阶段测试人员做什么**：参与 **需求评审**，提测试点、挖隐含需求和异常场景，评估 **可测性**。

**2. 技术方案评审阶段**：评审 **接口契约**（字段、错误码、幂等设计）、异常设计、日志/可观测性是否到位。

**3. 提测条件**：开发 **自测通过 + 冒烟通过 + 接口联调通过** 才允许提测，避免带病提测。

**4. 代码合并时跑哪些测试**：单元测试 + 接口冒烟，作为合并的前置门禁。

**5. 流水线自动执行**：把测试嵌入 CI，提交即触发，快速反馈。

**6. 自动化用例何时开始写**：**需求/设计阶段就开始**，接口契约定了就能写接口用例，不等界面做完。

**7. 接口级测试嵌入流水线**：接口稳定后接口自动化纳入每次构建，比 UI 自动化更稳定、更快、更适合卡门禁。

**8. 避免形式主义**：左移不是"开个评审会签个字"，要有 **实质产出**（测试点清单、可测性意见、契约用例）和 **效果度量**。

**9. 是否全员参与**：是。左移强调 **质量内建**，开发写单测、做自测，测试提供能力和门禁，是协作而非测试单方面加班。

**10. 衡量效果**：**缺陷前移率**（需求/设计阶段发现的缺陷占比上升）、**线上缺陷率下降**、**返工率下降**、提测质量提升。
