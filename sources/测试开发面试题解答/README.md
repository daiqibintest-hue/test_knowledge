## 答题原则（贯穿全篇）

1. **先讲结论，再讲实现**：一句话定性，再展开细节。
2. **结合真实项目场景**：不空谈概念，用"我在项目里怎么做"的口吻。
3. **不只讲工具名，要讲为什么用**：例如不是"我用了 pytest fixture"，而是"为了让登录只执行一次、避免被风控，我用 session 级 fixture"。
4. **不只讲正常流程**：主动补充异常处理、边界、数据清理。
5. **避免把"框架"吹成"精准测试 / 平台"**：能力边界要诚实。
6. **不神化 AI Agent**：它没有自我意识，是概率模型，要讲清楚幻觉与风险控制。
7. **性能瓶颈不能只看 CPU/内存**：要讲全链路定位思路。
8. **平台问题从架构维度答**：不能只答"脚本放网页上跑"。
9. **不确定的技术点诚实说明掌握程度**，不要编造。
10. **答案中突出工程化、可维护性、可观测性和风险控制**。

> 核心表达：测试开发不是单纯写自动化脚本，而是通过代码、框架、工具和平台，把测试能力工程化、自动化和平台化。
