# 十九、中间件与微服务排障

> 5 年测开常被追问的不只是会不会写自动化，还包括能不能看懂系统链路，定位 Redis、MQ、网关、容器和微服务问题。

---

## Q65：Redis 缓存和数据库不一致怎么定位？

先确认不一致的类型：缓存旧、缓存空、缓存多、数据库没有、还是页面展示用了多级缓存。定位时同时看接口响应、Redis key、数据库记录和更新链路日志。

常见原因有四类。第一是更新策略问题，比如先删缓存再写 DB，中间有并发读把旧值重新写入缓存；更稳妥的策略通常是先写 DB 再删缓存，必要时延迟双删。第二是缓存过期时间过长，业务变更后没有主动失效。第三是多服务写同一份数据，但只有部分服务删了缓存。第四是读写分离延迟，缓存刷新时读到了从库旧数据。

测试设计要覆盖：更新后立即查询、并发更新查询、删除恢复、缓存击穿、服务重启、消息通知失败。验证不能只看接口返回，要同时校验 Redis 和 DB，并记录 traceId，方便证明问题发生在哪一层。

## Q66：Redis 缓存穿透、击穿、雪崩分别怎么测？

缓存穿透是大量查询不存在的数据，绕过缓存打到数据库。测试可以构造不存在的 id 高频请求，观察 DB QPS 是否异常升高，预期系统用空值缓存、布隆过滤器或参数校验拦截。

缓存击穿是热点 key 过期瞬间，大量请求同时打到 DB。测试可以设置一个短 TTL 热点 key，用 JMeter 在过期点附近并发请求，观察是否只有一个线程回源，其余等待或返回旧值。常见方案是互斥锁、逻辑过期、热点 key 预热。

缓存雪崩是大量 key 同时过期或 Redis 不可用。测试可以批量设置相同 TTL，或在测试环境模拟 Redis 连接失败，观察服务是否限流、降级、熔断，DB 是否被打爆。回答时要强调不要在生产随便做破坏性测试，必须在压测环境或演练环境进行。

## Q67：MQ 消息积压怎么排查？

先判断是生产太快、消费太慢、消费者异常，还是下游依赖慢。指标上看 topic/queue 积压量、生产速率、消费速率、消费者实例数、失败重试数和死信队列。

排查路径：第一看消费者是否在线，是否频繁重启；第二看消费日志是否有异常、超时、反序列化失败；第三看下游接口或数据库是否慢；第四看单条消息是否过大或顺序消费导致阻塞；第五看分区数、消费者并发数和消费组配置是否匹配。

测试用例要覆盖：正常消费、消费失败重试、重复投递、乱序、消息过期、死信转移、消费者重启、下游超时。验证重点是幂等性，消息重复消费不能重复扣款、重复发通知或重复生成单据。

## Q68：如何测试 MQ 消费幂等？

核心思路是让同一业务消息被消费多次，验证最终业务状态只生效一次。

操作方式可以有三种：用相同 messageId 或业务 id 手动投递多次；让消费者处理成功后在 ack 前异常退出，触发 broker 重投；在并发消费者下同时发送相同业务键。验证时不能只看接口返回，要看数据库唯一记录、状态机流转次数、通知发送次数、库存或金额变化。

幂等实现通常靠业务唯一键、去重表、唯一索引、状态机校验或分布式锁。面试回答要补风险：只靠 Redis 去重可能因过期或服务异常失效，关键业务最好有数据库唯一约束兜底。

## Q69：服务接口突然大量 502/504 怎么定位？

502 通常表示网关或代理拿到了上游异常响应，504 通常表示网关等待上游超时。定位要从入口到后端逐层看。

第一看 Nginx/API Gateway 日志，确认是全部接口还是单个接口，是所有实例还是某个实例。第二看服务注册发现，确认上游实例是否健康、端口是否可达。第三看应用日志和线程池，确认是否卡在数据库、RPC、外部接口或锁等待。第四看容器资源，CPU、内存、重启次数、探针失败都会导致网关拿不到正常响应。

测试侧需要补充超时、重试、降级、熔断用例。尤其要验证客户端重试是否会放大流量，POST 类接口是否有幂等保护，避免网络抖动时造成重复业务操作。

## Q70：Kubernetes 中服务不可用怎么排查？

可以按 Pod、Service、Ingress、依赖四层排查。

Pod 层看状态是否 Running、是否 CrashLoopBackOff、Readiness Probe 是否失败、最近是否 OOMKilled。Service 层看 selector 是否匹配 Pod label，endpoint 是否为空，端口映射是否正确。Ingress/Gateway 层看路由规则、证书、域名解析和网关日志。依赖层看数据库、Redis、MQ、配置中心是否可达。

常用命令包括 `kubectl get pod -o wide`、`kubectl describe pod`、`kubectl logs`、`kubectl get svc/endpoints`。面试不一定要求你是运维专家，但要能讲出排查顺序和证据链，而不是只说"重启一下"。

## Q71：发布后部分用户报错，如何判断是灰度问题还是数据问题？

先圈定影响范围：用户、租户、地区、版本、渠道、角色、灰度批次。灰度问题通常和版本、实例、流量规则相关；数据问题通常和特定用户数据、历史脏数据、权限或配置相关。

定位动作：查用户请求落到哪个实例、哪个版本；看灰度规则是否按用户 id、租户或 header 生效；对比正常用户和异常用户的数据差异；用同一账号切换版本或同一版本换账号做交叉验证。若同一用户在新版本异常、旧版本正常，更像灰度代码问题；若新旧版本都异常，更像数据或配置问题。

测试策略要补灰度验证：灰度名单内外、不同租户、回滚后数据兼容、灰度开关关闭后的行为。上线前最好准备可观测字段，比如响应 header 带版本号或灰度标记，方便快速定位。

## Q72：分布式事务异常怎么测试？

先搞清业务采用的是本地事务、TCC、Saga、消息最终一致性，还是简单的补偿任务。不同方案测试点不同。

以消息最终一致性为例，要覆盖：本地事务成功但消息发送失败、消息发送成功但消费失败、消费者重复消费、下游部分成功、补偿任务执行失败、人工补偿。验证不只看主表，还要看消息表、业务状态、补偿记录和告警。

面试里可以强调测试原则：分布式系统不要只验证"一次成功"，更要验证"半成功后能不能恢复"。测试开发可以通过 Mock 下游失败、注入超时、暂停消费者、制造重复消息来验证最终一致性。

## Q73：慢 SQL 如何从测试角度定位和推动修复？

测试侧先用现象证明问题：接口 P95 变高、数据库慢查询日志出现、执行计划显示全表扫描或索引失效。不要只说"数据库慢"，要提供 SQL、参数、数据量、执行耗时和 explain 结果。

常见原因包括：where 条件没有索引、组合索引顺序不对、函数或隐式转换导致索引失效、分页 offset 太大、返回字段过多、join 表数据量过大、锁等待。推动修复时可以和研发 DBA 一起看执行计划，评估加索引、改 SQL、拆查询、异步化或缓存。

回归验证要注意数据量。小数据量下 SQL 都很快，必须准备接近生产规模的数据，至少覆盖高频查询条件、边界分页和热点租户。

## Q74：服务降级和熔断怎么测？

降级是下游不可用时返回兜底结果或关闭非核心能力；熔断是错误率或超时达到阈值后短时间内不再请求下游，避免雪崩。

测试步骤：先让下游正常，确认主流程；再模拟下游超时、500、连接失败、慢响应，观察是否触发降级或熔断；继续请求确认熔断窗口内是否快速失败；等待恢复时间后确认半开探测和自动恢复。指标上看错误率、RT、熔断状态、降级次数和告警。

风险点是降级不能掩盖核心错误。例如商品推荐可以降级为空，支付扣款不能随便降级成功。回答时要按业务重要性区分，不要把所有异常都说成"返回默认值"。

## Q75：如何测试配置中心变更带来的风险？

配置中心风险在于变更快、影响面大、经常不走完整发布流程。测试要关注配置格式、默认值、热更新、回滚和权限。

用例包括：配置缺失、格式错误、非法值、边界值、灰度配置、热更新是否立即生效、旧实例是否同步、回滚后是否恢复。对于限流阈值、开关、路由、模型参数这类高风险配置，要有审批和审计记录。

面试可以补一句：配置变更也应该纳入质量门禁，至少要有 schema 校验、变更 diff、影响范围说明和线上监控，不然很多线上事故不是代码发布导致的，而是配置改错导致的。

## Q76：接口偶发超时怎么定位？

偶发超时最怕只看平均值。先收集 P95/P99、慢请求 traceId、发生时间、实例、用户、参数和依赖耗时。

排查方向包括：线程池排队、连接池等待、GC 停顿、数据库锁等待、慢 SQL、下游 RPC 抖动、网络丢包、容器 CPU throttle。需要把一次请求拆成网关耗时、应用处理耗时、DB 耗时、缓存耗时、下游耗时。

测试侧可以通过压测、弱网、下游延迟注入、并发锁冲突来复现。结论要用证据闭环，比如某些 trace 显示 90% 时间花在 DB 连接等待，而不是凭感觉说"可能是网络问题"。
