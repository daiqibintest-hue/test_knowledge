# 八、自动化测试平台（Q25–Q26）

---

## Q25：自动化测试框架和自动化测试平台的区别

一句话：**框架解决"怎么写和跑脚本"，平台解决"怎么管理、协作、调度和可视化"。**

**1. 我的接口自动化框架解决了什么**：脚本分层、数据驱动、统一请求/断言/日志、报告生成——解决"高效写脚本、稳定跑脚本"。

**2. 框架到平台需要增加什么能力**：用例管理、环境管理、任务调度、执行节点管理、报告聚合、权限管理、与外部系统打通。

**3. 平台需要用例管理**：在线增删改查用例、版本化、按项目/模块组织，非代码人员也能参与。

**4. 平台需要环境管理**：多环境配置集中管理，一键切换，避免散落在各处的配置。

**5. 平台如何任务调度**：定时触发、手动触发、流水线触发，任务进队列统一编排。

**6. 平台如何管理执行节点**：节点注册、心跳检测、负载分配、上下线管理。

**7. 平台如何统一展示报告**：所有任务报告集中存储展示，支持历史趋势、通过率曲线、跨任务对比。

**8. 平台如何管理用户和权限**：按角色（管理员/普通用户/只读）控制谁能改用例、谁能跑生产环境任务。

**9. 平台如何对接 Jenkins/Jira/禅道/飞书**：触发 Jenkins 构建、自动建缺陷单到 Jira/禅道、结果推送飞书。

**10. 为什么不能把脚本放网页上执行就叫平台**：那只是给框架套了个 Web 入口。**平台是体系化的管理 + 调度 + 协作 + 可视化 + 多租户**，核心价值在"工程化和协作能力"，不是"能在网页点一下运行"。

## Q26：如何设计一个高可用的自动化测试平台

我会按 **前端 + 后端服务 + 任务调度 + 执行节点 + 存储** 分层设计。

```
┌─────────────┐     ┌──────────────┐     ┌──────────────────┐
│  前端展示层  │ →   │  后端服务层   │ →   │  任务队列         │
│ 用例/任务/报告│     │ 用例/任务/调度 │     │ Redis/RabbitMQ   │
└─────────────┘     │ 节点/权限管理 │     │ /Kafka            │
                    └──────────────┘     └────────┬─────────┘
                            │                      │ 分发
                    ┌───────┴────────┐    ┌────────┴────────┐
                    │   存储层        │    │  执行节点集群    │
                    │ MySQL+对象存储  │    │ Node1 Node2 ... │
                    └────────────────┘    └─────────────────┘
```

**1. 前端展示层**：用例管理页、任务管理页、报告页、节点监控页、环境配置页。

**2. 后端服务层核心服务**：用例服务、任务服务、调度服务、节点服务、报告服务、权限服务。

**3. 用例管理模块**：用例 CRUD、版本管理、按项目/模块/标签组织、支持数据驱动。

**4. 测试数据管理模块**：测试数据集中管理、环境隔离、动态变量、数据初始化与清理。

**5. 测试任务创建与下发**：选用例集 + 环境 + 节点策略，生成任务进队列。

**6. 任务队列与"至少一次"投递的代价**：MQ 几乎都是 **at-least-once**——同一任务可能被投递多次、甚至被两个节点同时消费。**这不是 bug，是常态**，所以幂等是必选项不是可选项（见第 8 点）。选型：要可靠投递 + 延迟队列 → RabbitMQ（有 ack/nack/重入队）；要高吞吐 + 可回溯 → Kafka（offset + consumer group）；轻量小规模 → Redis Stream（consumer group + XACK + pending 列表）。

**7. 失活检测的真正难点：心跳超时 ≠ 节点真死（脑裂）**：节点心跳断了，可能是节点死了，也可能只是 **网络分区**——节点还在闷头跑任务。如果"心跳一超时就把任务重新分给别人"，就会 **两个节点同时跑同一个任务**，在造数/下单场景是灾难。正确做法：

- **租约（lease）**：节点领任务时拿一个带 TTL 的租约，执行中 **定期续租**；调度端只在 **租约真正过期**（而非心跳一断）才回收重分。
- **Fencing**：节点续租失败（发现自己租约已被抢）时 **主动中止当前任务、不再写结果**——保证即使被误判死亡，老节点也不会和新节点同时产出。
- 本质是拥抱 at-least-once + 幂等，而 **不追求 exactly-once**（分布式下没有真正的 exactly-once）。

**8. 幂等的具体实现（不能只说"唯一 ID + 状态机"）**：

- 任务状态机 `PENDING → RUNNING → SUCCESS/FAILED`，流转用 **数据库 CAS / 乐观锁** 抢执行权：

  ```sql
  UPDATE task SET status='RUNNING', node=?, lease_expire=?
  WHERE id=? AND status='PENDING';
  ```

  **只有 affected rows = 1 的节点抢到执行权**，其余节点拿到 0 行就放弃。这才是去重的真正落地（Redis `SETNX` 分布式锁是另一种，但要处理锁过期与误删）。
- 结果写入幂等：用 `task_id` 做结果表唯一键，重复写 upsert/忽略。
- **被测系统副作用的幂等才是最难的、也要诚实划清边界**：平台能保证"同一任务不被重复触发执行"，但 **保证不了"被测系统不产生重复副作用"**——如果被测下单接口本身不幂等，任务重跑可能造两条单。这要靠用例自带幂等号 + 数据清理来兜，不能甩锅给平台，也不能假装平台能解决。

**9. 数据隔离：并发跑同环境怎么不互相踩**：

- **账号池**：用例领账号 → 执行 → 归还，池空则排队，避免两个并发任务用同一账号改同一条数据。
- **数据命名空间**：造数带 `任务ID/时间戳` 前缀，天然隔离 + 便于按前缀批量清理。
- **环境级互斥**：改了全局配置的任务（如全量回归）必须独占环境 → 调度端给环境加锁，这类任务串行执行。

**10. 调度公平性与背压**：不能只"谁空闲发给谁"。要用 **优先级队列** 防止大任务饿死冒烟任务；按 **租户配额** 防止某项目占满所有节点；队列堆积超阈值要对上游 **背压**（拒绝新建或降级），否则雪崩。

**11. 平台自身高可用的真问题是调度器单点**：服务无状态 + 多实例是基本功；真问题是 **多个调度实例同时跑会重复下发** → 用 **选主**（ZooKeeper/etcd/Redis 锁）保证同一时刻只有一个 active 调度；主挂了备接管，接管时从 DB 读出 **`RUNNING` 且租约已过期** 的在途任务重新调度，做到状态可恢复。

**12. 存储分层 + 可观测**：结构化数据（任务、结果）存 MySQL，日志/报告/截图存 **对象存储（OSS/MinIO）**；监控 **队列 lag、节点租约、任务失败率、各阶段耗时**，失败率突增告警。

> 这一题的"深"不在于画出几个框，而在于能讲清 **至少一次投递、脑裂下的重复执行、CAS 抢执行权、副作用幂等的边界、调度器选主**——这些才是分布式执行平台真正会踩的坑。
