# 三、接口依赖与业务链路（Q11–Q13）

---

## Q11：登录 → 下单 → 支付 完整链路实现

链路本质是"前一个接口的输出作为后一个接口的输入"。我用 **上下文对象 + 业务流封装** 来串。

```python
def order_pay_flow(ctx):
    token = LoginApi.login("test01", "123456")          # 1. 登录拿 token
    ctx.save("token", token)
    order_id = OrderApi.create(ctx, sku="1001", num=1)  # 2. 下单拿 orderId
    ctx.save("order_id", order_id)
    PayApi.pay(ctx, order_id=ctx.get("order_id"))       # 3. 支付用 orderId
```

**1. token 从登录接口拿**；**2. 存上下文/session**；**3. 注入 headers**（统一在请求层加 `Authorization`）。

**4. orderId 从下单响应提取**（`extract: data.orderId`）；**5. 传给支付接口** 作为入参。

**6. 下单失败支付是否继续**：**不继续**。前置失败应直接终止链路——用 fixture 管前置时会标记 error 并跳过后续，避免基于错误状态继续跑产生误导性结果。

**7. 库存不足 / 8. 余额不足**：测试前 **造数准备**（充值、补库存），或专门设计异常用例去 **断言对应业务码**（如 `code=余额不足`）。

**9. 测试前准备账号/库存/余额**：用前置 fixture 调初始化接口或直接造库。**10. 测试后清理订单**：teardown 清理，或每次用独立的一次性数据。

**11. 保证可重复执行**：每次 **造新数据**（唯一手机号/订单号）+ **幂等清理**，不要依赖"上次跑剩下的数据"。

**12. 封装可复用业务流**：把"登录→下单→支付"封装成 `order_pay_flow()` 这种业务流函数，其他用例（退款、对账）直接复用，不用重复编排。

## Q12：强依赖时如何避免用例互相影响

**1. 多个用例共用一个订单**：尽量不要，极易互相污染（一个改了状态，另一个断言就挂）。

**2. 并发数据隔离**：每个用例/线程 **造独立数据**，账号、订单各自独立。

**3. 账号分组管理**：用 **账号池**，按线程/用例分配，用完归还，避免并发抢同一账号。

**4. 数据初始化**：前置接口或直接造库，保证每次起点一致。

**5. 脏数据清理**：teardown 清理 + 定期清理任务兜底（防止用例中途崩了没清干净）。

**6. 前置接口不稳定**：加重试；若仍失败，把用例标记为 **blocked/error 而非 fail**，区分"环境问题"和"功能 bug"。

**7. 区分依赖失败和当前接口失败**：前置放 fixture，前置挂 → **error**；被测接口断言挂 → **fail**。报告里一眼能分。

**8. 是否都依赖真实前置**：核心链路用真实前置保证端到端可信；非核心、第三方不可控的部分可以 mock。

**9. 哪些场景用 mock**：第三方不可控（支付回调、短信、银行）、异常场景难造（超时、特定错误码）、依赖服务未就绪时。

## Q13：token 过期如何处理

**1. 有效期**：看后端配置（常见 30min~2h），面试就说"以后端约定为准"。

**2. 执行中失效**：在请求层 **拦截 401**，自动重新登录刷新 token 后重发原请求。

**3. 自动重新登录刷 token**：可以，且应该。

**4. 刷新逻辑放哪层**：放 **传输层 / 鉴权拦截器**（request_util 里），对上层透明，用例无感知。

**5. 多线程安全共享**：token 用单例 + **加锁**，刷新时双重检查，避免多个线程同时发现过期、并发重复刷。

```python
with self._lock:
    if self._token_expired():     # 双重检查，只让一个线程刷
        self._token = self._login()
```

**6. token 写 YAML 是否安全**：**不安全**——明文泄露 + 很快过期。token 应 **运行时动态获取**，不落静态文件。

**7. token 是否脱敏**：日志中必须脱敏，只留前几位。

**8. 避免多线程同时刷**：双重检查锁（上面代码）。

**9. 401 vs 403**：

- **401 Unauthorized**：未认证——没登录 / token 失效 / token 错误。
- **403 Forbidden**：已认证但 **无权限**——身份合法但这个账号不能访问该资源。
