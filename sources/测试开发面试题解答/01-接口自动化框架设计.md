# 一、接口自动化框架设计（Q1–Q5）

---

## Q1：接口自动化框架整体架构

我的框架是 **分层 + 数据驱动** 结构：

```
project/
├── api/            # 接口层：一个接口一个方法，封装 URL / method / 参数组装 / 默认 header
├── common/         # 公共能力层：request_util、assert_util、yaml_util、log_util、db_util
├── config/         # 配置层：环境地址、账号、超时、DB 连接，按 env 区分
├── data/           # 数据层：YAML 用例数据，按业务模块分目录
├── testcases/      # 用例层：只做编排 + 断言，不碰 requests
├── logs/           # 运行日志
├── report/         # Allure 结果与报告
├── conftest.py     # 全局 fixture（token、环境切换、数据清理）
└── pytest.ini      # 运行配置
```

**1. 各层职责**

- `api`：业务语义层，知道"登录接口长什么样"（路径、参数结构、默认头），对外暴露 `login(username, pwd)` 这种方法。
- `common`：纯技术能力下沉，和业务无关（怎么发请求、怎么断言、怎么读 YAML、怎么打日志）。
- `config`：环境差异隔离，地址/账号/超时/DB 全在这里，切环境只改这里。
- `data`：测试数据，和代码分离，测试人员改数据不用动代码。
- `testcases`：编排业务流 + 断言，可读性优先。
- `logs`/`report`：产物输出。

**2. testcases 调 api 层，不直接调 requests。** 用例里出现 `requests.post(...)` 就说明分层塌了——接口一变要改一堆用例。用例只应该看到 `OrderApi.create(...)`。

**3. api 层 vs `request_util` 的边界**（高频考点）：

- `request_util`：**协议传输层**，只管"怎么把一个 HTTP 发出去"——超时、重试、日志、脱敏、统一返回封装，**不知道任何业务**。
- `api` 层：**业务语义层**，知道某个接口的 URL、参数怎么拼、默认带什么头，然后调用 `request_util` 发出去。
- 一句话：`request_util` 回答"怎么发"，`api` 层回答"发什么"。

**4. 接口入参变化** → 只改 `api` 层对应方法 + `data` 里的 YAML，`testcases` 和 `request_util` 不动。

**5. 新增支付接口** → 新增 `api/pay_api.py` 一个方法 + `data/pay/*.yaml` + `testcases/test_pay.py`；若是新服务再在 `config` 加地址。**公共层一行都不用动**。

**6. 统一加 token** → 放在 `request_util` 的统一入口（或 `requests.Session` 的默认 header），通过 conftest 的 fixture 注入；**绝不在每个接口里手写**。需要更灵活时用一个"鉴权拦截器/基类"，区分需要鉴权和不需要鉴权的接口。

**7. 可扩展/可维护性体现在**：分层解耦（改一处不牵动全身）、数据与代码分离、配置与代码分离、公共能力只写一遍。判断标准就是上面 4/5/6 问——改动是否被限制在最小范围。

## Q2：requests 请求封装设计

核心是一个 `RequestUtil`，对外只暴露一个 `request(method, url, **kwargs)`，内部统一处理传输细节。

```python
class RequestUtil:
    def __init__(self):
        self.session = requests.Session()  # 复用连接 + 共享 cookie

    def request(self, method, url, *, params=None, data=None,
                json=None, headers=None, files=None, timeout=10, retry=2):
        log.info(f"请求 {method} {url} | params={params} | json={_desensitize(json)}")
        for i in range(retry + 1):
            try:
                resp = self.session.request(
                    method=method.upper(), url=url, params=params,
                    data=data, json=json, headers=headers,
                    files=files, timeout=timeout)
                log.info(f"响应 {resp.status_code} | {_mask_text(resp.text)}")
                return ApiResponse(resp)              # 统一返回封装
            except (ConnectionError, Timeout) as e:   # 只重试网络类异常
                log.warning(f"第 {i+1} 次请求异常，重试：{e}")
                if i == retry:
                    raise
```

**1. 方法支持**：`method` 参数透传，GET/POST/PUT/DELETE/PATCH 全支持，不为每个方法写一个函数。

**2. 超时**：必须显式设 `timeout`，连接和读取可分开 `(connect, read)`；**绝不允许不传 timeout**，否则一个挂死接口会拖垮整个执行。

**3/4. 重试**：两类才重试——**网络类异常**（`ConnectionError` / `Timeout`）和 **网关类状态码**（`502` / `503` / `504`，注意这是 HTTP 状态码不是异常，要在拿到响应后判断）；**业务异常不重试**——比如返回 400「参数错误」、401，重试只会刷脏数据、掩盖问题。而且 **重试只对幂等方法开启，非幂等接口（下单、支付）默认不重试**，否则可能重复下单（完整可运行实现见 [11-代码实战.md](11-代码实战.md) 的 Q33）。

**5. 日志**：打印 method、URL、入参、状态码、响应体、耗时。

**6. 脱敏**：`password`、`token`、`idCard`、`phone` 等在打日志前做字段替换，落地的是 `pass****`，绝不打印明文。

**7. 文件上传**：用 `files=` 传 multipart，`Content-Type` 交给 requests 自动生成，不要手写。

**8. params / data / json 的区别**（必问）：

- `params` → 拼到 URL 的 query string（GET 用）。
- `data` → `application/x-www-form-urlencoded` 表单体。
- `json` → `application/json`，requests 自动序列化并设 Content-Type。
- 用错（该 `json` 用了 `data`）是新手最常见的 400/415 原因。

**9. 统一返回**：包成 `ApiResponse`，暴露 `.status_code` / `.json()` / `.get("data.id")`（支持点路径取嵌套字段），给断言层一个稳定接口。

**10. 网络异常 vs 业务异常**：网络异常 = 请求根本没拿到合法 HTTP 响应（抛异常、触发重试）；业务异常 = 拿到了响应但状态码/业务码不符合预期（不重试，交给断言层判定）。两者处理路径完全不同。

### 深挖：生产级请求层真正难的几点（容易被追问穿）

**a. `requests.Session` 不是完全线程安全的**：并发跑（pytest-xdist 多线程、或多线程压自研工具）时多个线程共享同一个 Session，其底层连接池和 cookie jar 会竞争，偶发 `Connection reset`、串 cookie。正确做法：**每线程/worker 独立 Session**（thread-local）或每线程一个 client 实例。这点没注意，并发下会出难复现的偶发失败。

**b. 超时语义的坑——`timeout` 不是"总耗时上限"**：`timeout=(connect, read)` 里的 read 是 **两个数据包之间** 的最大间隔，**不是整个请求的总时长**。一个每隔几秒滴一个字节的响应可以无限拖下去而永不触发 read 超时；requests **没有内置总超时**。要兜总时长得靠外层机制。面试能说出"read 超时 ≠ 总超时"就比绝大多数人深。

**c. 连接池会成为并发瓶颈**：Session 通过 `HTTPAdapter` 复用连接，默认 `pool_maxsize=10`。并发 > 10 会看到 `Connection pool is full, discarding connection`，连接反复建/弃、TPS 上不去。高并发要 `session.mount("https://", HTTPAdapter(pool_maxsize=并发数))`。

**d. 对 POST 重试的隐藏风险**：即便是网络异常，若是 **read 超时**，请求很可能 **已经到达服务端并被处理**，只是响应没回来——这时重试 POST 就会重复下单。所以"非幂等不重试"不仅针对 5xx，也针对 **超时**：**超时 ≠ 没执行**。urllib3 的 `Retry` 默认也只对幂等方法重试，同理。

**e. `resp.text` 会把整个响应读进内存**：大文件下载/导出接口用 `.text`/`.json()` 会 OOM，要 `stream=True` 分块读。统一封装要给"流式响应"留口子。

**f. 统一返回封装的取舍**：把异常都吞成统一返回对象会 **掩盖失败**，让用例误以为通了。原则：**传输层失败（连不上/超时/5xx）该抛就抛**，只有"拿到合法响应"才包成 `ApiResponse` 交给断言层判业务码——该 raise 的别 return。

## Q3：YAML 测试数据设计

**1. 一条用例字段**：

```yaml
- name: 正常登录
  url: /api/login
  method: POST
  headers: {Content-Type: application/json}
  json: {username: test01, password: "123456"}
  extract:                       # 提取动态变量给后续接口用
    token: data.token
  validate:                      # 断言规则
    - {eq: [status_code, 200]}
    - {eq: [body.code, 0]}
    - {is_not_null: body.data.token}
```

**2.** 请求参数、预期结果、断言规则 **都放 YAML**，用例代码完全数据驱动，加用例 = 加数据。

**3. 多组数据**：一个文件用 YAML 列表，每个 `-` 是一条用例。

**4. 结合 pytest 参数化**：读 YAML 得到 list，喂给 `@pytest.mark.parametrize`，N 条数据 = N 条用例。

```python
cases = yaml_util.load("data/login.yaml")
@pytest.mark.parametrize("case", cases, ids=[c["name"] for c in cases])
def test_login(case):
    resp = LoginApi.send(case)
    assert_util.validate(resp, case["validate"])
```

**5. `${token}` 动态参数**：运行时维护一个 **上下文 dict**，提取出来的 `token`/`orderId` 存进去；发请求前用正则把 YAML 里的 `${xxx}` 替换成上下文真实值（见 Q36）。

**6. 文件多了** → 按业务模块分目录（`data/user/`、`data/order/`、`data/pay/`），和 `testcases` 目录结构对齐。

**7. 字段缺失校验**：读取后做 schema 校验（必填 `url`/`method`），缺失就抛 **清晰报错**："login.yaml 第 2 条用例缺少 url 字段"，而不是等发请求时报 KeyError。

**8. 读取失败**：捕获 `YAMLError`/`FileNotFoundError`，给出文件名 + 行号，不抛裸异常。

**9. 数据 vs 环境配置：分开。** YAML 放业务数据，环境地址/账号放 `config`，否则切环境要改一堆 YAML。

**10. YAML / JSON / Excel 对比**：

- **YAML**：可读性最好、支持注释、适合复杂嵌套与断言规则——**首选**。
- **JSON**：机器友好但没注释、写起来啰嗦。
- **Excel**：业务/非技术人员易上手，但不擅长嵌套结构和断言规则，diff 不友好、不利于 Git 管理。

## Q4：断言模块封装

封装成一个 `validate(resp, rules)`，规则从 YAML 来，支持多种比较器。

**1. 状态码 vs 业务码：两个都断，重点是业务码。** HTTP 200 只代表请求通了，业务上可能 `code=10001 余额不足`。只断 200 会漏掉大量业务 bug。

**2. 嵌套字段**：用 jsonpath / 点路径，`body.data.user.id` 取值。

**3. 数据库断言**：支持，调 `db_util` 查库，把"接口说成功"和"库里真落了数据"对上（见 Q15）。

**4. 比较器**：eq / ne / contains / not_contains / is_null / is_not_null / type / gt / lt / len 等，做成一个比较器字典，YAML 里点名用哪个。

**5. JSON Schema 校验**：支持，用于校验响应 **结构和类型**（字段在不在、类型对不对），比逐字段断言更适合契约校验。

**6. 断言失败 Allure 里能看到**：期望值 vs 实际值、失败字段路径、完整请求和响应、对应日志——一眼定位。

**7. 失败归因**：

- 接口返回的业务码本身错 → 接口问题。
- 期望值写错、数据没造好 → 测试数据问题。
- 连不上、超时、500 → 环境问题。
  断言信息里带上这三类线索，不用翻代码就能判断。

**8/9. 软断言**：一条失败后面不执行 = 硬断言，会漏报。我支持 **软断言**：收集所有失败，最后一次性抛出，一轮看到全部错误字段。用 `pytest.assume` 或自己收集错误列表实现。

**10. 避免规则过度复杂**：断言只校验 **业务关心的关键字段**，不要把整个响应逐字段断死——否则后端加个字段全挂红，维护成本爆炸。

## Q5：日志模块封装

**1. `logging.getLogger(__name__)` 里的 `__name__`**：当前模块名（如 `api.order_api`），用它建 logger 能在日志里看到 **是哪个模块打的**，也方便按模块控制级别。

**2. 日志级别**：DEBUG / INFO / WARNING / ERROR / CRITICAL。

**3. 输出**：控制台 + 文件 **两者都要**——控制台看实时，文件留存归档/给 Allure。用两个 Handler 实现。

**4. 切割**：按大小或日期切割，用 `RotatingFileHandler` / `TimedRotatingFileHandler`，防止单文件无限膨胀。

**5. 重复打印**（高频坑）：根因是给 logger **重复添加了 handler**，或同时配了 root logger。解决：handler 只配置一次（单例/模块级初始化），并设 `logger.propagate=False` 防止向上冒泡重复输出。

**6.** 记录 URL、headers、请求体、响应体——接口测试排障的命脉。

**7. 脱敏**：和 Q2 一样，token/密码/身份证在打印前统一替换。

**8. 关联 Allure**：失败时把请求/响应/日志片段 `allure.attach` 进报告，报告里直接看，不用翻 log 文件。

**9. 并发区分用例**：日志格式里加 **caseId / traceId**（用 `LoggerAdapter` 或 `contextvars` 注入），否则并发执行时日志交织成一锅粥。

**10. traceId / caseId**：需要。traceId 串起 **前后端全链路**（和后端日志对齐定位问题），caseId 区分是哪条用例——排障效率的关键。
