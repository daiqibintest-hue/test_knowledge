# 七、精准测试（Q23–Q24）

> 这是最容易"讲得头头是道、一追问就穿"的题。下面按"真做过的人会怎么答"来写：重点不在流程图，而在 **两条技术路线为什么要并用、单用例覆盖的真实成本、会过期/漂移的坑、它抓不到什么、怎么验证没漏测**。

---

## Q23：如何理解精准测试

**一句话**：基于代码变更，用 **历史覆盖数据 + 调用关系** 反推出"这次改动真正影响到的最小用例集"，把回归从"全跑"变成"按需跑"。但它是 **优化项不是替代项**——理由见第 4 点。

### 1. 正向(静态) 和 反向(运行时) 是两套路子，且几乎必须都用

- **正向（静态调用图）**：变更方法 → 静态分析谁调用它 → 受影响接口 → 关联用例。
  **致命伤**：JVM 应用的静态调用图天生不全——Spring AOP 代理、MyBatis Mapper（只有接口没实现，trace 不到）、`@Async`/线程池、MQ 生产者→消费者解耦、反射、SPI、接口多态，都会让静态图 **低估** 影响面 → **漏测**。所以静态图绝不能单独拿来"减用例"，它更适合做"补充发现没有用例覆盖的新路径"。

- **反向（运行时覆盖）**：测试环境采集 **每条用例** 的方法级覆盖，建 `用例 → 覆盖到的方法` 的倒排索引。代码改了哪些方法，直接反查哪些用例覆盖过 → 选出来。
  它天然绕开了静态图的盲区：因为用例在运行时真实地跨过了 controller→service→dao→甚至 MQ 消费链，倒排索引里记录的是 **真实可达路径**。**这才是工程上的主干**，静态图是补充。

### 2. "每条用例的覆盖"才是真成本（最容易被问穿的点）

覆盖工具默认是 **整个进程累计** 覆盖，拿不到单用例粒度。要做到按用例：

- **Java/JaCoCo**：在 **每条用例前后** 通过 agent 的 `getExecutionData(reset=true)` 把这一段 exec 数据 dump 出来再清零，和"当前用例 ID"绑定。需要在测试框架挂钩子（JUnit `RunListener` / TestNG `IInvokedMethodListener`）。
- **Python/coverage.py**：更优雅——直接用 **dynamic contexts** 功能，`[run] dynamic_context = test_function`，coverage.py 会原生地把覆盖数据按"当时正在跑的测试"打标，省掉手动 dump/reset。

代价是：这套采集会 **显著拖慢** 测试执行（JaCoCo 频繁 dump、coverage.py 上下文记录都有开销），所以一般 **不在每次跑都开**，而是 **定期重建覆盖基线**（比如每晚全量跑一次带覆盖采集）。

### 3. 覆盖会过期、行号会漂移

覆盖是针对 **某一个 commit** 采的。代码一改，行号全变，**不能拿旧覆盖的行号去套新代码**。两个应对：

- 匹配粒度用 **方法签名**（`类#方法(参数)`）而不是行号——方法签名比行号稳得多。
- 基线 **定期重采**。**这套系统烂掉的头号原因就是基线不更新**：覆盖越来越旧 → 推荐越来越不准 → 最后没人敢信，沦为摆设。

### 4. 必须诚实承认它"抓不到"什么 + 三层安全网

精准测试基于 **历史覆盖** 选用例，天然盲区：

- **新代码路径 / 新交互**：没有任何老用例覆盖过 → 选不出来。新行为只能靠新用例，精准选择无中生有不了。
- **非代码变更**：DB schema、配置、开关、基础设施——根本不在调用图里。
- **并发 / 时序 / 状态累积** 类 bug：方法级行覆盖根本表达不了。

所以一定保留三层安全网，精准测试只在这之上做优化：

1. **核心链路冒烟无条件全跑**（登录/下单/支付这种，不管 diff 是什么）。
2. **夜间全量回归兜底**。
3. **高爆炸半径规则**：动了基础工具类 / 公共 SQL / 拦截器 / DTO 基类 → 直接 **升级为全量**，不走精准。

> 谁敢只讲"我们现在只跑 10% 用例"而不讲这三层兜底，谁就在等一个线上 P0。

### 5. 怎么证明它没在偷偷漏测（没这步就是耍流氓）

- **缺陷回放**：把过去 N 个月的逃逸缺陷拉出来，对每个缺陷当时那次提交跑一遍精准选择，看 **会不会选中能抓到它的用例**，统计 **漏选率（false-negative rate）**。
- **影子运行**：上线初期，精准集和全量集 **并行跑一段时间**，凡是"全量挂了但精准没选中"的用例立刻告警。漏选率降到可接受（比如 < 1%）后，才让精准测试去 **卡流水线**。

## Q24：代码变更 ↔ 用例映射的工程实现

### 数据模型（核心就这几张表）

```
method_coverage   (baseline_commit, test_case_id, class, method_sig)   -- 倒排索引：用例覆盖了哪些方法
case_meta         (test_case_id, module, covered_apis[], priority, last_result, flaky)
change_impact     (commit, changed_method, impacted_case_ids[], source)  -- source: coverage / static / rule
```

倒排索引 `method_coverage` 是心脏；`change_impact` 是每次提交算出来的结果。

### 第一步：变更 → 方法（为什么不能用行号 diff）

`git diff` 给的是 **变更行号区间**。但"行 40 改了"不等于"方法 foo 改了"——foo 可能跨 35~60 行。所以：

1. 在 **新 commit** 解析文件 AST，拿到每个方法的 **行号区间**；
2. 把 diff 的变更行 **落到所属方法**；
3. 特殊处理：改了方法签名本身 / 新增删除方法 / 改了字段或常量（影响 = 所有引用它的方法）/ 改了共享 SQL、配置、模板（影响面可能巨大 → 这类文件直接走全量）。

### 第二步：方法 → 用例（覆盖倒排 + 静态补充）

- 主路：拿变更方法集去 `method_coverage` 反查，得到覆盖过这些方法的用例 → 这是 **有历史证据** 的推荐。
- 补充：用静态调用图找出"变更方法可达、但当前没有任何用例覆盖"的接口 → 这些是 **覆盖缺口**，标记出来提示"需要补用例"，而不是假装已覆盖。

### 第三步：方法 → 接口（给报告用，也给"接口级用例"用）

解析路由注解（Spring 的 `@RequestMapping`/`@GetMapping`、或框架路由表），把 Controller 方法映射到 **接口路径**；接口再通过 `case_meta.covered_apis` 关联到接口级用例。这样能输出"本次改动影响了哪几个接口、各自有哪些用例"。

### 第四步：历史缺陷加权 + 映射不准的修正闭环

- **历史缺陷加权**：高缺陷密度的模块即使影响分析只是弱命中，也 **提高其用例的入选优先级**——缺陷是聚集的，碰过的地方更可能再出问题。
- **修正闭环**：用执行结果反哺——如果某次精准集 **漏掉** 了后来暴露的 bug，把"该缺陷涉及的用例 ↔ 该变更"补进映射；如果某用例总是被选中又从不相关，降低其关联权重。映射关系是 **持续迭代** 出来的，不是一次建好就完事。

### 第五步：接入流水线（怎么"卡"又不误杀）

- **MR/提交触发**：算出精准集 + 核心冒烟，跑这两部分作为合并门禁（快，几分钟）。
- **合并后/夜间**：跑全量兜底。
- **降级开关**：影响分析失败、覆盖基线过期超过阈值、或命中高爆炸半径规则 → 自动 **降级为全量**，宁可多跑不可漏跑。

### 完整链路

```
git diff → AST 定位变更方法 ──┬─→ 覆盖倒排反查(主) ─┐
                              ├─→ 静态调用图(补:找覆盖缺口) ─┤→ 候选用例集
                              └─→ 高爆炸半径规则(命中→全量) ─┘
                                          │
                历史缺陷加权 + 核心冒烟必跑 ┘ → 最终执行集 →（缺陷回放/影子运行验证漏选率）→ 卡流水线
```
