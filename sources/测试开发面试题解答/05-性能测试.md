# 五、性能测试（Q16–Q19）

---

## Q16：用 JMeter 做过哪些类型的性能测试

**1. 各类型区别**：

- **基准测试**：单用户/小并发，拿到单接口性能基线。
- **负载测试**：逐步加压，找系统在预期负载下的表现，确定能稳定支撑的并发。
- **压力测试**：持续加压直到系统崩溃，找极限和拐点。
- **容量测试**：评估系统最大处理能力，为扩容提供依据。
- **稳定性测试（疲劳测试）**：较高负载长时间运行（如 8/24h），看有没有内存泄漏、性能衰减。

**2. 单接口 vs 全链路**：核心接口先单压定位单点能力，再按真实比例做混合场景全链路压测，更贴近线上。

**3. 登录 token 处理**：压测不重复登录——提前批量登录拿一批 token 存 CSV，用 `CSV Data Set Config` 参数化，或建 token 池。

**4. 库存/订单不冲突**：提前造足量数据 + 参数化唯一数据（每个线程用不同账号/商品），避免互相抢同一行锁。

**5. 支付接口是否直接压**：一般 **不压真实支付**（涉及真实资金和第三方），用 **挡板/mock** 替代支付网关。

**6. 避免污染生产**：用 **独立压测环境** 或 **影子库/影子表**，绝不在生产真实库直接造压测数据。

**7. 阶梯加压**：用 `ramp-up` 或阶梯线程组（Stepping/Concurrency Thread Group），逐级加并发观察拐点，而不是一上来全量并发。

**8. 确定目标并发**：按生产峰值 QPS/TPS + 余量倒推，或用"日活 × 峰值系数 / 时间窗"估算。

**9. 模拟真实行为**：加 **思考时间**（Timer）、参数化数据、集合点（Synchronizing Timer）、按真实业务比例配置事务占比。

**10. 为什么不用大量监听器**：GUI 模式 + 大量监听器（如"查看结果树""图形结果"）本身极耗 CPU/内存，会 **干扰压测结果**。正确做法：**非 GUI 模式** 跑（`jmeter -n -t`），结果落 jtl 事后分析。

## Q17：性能测试核心指标

**1. TPS vs QPS**：

- **TPS**：每秒事务数，一个事务可能含多个请求（业务维度）。
- **QPS**：每秒查询/请求数（请求维度）。
- 一次"下单事务"可能产生多个 QPS。

**2. 并发用户数与 TPS 关系**：利特尔法则 —— `TPS ≈ 并发用户数 / (平均响应时间 + 思考时间)`。没有思考时间的简化场景下退化为 `TPS ≈ 并发用户数 / 平均响应时间`。所以并发高不代表 TPS 高，响应时间是关键变量；加了思考时间后，同样并发数的 TPS 会更低。

**3. 平均/P90/P95/P99**：P95 = 95% 的请求响应时间在该值以内。它们描述响应时间分布。

**4. 为什么不能只看平均响应时间**：平均值会被 **长尾掩盖**——平均 100ms 看着很好，但 P99 可能 5s，意味着 1% 用户体验极差。要看分位值。

**5. 错误率**：失败请求占比，一般要求 < 0.1%~0.5%（按业务定）。错误率升高往往是系统到瓶颈的信号。

**6. 达到拐点的表现**：并发再加，**TPS 不再上升甚至下降，响应时间陡增，错误率上升**——这就是容量拐点。

**7. CPU/内存/磁盘IO/网络IO**：分别反映计算、内存、磁盘读写、网络传输是否成为瓶颈。

**8. 判断容量上限**：综合"TPS 拐点 + 资源是否打满 + 响应时间/错误率是否劣化"。

**9. 是否只关注前端接口**：**不是**。要关注全链路 + 后端资源（DB、缓存、MQ、下游服务），瓶颈往往在后端而非接口本身。

## Q18：压测中 TPS 上不去如何定位

**先讲方法论，不要上来就猜。** TPS 上不去在数学上只有两种根因，先分清是哪一种再下钻：

> 由利特尔法则 `TPS = 有效并发 / 响应时间(RT)`，TPS 不涨 ⟺ 要么 **RT 在变长**，要么 **有效并发被卡死**（线程/连接/锁池满了，加压根本加不进去）。

所以我的定位顺序是：**先拆 RT、再看饱和度**，而不是一上来看 CPU。

### 第 0 步：先排除压力机自己

很多"TPS 上不去"是 **JMeter 单机** 先到瓶颈了——压力机 CPU 打满、或端口耗尽（大量 `TIME_WAIT`）。先确认压力机 CPU < 60%、用分布式压力机，再谈被测系统。**这一步不做，后面全是错的。**

### 第 1 步：拆 RT，看时间花在哪一段（最快的定位手段）

用链路追踪（SkyWalking/Zipkin）把单请求 RT 拆成 `网关 → 应用 → DB → 下游` 各段耗时。哪段占大头就下钻哪段——这比盲目看一堆机器指标快得多。

### 第 2 步：按"两种根因"分别下钻

**A. RT 变长型**

- **CPU 高（>80%）**：先分 `us` 还是 `sy`（`top` 看）。`us` 高 → 业务/算法热点，上 **火焰图**（async-profiler）找最耗 CPU 的方法；`sy` 高 → 线程过多、上下文切换频繁或频繁系统调用。
- **GC**：`jstat -gcutil <pid> 1000` 看 GC 频率和耗时。**Full GC 频繁**（比如几秒一次、每次几百 ms）→ 老年代回收不掉 → 内存泄漏或单对象过大 → `jmap` dump 堆 + MAT 分析。表现是 TPS **周期性抖动**、RT 出现 **尖刺**。
- **慢 SQL**：慢查询日志 + `explain`（看 `type=ALL` 全表、`key=NULL` 没走索引、`rows` 巨大）。`show processlist` 看是否大量 `Sending data` 或锁等待。

**B. 有效并发卡死型（CPU 不高 <40% 但 RT 高、TPS 平）—— 实战里最常见**

这是 **等待型瓶颈**，机器没用满却上不去。手段：**连着 `jstack <pid>` 打几次**，看线程大量停在哪：

- 大量 `BLOCKED` 在某个 `synchronized` / 分布式锁 → **锁竞争**。
- 大量 `WAITING` 在 `getConnection`（Druid/HikariCP）→ **数据库连接池太小**。Druid 监控看 `active == max` 且有 `wait`。
- Tomcat `maxThreads` 满 → 请求在队列排队，RT 线性上涨。
- 卡在调用下游 → 下游慢，回到第 1 步看链路。

> ⚠️ 关键取舍：连接池/线程池调大 **不是免费的**——把应用的池调大，压力会直接传导到 DB，要先确认 DB 扛得住，否则只是把瓶颈从应用挪到 DB。

### 第 3 步：用阶梯加压的曲线形态定性

- RT 平稳、TPS 随并发 **线性上涨** → 还没到瓶颈，继续加。
- TPS 不再涨、**RT 陡增** → 到了 **容量拐点**，记下此时并发数即为容量上限。
- TPS 平了、**但 RT 也平、资源也都不满** → 典型的 **"假饱和"**：被某个池或 **限流** 卡住了，去查线程池/连接池/限流配置。

### 诚实的经验值

**80% 的"TPS 上不去"，最后查出来是 连接池/线程池配小了、一条慢 SQL、或一个同步锁，而不是机器不够。** 所以遇到这题先别喊"加机器"——先拆 RT、再 `jstack` 看线程卡在哪。能把这套顺序讲出来，比背十条症状有用得多。

## Q19：压测中出现大量 HTTP 500 如何排查

**1. 全部接口还是单个接口**：先定位范围——全局 500 多是公共依赖（DB/中间件）挂了，单接口 500 是该接口逻辑问题。

**2. 从哪个并发量开始出现**：低并发就有 → 代码 bug；高并发才出现 → 资源/容量问题。

**3. 是否随并发增加而增加**：正相关说明是资源耗尽类问题。

**4. 应用日志报什么**：**这是最关键的一步**——直接看异常堆栈，500 的根因几乎都在应用日志里。

**5. 数据库连接是否打满**：`Too many connections` / 获取连接超时。

**6. 是否请求超时**：下游超时导致上游报错。

**7. 空指针/数组越界等代码异常**：高并发暴露的边界 bug。

**8. 是否死锁**：数据库行锁/死锁，日志会有 `Deadlock found`。

**9. 是否触发限流/熔断/降级**：被保护机制拦截后返回错误（注意这类可能是 429/503，但配置不当也会 500）。

**10. 是否下游服务异常导致**：依赖的服务挂了，异常向上传导成 500。
