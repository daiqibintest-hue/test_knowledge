# 二十、高频短板补强

> 这一章专门补强原题库里容易被一句话带过、但面试官很爱深挖的短追问。

---

## Q77：session 级 fixture 一般放在哪里？

一般放在项目根目录或测试目录下的 `conftest.py`，让多个测试模块自动共享，不需要手动 import。比如登录获取 token、初始化测试环境、创建数据库连接池这类跨模块复用的前置条件，适合做成 session 级 fixture。

但不是所有前置都适合 session。若数据会被用例修改，或者不同模块需要不同账号和权限，就不要强行做全局 session，否则会造成用例互相污染。我的原则是：稳定、只读、成本高的前置用 session；会变更、有隔离要求的前置用 function 或 class。

## Q78：不同业务模块需要不同前置条件时如何组织？

不要把所有 fixture 都塞进一个巨大的 `conftest.py`。可以按模块分层：公共能力放根目录 `conftest.py`，例如环境配置、日志、通用 token；业务模块自己的前置放模块目录下的 `conftest.py`，例如订单模块造订单、支付模块造支付单。

如果多个模块共享但参数不同，可以做 fixture 工厂，通过参数传入角色、状态、数据模板。这样既复用能力，又避免不同业务互相影响。面试里可以补一句：前置条件设计的核心不是写得少，而是隔离清楚、失败好定位、数据能清理。

## Q79：多个参数如何组合？

pytest 可以用多个 `@pytest.mark.parametrize` 形成笛卡尔积，也可以把多字段组合成一个参数对象。是否使用笛卡尔积取决于业务是否真的需要全组合覆盖。

如果是协议层或算法层，笛卡尔积能系统覆盖边界；但业务接口常常不适合全组合，因为很多组合没有业务意义，还会让用例数量膨胀。更稳妥的方式是基于等价类和风险挑组合：核心正常链路、典型异常、边界值、高风险历史缺陷。参数化不是为了生成越多用例越好，而是为了让同一类逻辑更容易维护。

## Q80：marker 设计过多会带来什么问题？

marker 太多会导致筛选混乱，团队不知道该跑哪些用例，甚至同一含义出现多个标签，比如 `smoke`、`p0`、`core` 混用。结果是流水线配置越来越复杂，质量门禁失去一致性。

我的做法是把标签控制在几个维度：优先级如 `p0/p1/p2`，执行场景如 `smoke/regression/nightly`，专项类型如 `api/ui/performance/security`，稳定性如 `flaky`。新增 marker 要在 `pytest.ini` 注册并写清楚含义。标签本质是调度语言，必须让全团队有统一语义。

## Q81：如何只执行 smoke 用例？

前提是 smoke 用例要统一打 `@pytest.mark.smoke`，并在 `pytest.ini` 注册。执行时使用 `pytest -m smoke`。如果要排除慢用例，可以组合表达式：`pytest -m "smoke and not slow"`。

在 CI 里我一般把 smoke 放在提测或合并门禁，要求快、稳定、覆盖核心链路。注意 smoke 不是把所有 P0 都塞进去，而是验证系统是否具备继续测试的基本条件，例如登录、核心查询、核心写入、关键链路。

## Q82：如何排除 smoke 用例？

可以用 `pytest -m "not smoke"`。这个场景常用于夜间回归拆分：白天门禁跑 smoke，夜间全量时也可以把 smoke 和非 smoke 分成不同 job 并行跑，减少总耗时。

需要注意的是，排除 smoke 不代表排除核心风险。全量回归报告里仍然要汇总 smoke 结果和非 smoke 结果，否则容易出现核心用例失败但被另一个任务掩盖的情况。

## Q83：数据库连接池不足有什么表现？

典型表现是接口 RT 上升但数据库 CPU 不一定高，应用日志里可能出现获取连接超时、连接等待、线程堆积。压测时会看到 TPS 上不去，P95/P99 急剧升高，错误里出现 timeout 或 too many connections。

定位时要同时看应用连接池指标和数据库连接数。比如 HikariCP 可以看 active、idle、pending、timeout；MySQL 看当前连接数、最大连接数和慢 SQL。连接池不足不一定只靠调大解决，如果 SQL 慢或事务持有连接太久，盲目调大会把压力转移到数据库。

## Q84：Allure 报告如何帮助定位问题？

Allure 不只是好看的报告，它的价值是把失败现场结构化保留下来。接口自动化里我会把请求 URL、method、headers 脱敏后、请求体、响应体、状态码、耗时、traceId、数据库校验结果都挂到附件里。

这样失败时可以快速判断：是参数构造错、鉴权失败、接口返回异常、断言过严，还是数据未落库。和 Jenkins 结合后，还可以保留历史趋势，观察失败是新增问题还是长期 flaky。注意敏感字段必须脱敏，不能把 token、身份证、手机号完整写进报告。

## Q85：质量门禁应该卡哪些指标？

门禁不能只卡自动化通过率，否则容易为了通过率删除高风险用例。常见门禁包括：冒烟用例 100% 通过，P0/P1 自动化通过率达标，新增严重缺陷为 0，代码扫描无高危问题，接口覆盖核心链路，性能基线未明显下降。

不同阶段门禁强度不同。开发自测阶段可以偏快，卡 lint、单测和核心接口；提测阶段卡冒烟和环境准入；合并发布阶段卡自动化、未关闭缺陷、性能和安全风险。门禁要允许人工豁免，但必须记录风险、负责人和补救措施。

## Q86：自动化用例失败后如何做归因？

先不要默认是产品 bug。归因可以分五类：脚本问题、测试数据问题、环境问题、依赖服务问题、真实产品缺陷。

排查顺序：看失败是否集中在某个环境或某个接口；看错误类型是断言失败、连接超时、鉴权失败还是 5xx；看同一用例重跑是否稳定；看最近是否有需求、接口或数据变更。平台化后可以用响应码、异常关键字、traceId、历史失败模式做初步分类。归因的价值是减少无效沟通，让研发只处理真实缺陷，让测试修自己的脚本和数据问题。

## Q87：flaky 用例怎么治理？

flaky 用例是偶发失败、重跑又通过的用例。治理第一步是识别，而不是简单重试。可以统计近 N 次执行结果，失败率不稳定的打上 `flaky` 标签，从质量门禁中隔离，但不能从报告里消失。

常见原因包括环境不稳定、等待时间写死、数据互相污染、接口异步未完成、第三方依赖波动、并发执行顺序问题。治理方式是显式等待、数据隔离、Mock 不稳定依赖、清理前后置、增加唯一业务键。重试只能兜底，不能替代修复。

## Q88：测试左移如何避免变成口号？

测试左移不是测试更早加班，而是把质量活动前置到需求、设计和开发阶段。落地动作包括：需求评审时输出风险清单和验收标准；接口设计阶段评审字段、状态码、幂等和权限；开发阶段提供 Mock、契约测试和冒烟清单；合并阶段接入自动化门禁。

能否落地看三个指标：缺陷发现阶段是否前移，提测打回率是否下降，线上逃逸缺陷是否减少。没有指标和流程承接的"左移"只是口号。
