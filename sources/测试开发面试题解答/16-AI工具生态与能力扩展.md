# 十六、AI 工具生态与能力扩展（Q44-Q47）

---

## Q44：Function Tools、MCP、Skills、Plugins 分别是什么

这几个概念可以放在 **能力扩展与工具接入层** 里理解，它介于 Agent 决策和工程化治理之间。

**1. Function Tools**：单个可调用能力，通常由函数名、description、参数 JSON Schema 和返回结构组成。模型通过 Function Calling / Tool Calling 选择工具并生成入参。

**2. MCP**：标准化工具接入协议，把外部系统能力封装成 MCP Server 暴露给 MCP Client。它不只包含 Tools，也包含 Resources 和 Prompts，适合接数据库、文件系统、浏览器、企业内部系统。

**3. Skills**：任务方法包 / 操作手册 / 能力封装，通常包含 SKILL.md、触发条件、操作流程、脚本、模板和参考资料。Skill 告诉 Agent **遇到某类任务应该怎么做**。

**4. Plugins / Extensions**：把 Skills、MCP Server、App UI、配置清单等能力打包分发，便于安装、升级和组织级复用。

**5. ToolOps**：工具注册、发现、路由、限流、日志、审计、权限、安全和成本统计。它解决工具变多之后如何治理的问题。

关系可以这样答：

```
Function Tools = 单个可调用能力
MCP = 标准化工具接入协议
Skills = 完成某类任务的方法包
Plugins = 把 MCP / Skills / App 能力打包分发
Agent = 决定什么时候用这些能力
ToolOps = 治理这些能力如何被安全、稳定、可观测地使用
```

## Q45：Function Calling 和 MCP 怎么选型

核心看 **工具规模、复用范围、接入对象和治理需求**。

**1. 简单项目用 Function Calling 就够**：如果只是当前服务里几个本地函数，比如查订单、生成摘要、计算字段，用模型厂商自带的 Function Calling / Tool Calling 成本最低。

**2. 多客户端复用适合 MCP**：如果同一套工具要给 Claude Desktop、Cursor、内部 Agent 平台、测试平台一起用，MCP 更合适，因为工具实现一次，多个兼容 Client 都能发现和调用。

**3. 外部系统接入适合 MCP**：数据库、文件系统、Jira、GitHub、Playwright 浏览器、企业微信这些能力，用 MCP Server 封装更清晰。

**4. 强治理场景适合 MCP + ToolOps**：当需要工具发现、权限分级、审计日志、只读资源、危险操作确认、远程部署时，MCP 比零散函数更容易平台化。

**5. 两者不是互斥**：一个 Agent 可以同时有厂商 Function Tools，也可以连接 MCP Server；甚至 MCP Tool 的内部实现也是普通函数。

面试表达：**Function Calling 更像模型内置的调用机制，MCP 更像跨模型、跨客户端的工具接入标准**。

## Q46：Skills 如何和 MCP、Agent 配合

Skills 更像 **流程和知识层**，MCP 更像 **工具和数据接入层**，Agent 负责根据任务选择并执行。

**1. Agent 识别任务**：用户说“帮我评估这个 RAG 问答效果”，Agent 先判断这是 RAG 评测任务。

**2. Agent 加载 Skill**：根据 description 匹配到 RAG 评估 Skill，读取 SKILL.md，知道要准备 Golden Set、调用评测脚本、计算 context recall / faithfulness、生成报告。

**3. Skill 调用 MCP / 工具**：评测过程中可能调用文件系统 MCP 读取测试集，调用 HTTP 工具请求 RAG 服务，调用脚本工具运行 RAGAS，调用企业微信 MCP 发送报告。

**4. Agent 汇总结果**：把工具返回的结构化结果整理成报告，说明退化项、失败样本和优化建议。

**5. 权限由工具层兜底**：Skill 不能绕过 MCP 的权限；数据库只读、生产操作审批、危险工具白名单都应在工具层和平台层控制。

一句话：**Skill 告诉 Agent 怎么做，MCP 给 Agent 能做什么，Agent 决定何时做，ToolOps 保证做得可控**。

## Q47：Tool Registry、Tool Router、权限沙箱怎么设计

工具一多，难点不是“能不能调”，而是 **能不能选对、调稳、管住、追得回**。

**1. Tool Registry 工具注册中心**：记录工具名、description、参数 schema、返回 schema、所属系统、权限等级、是否只读、是否危险、版本、负责人和 SLA。

**2. Tool Router 工具路由**：根据任务意图、用户角色、当前环境、工具健康度和成本选择工具；多个工具都能完成时优先选低风险、低成本、成功率高的。

**3. 参数校验**：所有入参先过 JSON Schema / Pydantic 校验，再做业务规则校验，比如金额上限、项目 ID 白名单、SQL 只允许 SELECT。

**4. 权限沙箱**：按 Agent 角色分配工具白名单；测试环境和生产环境隔离；生产默认只读；危险动作必须 dry-run + 人工确认。

**5. 调用观测**：记录 traceId、用户、Agent、工具名、参数摘要、耗时、返回状态、错误码、token / 成本和后续动作。

**6. 失败治理**：区分可重试错误和不可重试错误，设置最大重试次数、熔断、降级和人工接管，避免 Agent 死循环。

**7. 审计合规**：敏感参数脱敏存储，关键操作保留审批记录，工具版本变更要可追溯。

测试开发视角要强调：**工具接入不是越多越好，真正可用的工具必须有 schema、权限、日志、失败处理和审计闭环**。
