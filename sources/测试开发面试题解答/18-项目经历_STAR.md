# 十八、项目经历与 STAR 回答

> 这一章不是为了背概念，而是把已有知识点串成能在面试里讲清楚的项目故事。回答时建议按 STAR：背景、任务、行动、结果，再补风险和复盘。

---

## Q56：如何介绍一个接口自动化框架项目，才不像只是在背工具？

可以按"业务痛点 -> 架构设计 -> 关键实现 -> 结果指标"来讲。

背景：原来回归主要靠人工，每次版本需要 2 到 3 天，接口改动后经常出现核心链路漏测。我的目标不是简单写脚本，而是把核心接口回归沉淀成稳定、可维护、可接入流水线的框架。

行动上我做了四层设计：请求层统一封装 base_url、headers、token、超时和重试；数据层用 YAML 管理用例和参数，环境配置单独隔离；断言层支持状态码、字段值、JSONPath、数据库落库校验；报告层接入 Allure，把请求、响应、traceId、失败截图或日志附件带出来。对登录、下单、支付这类链路，用上下文池保存 token、orderId 等动态参数。

结果可以量化：核心接口覆盖从 0 提升到 100+，冒烟从半天压到 20 分钟以内，CI 中每次合并自动跑 P0/P1 用例。面试时要主动说明边界：不是所有接口都适合自动化，强依赖三方、数据难清理、频繁变更的接口需要先做 Mock、造数和稳定性治理。

## Q57：如何讲一个性能瓶颈定位项目？

先讲现象，再讲排查路径，最后讲结论和验证。

背景：某业务接口压测到 200 并发后 TPS 不再上涨，P95 从 300ms 上升到 2s，错误率开始升高。我的第一步不是猜数据库，而是先排除压力机瓶颈，确认 CPU、网络、连接数、JMeter non-GUI 模式都正常。

定位路径分三层：入口层看 Nginx/网关是否限流、连接数是否打满；应用层看线程池、GC、慢日志、接口内部耗时埋点；数据层看慢 SQL、锁等待、连接池和索引命中。最后发现主要耗时在订单列表查询，SQL 对状态和创建时间组合查询没有走合适索引，同时应用连接池偏小导致高并发下等待。

处理方式：研发加组合索引并调整查询字段，连接池从 20 调到 80，同时增加慢 SQL 告警。复测时要用同样脚本、同样数据量、同样阶梯加压模型对比，不能只说"优化后变快了"。结果表达可以是：TPS 从 180 提升到 420，P95 降到 450ms，错误率稳定在 0.1% 以下。

## Q58：如何讲一个自动化测试平台项目？

平台项目不要讲成"把脚本放到网页上运行"。要从任务调度、执行资源、数据模型、报告和权限讲。

背景：团队有多套自动化脚本，执行入口分散，报告格式不统一，失败后定位成本高。平台目标是把用例管理、执行调度、环境选择、报告聚合、通知和质量门禁统一起来。

核心设计包括：用例表、任务表、执行记录表、环境表、报告表；任务调度支持立即执行、定时执行和流水线触发；执行机用 agent 模式拉取任务，避免平台直接 SSH 到机器；报告层聚合 pytest、JMeter、Playwright 的结果；权限层区分管理员、项目负责人、普通执行人。

被追问高可用时，可以说平台服务无状态化，任务状态落库，执行机心跳上报，超时任务可重新分配；报告和附件放对象存储；任务队列可以用 Redis Stream 或 MQ。真正的难点是失败归因和稳定性治理，比如环境问题、脚本问题、产品缺陷要分开统计，否则平台只会变成"失败列表"。

## Q59：如何讲一个 RAG/AI 测试项目？

重点不要放在"我测了大模型"，而是讲清楚 RAG 链路每一层怎么测。

背景：业务知识库问答上线后，问题不是接口通不通，而是答案是否基于知识库、是否引用正确、是否胡编。我的测试方案分为文档解析、chunk 切分、检索召回、rerank、生成答案、引用校验和安全对抗。

行动上先做 Golden Dataset，覆盖高频问题、边界问题、知识库外问题、旧版规则问题和诱导问题。检索层看 Top1/Top3 命中率、context recall；生成层看 faithfulness、答案相关性、拒答是否正确；安全层看 Prompt Injection、越权检索和敏感信息泄露。

结果不是只给一个"准确率"，而是分层报告：检索失败多少、生成幻觉多少、引用错误多少、拒答错误多少。这样研发能知道该改 chunk、embedding、rerank、prompt，还是模型参数。复盘时可以补一句：AI 测试不是追求一次性判对错，而是建立可回归、可对比、可解释的质量基线。

## Q60：如何讲一次线上问题复盘，体现测试开发能力？

可以按"影响、止血、定位、修复、预防"讲。

例如一次通知策略上线后，部分用户收到重复通知。止血动作是先关闭新策略开关，保留日志和消息记录；定位时按 traceId 串起策略服务、消息队列、通知服务和数据库状态，发现重试任务没有正确识别幂等键，导致 MQ 消费超时后重复投递被当成新任务。

修复上，研发补了业务幂等键和唯一索引，消费端先查幂等记录再发送，发送成功后记录状态。测试侧补了 MQ 重复投递、消费超时、服务重启、并发消费的专项用例，并把通知发送次数加入监控。

面试里要强调复盘产物：缺陷不是关了就结束，要沉淀自动化回归、监控告警和上线 checklist。这样体现的是质量体系能力，而不是单点找 bug。

## Q61：如何回答"你在项目里最大的技术亮点是什么"？

不要泛泛说"封装得好"。建议选一个能讲出复杂度和结果的点。

示例回答：我认为亮点是把接口自动化从单接口校验升级成了业务链路校验。以前只看接口返回 200，漏掉了数据未落库、状态未流转、通知未触发这类问题。我做了上下文参数池、数据库校验、日志 traceId 关联和 Allure 附件，把登录、下单、支付、通知串成链路。

亮点背后的价值是：失败时能快速判断是鉴权、参数、业务状态、数据库还是下游服务问题。结果是核心链路回归时间缩短，线上回归缺陷减少。最后补边界：链路用例不能太多，否则执行慢、数据清理复杂，所以我只把 P0 路径和高风险路径做成链路自动化，普通分支仍然用单接口和参数化覆盖。

## Q62：如何回答"这个项目有什么不足"？

不足要真实，但不能把自己说垮。结构是：承认问题、说明影响、给出改进方案。

例如：目前框架对失败归因还不够自动化。Allure 能展示请求和响应，但环境异常、数据问题、真实缺陷还需要人工判断。影响是失败用例多时排查成本高。

改进方案是给执行结果增加失败分类：连接超时、断言失败、依赖数据缺失、鉴权失败、服务端 5xx；结合日志关键字和响应码做初步归因，再由人工确认。长期看可以统计 flaky 用例，把不稳定用例从质量门禁里隔离出来单独治理。

这样回答既诚实，也说明你知道下一步怎么工程化。

## Q63：如何把"负责测试平台"讲成负责人能力？

负责人能力不只是写代码，还包括目标拆解、风险识别、跨团队推进和结果交付。

可以说：我先把平台目标拆成三期。第一期解决统一执行和报告，先让团队愿意用；第二期接入 Jenkins 和定时任务，形成质量门禁；第三期做失败归因、用例标签、趋势报表和权限。每期都有可验收指标，比如接入项目数、执行成功率、报告生成耗时、失败定位耗时。

推进上，需要和研发约定接口文档、测试账号、环境稳定性，也要和项目经理对齐哪些用例进门禁、失败后谁处理。上线后我关注平台自身稳定性，比如任务积压、执行机离线、报告丢失、通知失败。这样讲能体现你不仅能实现功能，也能把平台真正落到团队流程里。

## Q64：如何回答"你做的 AI 测试和普通测试有什么区别"？

普通测试更多验证确定性规则，输入相同通常输出相同；AI 测试要面对概率输出、语义正确性、知识来源、上下文和安全边界。

区别体现在四点：第一，断言方式不同，不能只判断字段等于某个值，要引入语义评分、人工标注集、引用校验和规则校验；第二，质量指标不同，除了通过率，还看准确率、召回率、幻觉率、拒答率、Prompt Injection 通过率；第三，定位方式不同，要拆成检索问题、生成问题、prompt 问题、知识库问题；第四，回归方式不同，模型、prompt、知识库任一变化都要跑基线集对比。

最后要落到工程化：AI 测试不是让模型自己判断自己，而是用规则、样本集、人工抽检和自动化评估共同形成质量闭环。
