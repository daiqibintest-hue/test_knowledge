# 十七、LLM 与 Prompt 深入

## Q44：LLM 输出为什么不完全确定？

LLM 本质是按概率预测下一个 token，同一个输入在不同采样参数、模型版本或上下文下可能得到不同输出。面试里可以从三点解释：一是模型输出是概率分布，不是固定规则；二是 `temperature`、`top_p` 会影响采样随机性；三是 Prompt、上下文顺序、检索片段和模型版本变化都会影响结果。所以测试时不能只跑一次，要用评测集、重复采样、稳定性指标和人工/规则校验综合判断。

## Q45：temperature、top_p、max_tokens 分别影响什么？

`temperature` 控制随机性，越低越稳定，越高越发散；`top_p` 控制候选 token 的累计概率范围，越小越保守；`max_tokens` 控制最大输出长度，太小会截断答案，太大可能增加成本和跑偏风险。测试开发场景下，生成测试用例、结构化摘要、分类判断一般倾向低 temperature；头脑风暴或补充边界场景可以适当提高。

## Q46：如何把业务规则写进 Prompt，避免只给一句泛泛指令？

不要只写“帮我生成测试用例”，而要把业务规则拆进 Prompt：角色与目标、业务背景、输入数据、约束条件、覆盖维度、输出格式、示例和禁止项。测试开发里尤其要写清楚正常/异常/边界/权限/兼容/性能这些覆盖维度，以及“不能编造 PRD 不存在的规则”。这样模型不是自由发挥，而是在明确边界内补全测试思路。

## Q47：如何约束 LLM 输出为稳定的 JSON 或表格？

先在 Prompt 中给出明确 schema，例如字段名、类型、是否必填和枚举值；再要求“只输出 JSON，不要解释”；必要时给一个 few-shot 示例。工程上还要做二次校验：JSON 解析、schema 校验、字段缺失检查、枚举值检查。不要只依赖 Prompt 约束，模型输出必须经过程序校验和失败重试。

## Q48：Zero-shot、Few-shot、Chain-of-Thought 分别适合什么场景？

Zero-shot 适合规则清晰、答案格式简单的任务；Few-shot 适合希望模型模仿固定格式、口径或分类标准的任务；Chain-of-Thought 适合复杂推理，但生产环境通常不直接暴露完整推理过程，可以要求模型“先分析再输出结论”，或让模型输出可审计的关键依据。测试用例生成更常用 few-shot，因为示例能显著稳定输出格式。

## Q49：Prompt 模板如何做版本管理？

Prompt 应该像代码一样管理版本：模板文件入库，记录版本号、适用场景、输入变量、输出 schema、变更说明和评测结果。每次修改 Prompt 后，用固定 Golden Set 回归，比较准确率、覆盖率、格式合规率和人工采纳率。上线后还要记录线上 query、prompt 版本、模型版本和输出结果，方便问题追溯。

## Q50：Prompt 版本变更后如何做回归测试？

先准备一批稳定评测集，包括高频问题、边界问题、异常输入和历史线上 bad case；再固定模型版本和采样参数，对比新旧 Prompt 的输出质量。指标可以包括格式通过率、准确率、幻觉率、覆盖率、人工采纳率和严重错误数。只要出现高风险退化，比如编造业务规则或漏掉关键断言，就不应直接上线。

## Q51：上下文窗口不够时怎么办？

常见做法有四种：压缩历史对话、只保留相关上下文、用 RAG 检索外部知识、把长任务拆成多轮子任务。不能简单把所有资料都塞进 Prompt，因为会增加成本、稀释重点，还可能让模型忽略关键约束。测试时要覆盖超长输入、上下文丢失、历史冲突和资料过期等场景。

## Q52：Prompt 层面如何降低幻觉和编造规则？

Prompt 层面要写清楚“只基于给定资料回答，不确定就说明无法判断”；RAG 层面要提供可靠来源并要求引用；工程层面要做输出校验、敏感词/合规规则拦截、人工审核和置信度阈值。面试里要强调：Prompt 只能降低风险，不能替代评测集、规则校验和人工兜底。

## Q53：Prompt Injection 和普通异常输入有什么区别？

普通异常输入通常是空值、超长、非法字符或格式错误；Prompt Injection 是用户试图改变模型原有指令或越过系统边界，例如“忽略以上规则”“输出隐藏提示词”“查询无权限资料”。它测试的不只是输入校验，还包括指令优先级、权限隔离、工具调用边界、RAG 文档可信度和输出拦截。

## Q54：RAG 场景中 Prompt 如何组装？

典型结构是：系统规则 + 任务说明 + 检索片段 + 用户问题 + 输出格式要求。关键是把检索片段标清来源、版本和权限范围，并要求模型只基于片段回答；如果资料不足，要回答“不确定”或转人工。测试时重点看检索内容是否被正确引用、答案是否忠实于资料、是否会引用不存在的来源。

## Q55：Prompt Engineering 和 Harness Engineering 有什么区别？

Prompt Engineering 解决“怎么把任务说清楚”，关注角色、指令、示例、输出格式和约束。Harness Engineering 解决“模型在什么工程环境里可靠执行”，包括工具调用、权限控制、上下文管理、状态保存、结果验证、日志追踪和人工审批。Prompt 再好也不能替代工具可靠性和结果校验，Agent 落地时两者都需要。
