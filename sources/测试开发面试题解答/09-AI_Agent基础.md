# 九、AI Agent 基础（Q27–Q29）

---

## Q27：什么是 AI Agent

AI Agent 是 **以 LLM 为"大脑"，结合规划（Planning）、工具调用（Tools）、记忆（Memory），能自主感知环境、做决策并执行任务，直到完成目标的系统**。它不是单次问答，而是"感知→思考→行动→观察"的循环。

**1. Agent 和普通 LLM 区别**：普通 LLM 只能根据输入生成文本（一问一答）；Agent 能 **主动调用工具、改变外部世界、多步推进**，LLM 只是其中的推理内核。

**2. 和传统自动化脚本区别**：脚本是 **固定流程**，写死了 if-else，遇到没预设的情况就挂；Agent 能 **动态规划**，根据中间结果调整下一步。

**3. 和 ChatGPT 对话区别**：对话只是生成回复；Agent 能 **执行操作**（查数据库、调接口、写文件），有目标和反馈闭环。

**4. 为什么需要工具调用**：LLM 本身不能联网、不能算数、不能执行操作、知识有截止日期。工具让它 **突破自身能力边界**（查实时数据、执行代码、调 API）。

**5. 为什么需要 memory**：长任务/多轮交互需要记住之前发生了什么（上下文、历史决策、用户偏好），否则每步都"失忆"。

**6. 自主决策体现在哪**：**自己决定下一步调用哪个工具、传什么参数、是否结束**，而不是人预先编排好。

**7. 什么是环境感知**：Agent 通过 **工具返回结果** 来"观察"环境状态（接口返回、文件内容、报错信息），据此调整行动——这就是 ReAct 里的 Observation。

**8. Agent 是否真的具备自我意识**：**没有**。它是 **概率模型**，根据训练数据预测下一个 token，没有意识、没有真正的理解和意图。面试时千万别把它神化。

**9. 可能出现哪些错误**：幻觉（编造事实）、工具误用（传错参数）、死循环（反复调同一工具）、越权操作、目标偏离、错误累积放大。

**10. 如何限制行为边界**：**工具白名单**、**权限控制**（只读/沙箱）、**最大步数限制**、**关键操作人工审核**、输出校验、操作前确认。

## Q28：AI Agent 的核心模块

四大核心：**LLM + Planning + Tools + Memory**，外加 Reflection。

**1. LLM 负责什么**：**大脑**——理解任务、推理、做决策、决定调用什么工具、生成最终回答。

**2. Planning 模块负责什么**：**任务拆解和规划**——把复杂目标拆成可执行的子步骤（如 CoT、ReAct、Plan-and-Execute）。

**3. Tools 模块负责什么**：**执行能力**——把外部能力（查库、调接口、跑脚本、搜索）封装成 LLM 能调用的工具。

**4. Memory 模块负责什么**：**记忆**——保存上下文、历史对话、中间结果和长期知识。

**5. Function Calling 和 Tools 的关系**：Function Calling 是 LLM **选择调用哪个函数、并按 schema 生成入参** 的机制；Tools 是 **被调用的具体能力**。FC 是"决定怎么调"，Tools 是"被调的东西"。

**6. 短期记忆 vs 长期记忆**：

- **短期记忆**：当前会话的上下文，放在 LLM 的上下文窗口里，会话结束就没了。
- **长期记忆**：持久化的知识/历史，存在 **外部向量库/数据库**，需要时检索回来（RAG 就是长期记忆的一种实现）。

**7. Reflection/Feedback 模块作用**：**自我反思和纠错**——执行后回看结果是否合理、是否达成目标，发现错误就修正重试，提升可靠性。

**8. 如何根据工具返回结果继续决策**：**ReAct 循环**——Thought（想）→ Action（调工具）→ Observation（看结果）→ 再 Thought……直到完成或达到步数上限。

**9. 安全控制模块如何设计**：工具白名单、权限分级、敏感操作审批、输入输出校验、最大步数/超时、操作审计日志。

**10. Agent 执行失败后如何恢复**：重试（带退避）、回退到上一步、换工具/换策略、降级、必要时 **转人工介入**，并记录失败上下文便于排查。

## Q29：RAG 的完整流程

RAG（检索增强生成）= **先检索相关知识，再让 LLM 基于检索内容生成答案**，用外部知识弥补 LLM 知识不足和过期的问题。

```
离线：文档 → 清洗 → 分块(chunk) → embedding → 存入向量库
在线：用户问题 → embedding → 向量检索 Top-K → rerank → 拼 prompt → LLM 生成 → 带引用返回
```

**1. 文档为什么要分块**：文档太长塞不进上下文，且检索粒度太粗不精准；分块让检索能定位到 **相关片段**。

**2. chunk 太大的问题**：包含太多无关内容（噪声），稀释相关性，还可能超 token。

**3. chunk 太小的问题**：**语义被切断**，一句话被拆开，检索回来上下文不完整，答非所问。要用合理大小 + **重叠（overlap）** 缓解。

**4. embedding 是什么**：把文本转成 **高维向量**，语义相近的文本向量距离也近，是语义检索的基础。

**5. 向量数据库里存什么**：**向量 + 原文 chunk + 元数据**（来源、标题、更新时间），检索命中向量后取回原文。

**6. 余弦相似度解决什么**：衡量两个向量的 **语义相似程度**，用来从向量库里找和问题最相关的内容。

**7. Top-K 召回是什么意思**：取相似度最高的 **K 个片段** 作为候选上下文。

**8. rerank 为什么必要**：向量召回是"粗排"，可能把相关性一般的也召回来；**rerank（精排）** 用更强的模型对候选重新打分排序，把最相关的排前面，显著提升质量。

**9. prompt 如何组装**：把检索到的片段作为"参考资料" + 用户问题 + 指令（"仅根据以下资料回答，不要编造")拼成 prompt 给 LLM。

**10. RAG 如何降低幻觉**：让 LLM **基于检索到的真实资料** 回答，并 **强制引用来源**，约束它别凭空生成。

**11. RAG 是否能完全避免幻觉**：**不能**。检索内容不全/不准、或 LLM 没严格遵循资料时仍会幻觉。RAG 是 **大幅降低**，不是根除。

**12. 检索不到答案怎么办**：让模型 **明确回答"知识库中没有相关信息，无法回答"**，而不是硬编一个看似合理的答案——这是工程上必须显式处理的兜底。

**13. 如何保证知识库内容时效性**：**定期更新 + 版本管理 + 增量同步**，旧文档下线或标注失效，元数据带更新时间，避免基于过期文档作答。

## 深挖：RAG 工程上真正决定效果的点（textbook 之外）

**1. 纯向量检索会"语义相似但答非所问"——要混合检索**：余弦相似度找的是语义相近，但对 **精确关键词**（错误码 `ORA-00942`、接口名 `createOrderV2`、版本号）很弱——这些 token 在向量空间里被"平均"掉了。生产 RAG 几乎都用 **混合检索：BM25(关键词) + 向量(语义)**，再用 RRF（Reciprocal Rank Fusion）融合排名。测试知识库里全是接口名/错误码/字段名，纯向量会漏得很惨。

**2. chunk 不是"切多大"而是"按什么结构切"**：固定 512 token 硬切会把一个接口的参数表拦腰截断。正确做法是 **按文档结构切**（Markdown 按标题层级、接口文档按单个接口、代码按函数），再配 **overlap（10~20%）** 防边界信息丢失。进阶用 **small-to-big / parent-child**：用小 chunk 检索（精准），返回时给 LLM 它所在的大 chunk（上下文完整）。这比纠结 chunk_size 的数字有用得多。

**3. rerank 的代价要算清**：cross-encoder rerank 精度高但 **慢且贵**（每个候选都要过一次模型），所以是"向量粗召回 ~50 条 → rerank 精排到 5 条"的 **两段式**，而不是对全库 rerank。要在延迟和精度间权衡。

**4. 最危险的失败模式是"检索到了，但检索错了"**：检索不到还能兜底说"不知道"；可怕的是召回了 **貌似相关实则错误/过期** 的片段，LLM 言之凿凿照着答、引用还煞有介事——这种幻觉最难发现。对抗：元数据按版本/时间过滤优先取最新、答案强制标注来源、对低相似度命中直接拒答。

**5. 多跳问题（multi-hop）单轮检索答不了**："A 接口依赖的 B 服务超时配置是多少"要先查 A 再查 B，一次检索拿不全。靠 query 改写/分解或 agentic RAG 多轮检索。

**6. 怎么评估 RAG（不能只"看着还行"）**：分两层量化——
- **检索层**：context recall（该召回的相关片段召回了没）、context precision（召回的有多少真相关）。
- **生成层**：faithfulness（答案是否忠于检索内容、有没有脱离资料发挥）、answer relevance（答没答到点上）。
- 落地用 **RAGAS** 这类框架 + 标注评估集 + LLM-as-judge，每次改 chunk/rerank/prompt 都回归这套指标，横向可比——这才是"持续优化"的抓手，不是拍脑袋调。

## 深挖：Agent 落地真正的难点是"可靠性"，不是"跑通 demo"

**1. 错误会累积放大**：多步任务里第 2 步的小偏差会被第 5 步放大成完全跑偏。单步 90% 正确率，10 步连乘只剩 ~35%。对抗：每步做 **结果校验/反思（reflection）**、关键节点设 checkpoint。

**2. 上下文窗口会被吃光**：长任务里历史对话 + 工具返回不断堆积，迟早超窗口。要做 **记忆压缩/摘要、只保留相关历史、长期记忆外置向量库**。

**3. 工具调用死循环**：Agent 反复调同一个失败的工具。要 **限制最大步数、检测重复调用、失败 N 次熔断转人工**。

**4. 怎么测一个 Agent**：不能只看最终答案对不对（end-to-end），还要看 **过程**——轨迹评估（每步该不该调这个工具、参数对不对）、工具调用准确率、任务完成率、同输入多次运行的 **稳定性**（temperature 带来的不确定性）。这正是"AI 测试"要解决的问题，和评估 AI 生成用例的质量一脉相承（见 [10-AI与测试开发结合.md](10-AI与测试开发结合.md)）。
