## 一、项目经历与业务测试类（答题框架，非虚构事实）

> 此节多为"介绍你的项目"类问题，按要求剔除具体项目复述，仅给出**通用答题思路**，真实面试时请用自己的经历填充。

**11 / 一人同时负责大改动+压测+线上监控+稳定性，如何分配时间？**
按"风险优先 + 阶段并行"分配：
1. 先做需求/技术评审，识别核心链路与高风险点（占比约 10%）。
2. 功能测试优先保证主流程与核心用例（40%），用自动化回归覆盖存量功能减少人力。
3. 压测在功能基本稳定后介入，复用线上流量模型，跑非高峰时段（20%）。
4. 线上监控/稳定性做成长效机制（告警、巡检、降级预案）而非临时投入（20%）。
5. 留缓冲应对突发（10%）。核心原则：能自动化的不手工、能左移的不右移。

**12 / 迭代节奏**：说明自己团队真实节奏即可（如双周迭代/周迭代+热修），强调测试介入时机（需求评审即介入，左移）。

**13 / 是否涉及打包发布**：如实回答；可补充 CI/CD 中测试卡点（流水线门禁、自动化回归通过才允许发布）。

**14 / 未来规划**（框架）：往"测试架构/质量平台/质量度量体系"方向深入，把重复测试工程化，用 AI/工具提升效率。

**15 / 离职动机**（框架）：聚焦"寻求更大技术挑战/平台/成长空间"，避免抱怨前东家与同事。

**16 / 期望薪资**（框架）：先问对方薪资结构与职级范围，给一个有依据的区间（参考市场分位 + 当前涨幅预期），表达可谈。

**17 / 反问环节**（参考话术）：团队当前最大的质量痛点是什么？测开在团队的定位（造工具 vs 测业务占比）？技术栈与自动化覆盖现状？团队成长/晋升机制？

## 二、测试基础与测试流程类

**1&2 / 常用测试用例设计方法**
- 等价类划分、边界值分析、判定表（因果图）、正交实验法、场景法、错误推测法、状态迁移法。
- 实战常用组合：**边界值 + 等价类**覆盖输入校验，**场景法**覆盖业务流程，**判定表**处理多条件组合，**正交法**做大规模参数组合降维。

**3&4 / 一个好的测试用例的特点 / 包含内容**
- 特点：覆盖明确、可执行、可复现、独立、有明确预期结果、优先级清晰、便于维护。
- 内容：用例编号、标题、前置条件、测试步骤、测试数据、预期结果、实际结果、优先级、所属模块。

**5 / 给一个接口怎么写用例**
1. 正常场景：必填参数、典型组合，校验状态码/响应结构/业务字段/数据落库。
2. 参数校验：必填缺失、类型错误、边界值、超长、特殊字符、空值/null。
3. 业务逻辑：状态流转、幂等、并发、权限/鉴权（无 token/过期 token/越权）。
4. 异常与容错：依赖三方超时/异常、限流、降级。
5. 安全：SQL 注入、越权、敏感信息泄露。
6. 性能：高频接口的响应时间与并发表现。

**6 / 软件测试流程**
需求评审 → 测试计划 → 用例设计与评审 → 测试环境准备 → 执行测试（冒烟→功能→集成→回归）→ 缺陷管理与跟踪 → 测试报告 → 上线验证 → 复盘。强调测试左移（需求阶段介入）与右移（线上监控）。

**7 / 黑盒测试怎么测出最多 bug**
组合多种用例设计方法（边界值最易出 bug），重点覆盖边界/异常/并发/状态流转；结合探索式测试；按风险优先级投入；关注历史高频缺陷区域与新老功能交互。

**8 / 测试标准怎么定**
基于需求 + 验收标准 + 质量门禁（用例通过率、核心用例 100%、缺陷收敛、无遗留致命/严重缺陷、性能达标）。标准应可量化、可验证。

**9 / 核心用例的定义**
覆盖核心业务主流程、影响面大、一旦失败损失最大的用例（如下单支付主链路）。冒烟/回归优先执行，必须 100% 通过。

**10&11 / 产品上线后/线上出现 bug 怎么处理**
1. 评估影响范围与严重程度，先止血（回滚/降级/限流/开关关闭）。
2. 定位根因（日志、监控、复现）。
3. 修复并验证（优先热修或灰度）。
4. 回归相关功能，确认无连带影响。
5. 复盘：根因、流程漏洞、补充用例与监控，避免再次发生。

**12 / 上线前发现影响小但改动代码多的 bug**
评估风险与收益：改动大意味着回归成本与引入新 bug 风险高。若影响小且非阻塞，倾向**本次不改、记录跟踪、下个版本修复**；与产品/研发共同决策并留痕。优先保证上线稳定性。

**13&14 / 怎么判断前端 bug 还是后端 bug / 前后端定位**
抓包看请求与响应：
- 请求参数错误（前端没传对）→ 前端问题；
- 请求正确但响应数据/状态码错误 → 后端问题；
- 响应正确但页面展示错误 → 前端渲染问题。
结合后端日志、接口文档比对，必要时直接调接口（Postman/curl）排除前端。

**15 / App 端和 Web 端测试区别**
- App 需关注：多机型/分辨率/系统版本兼容、安装/升级/卸载、弱网/断网、前后台切换、推送、权限、耗电/内存/流量、Crash/ANR。
- Web 关注：多浏览器兼容、响应式布局、Cookie/Session、前端性能。
- App 测试更依赖真机/云测平台，自动化用 Appium；Web 用 Selenium。

## 三、接口测试与接口自动化类

**1&2 / 多接口自动化以什么形式做**
以**业务场景串联**为主：通过提取上一个接口响应中的字段（如 token、id）作为下一个接口的入参，形成完整业务链路（如登录→下单→支付→查询），而非孤立单接口。用例数据与代码分离，参数化驱动。

**3&5&6 / Python 如何搭建接口自动化框架 / 用什么框架**
典型组合：**pytest + requests + allure + yaml/excel + Jenkins**。
- requests 封装统一的请求方法（含 header、超时、重试、日志）。
- pytest 做用例管理、fixture 管理前后置（如登录获取 token、数据准备/清理）、参数化（`@pytest.mark.parametrize`）。
- 数据分层：测试数据用 yaml/excel/数据库，与代码解耦。
- 断言封装：状态码、JSON schema、业务字段、数据库校验。
- allure 生成报告，Jenkins 定时/触发执行，结果通知（钉钉/企微）。
- 工程结构：`config / common(封装) / testcases / data / report` 分层。

**4 / 自动化搭建原理**
本质是用代码模拟客户端发起请求 → 获取响应 → 断言 → 报告。核心是**封装复用 + 数据驱动 + 用例与数据分离 + 持续集成**。

**7 / 自动化框架做过哪些迭代（框架思路）**
常见迭代方向：单接口→场景串联；硬编码→数据驱动；本地执行→CI 定时执行；无报告→allure+通知；增加失败重试、token 自动刷新、测试数据自动清理、并发执行、环境多套切换。

**8&9&10 / 写自动化遇到的问题与解决**
- **用例不稳定（flaky）**：依赖外部环境/数据 → 用 mock 隔离三方、用例自带数据准备与清理、加合理重试。
- **数据耦合**：用例间共享数据导致互相影响 → 每条用例独立造数、用完即清。
- **token/鉴权**：过期导致批量失败 → fixture 统一管理、自动刷新。
- **断言不充分**：只断状态码漏业务错误 → 增加业务字段与数据库校验。
- **维护成本高**：接口变更全改 → 请求与数据分层、统一封装。

**11&12&13 / 自动化稳定性如何提升，首先改善什么**
首先解决**环境与数据稳定性**（最大不稳定来源）：
1. 测试数据隔离与自愈（每条用例独立造数清数）。
2. 三方依赖 mock，避免外部抖动。
3. 合理重试 + 超时控制，区分"真失败"与"偶发抖动"。
4. 断言精准化，减少误报。
5. 用例幂等、无执行顺序依赖。
6. 监控失败率与失败归因，持续治理 flaky 用例。

**14&15 / 自动化 case 执行时 Agent 层面异常如何处理 / 在哪层做**
异常处理统一在**框架的请求封装层（公共层）**做：捕获连接异常/超时/解析异常，记录详细日志（请求+响应），按策略重试或标记跳过，避免单条异常导致整体中断。业务层只关注断言。

**16&17 / 降级测试多吗 / 除降级外的稳定性手段**
除降级外还有：限流、熔断、超时控制、重试、兜底数据、开关（feature flag）、故障演练（混沌工程，如随机杀进程/注入延迟）、全链路压测、线上巡检与监控告警。

**18 / 怎么模拟三方接口**
用 mock server（如 `unittest.mock`/`responses`/`mockserver`/`WireMock`/自建 Flask mock 服务），按契约返回约定数据；可配置返回正常、异常、超时、错误码等不同响应。

**19 / 三方接口异常场景怎么设计**
覆盖：超时、连接失败、返回错误码（4xx/5xx）、返回空/格式错误、返回慢（接近超时边界）、限流被拒、数据不一致。验证本系统的容错、重试、降级、兜底是否正确。

## 四、Web / UI / App 自动化类

**1&3&5 / Web/UI 自动化怎么写、有没有做过**
基于 Selenium（或 Playwright），采用 **PO（Page Object）模式**：页面元素与操作封装到页面类，用例只调用页面方法，元素变更只改一处。用 pytest 组织用例 + 显式等待 + 截图 + allure 报告 + Jenkins 执行。

**2 / Web 自动化元素定位**
优先级：id > name > class > css selector > xpath。推荐 **CSS selector**（快、简洁）和稳定的 **xpath**（支持文本/层级定位）。尽量用稳定属性（如自定义 `data-testid`），避免依赖易变的绝对路径与动态 class。配合显式等待（`WebDriverWait + expected_conditions`）解决元素未加载问题。

**6 / UI 自动化原理**
- **Selenium / Appium**：基于 **W3C WebDriver 协议**驱动浏览器/设备。测试脚本 → WebDriver（如 ChromeDriver / uiautomator2 / XCUITest）→ 浏览器/App，模拟用户操作（点击/输入/滑动）并读取页面状态做断言。
- **Playwright**：不走 WebDriver 协议，而是通过浏览器自带的 **DevTools/调试协议（CDP 等）**直接与浏览器内核通信，自带自动等待、单进程多上下文，通常更快更稳。
- 共同本质：用程序模拟用户操作页面/控件并断言其状态。

**7 / UI 自动化优缺点**
- 优点：贴近真实用户、端到端覆盖、能发现集成与展示层问题。
- 缺点：执行慢、稳定性差（受页面/网络影响易 flaky）、维护成本高。
- 结论：UI 自动化只覆盖核心主流程，大量逻辑下沉到接口/单元层（测试金字塔）。

**8&9 / App 自动化怎么做**
用 **Appium**（兼容 Android/iOS，基于 WebDriver 协议）：连接真机/模拟器，通过 uiautomator2(安卓)/XCUITest(iOS) 驱动；同样用 PO 模式封装；处理 App 特有问题：等待元素、控件定位（uiautomatorviewer/Appium Inspector）、原生与 WebView 切换、弹窗/权限处理、多机型适配。结合云真机平台做兼容性。

## 五、性能测试 / 压测 / 稳定性类

**1 / 压测类型**
负载测试、压力测试（找极限/拐点）、稳定性测试（长时间高负载看是否泄漏/退化）、并发测试、容量测试、峰值/尖峰测试（突发流量）。

**2&3&4 / 压测/性能测试怎么做、场景**
1. 明确目标与指标（TPS/QPS、RT、错误率、资源使用率）。
2. 基于线上流量分析建立**压测模型**（接口比例、参数分布）。
3. 准备压测环境与数据（数据量级贴近生产）。
4. 用工具（JMeter/Locust）逐步加压，找性能拐点。
5. 监控全链路（应用、DB、缓存、中间件、机器资源）。
6. 定位瓶颈（慢 SQL、锁、线程池、GC、连接数、缓存命中），优化后回归压测。
7. 输出报告与容量评估。

**5 / 性能测试相关**：见上，核心是"建模型→加压→看指标→定位瓶颈→优化回归"。

**6 / 客户端性能测试**
关注：启动时间、页面渲染/帧率（FPS、卡顿/丢帧）、CPU/内存（含内存泄漏）、耗电、流量、Crash/ANR。工具：Android 用 Perfetto/GPU 呈现/adb，iOS 用 Instruments，通用可用厂商性能平台。

**7 / 服务端压测用什么平台/工具**
开源常用 **JMeter、Locust（Python，易扩展）、wrk/Gatling**；大厂多用自研全链路压测平台（流量录制回放 + 影子库 + 链路打标）。

**8&9&10 / 关注哪些压测指标 / 如何评估压测通过 / 通过标准**
核心指标：**TPS/QPS、响应时间（平均、P95、P99）、错误率、并发数、资源使用率（CPU/内存/IO/网络）**。
通过标准（举例）：在目标并发下，P99 RT ≤ 约定阈值、错误率 < 0.1%、CPU/内存在安全水位（如 CPU<70%）、无内存泄漏、系统能稳定持续运行且可线性扩展。

**11&12 / 压测模型 / 并发多大**
压测模型来自**线上真实流量画像**：各接口调用比例、参数分布、读写比例、峰值 QPS，再按容量目标（如支撑 N 倍峰值）设定并发。并发量需结合具体业务真实数据，不应凭空给数。

**13 / 事务数（TPS）**
TPS = 每秒处理事务数，一个"事务"可由一个或多个请求组成，是衡量系统吞吐的核心指标。需明确事务的业务边界（如"一次完整下单"算一个事务）。

**14&15&16 / 整系统压测 / 全链路压测 / 全链路核心目标**
全链路压测 = 在**生产或仿真生产环境**对完整业务链路施压，目标：
- 验证大促/峰值下整体系统容量与稳定性；
- 发现单接口压测发现不了的**链路瓶颈与木桶短板**（某个下游/中间件先挂）；
- 验证限流、降级、扩容预案有效性；
- 容量规划与扩容依据。

**17&18 / 压测环境**
- 独立性能测试环境（成本低，但与生产有差异，结果仅供参考）；
- 仿真/预发环境；
- **生产环境全链路压测**（最真实，需做流量隔离、影子库、数据打标防污染）。

**19&20 / 压测是否做巡检 / 巡检机制**
做。定期（如每日/每周或大促前）自动跑基线压测，对比历史指标，发现性能劣化（RT 上升、TPS 下降）即告警，防止迭代引入性能回退。机制：定时触发→执行基线场景→指标对比阈值→异常告警与归因。

**21&22 / 生产压测数据污染怎么解决 / 数据怎么透传**
- **流量打标**：压测请求带特殊标识（如 header `pressure-test=true`），全链路透传该标识；
- **影子库/影子表**：带标识的写操作落到影子存储，不污染真实数据；
- 中间件（MQ、缓存）按标识隔离；
- 标识通过上下文（如 ThreadLocal / 请求头透传）在全链路传递，保证下游都能识别压测流量。

**23 / 线上性能情况**：如实结合监控数据回答（核心接口 P99、峰值 QPS、资源水位）。

**24（同三 17）/ 除降级外稳定性手段**：限流、熔断、超时、重试、兜底、开关、故障演练、巡检监控。

## 六、AI / Agent / Skills 类

**1&2 / AI 应用场景 / 用过哪些 AI 工具**
测开中的 AI 应用：用例自动生成/补全、根据接口文档生成接口自动化代码、日志/报错智能归因、测试数据生成、缺陷描述润色与分类、UI 自动化元素自愈、需求/PRD 转测试点。工具：Claude Code、Cursor、Copilot 等编码助手，以及调用大模型 API 做自定义工具。

**3&4 / 开发过 Agent 吗 / 用 AI 写过什么脚本**
Agent 即"大模型 + 工具调用 + 记忆/循环"的自动化体：可做"自动读接口文档→生成用例→执行→分析失败→给修复建议"的测试 Agent。AI 辅助写脚本：日志分析、数据造数、批量接口校验、报告汇总等。（如实结合自身实践回答）

**5 / AI 怎么做长时间执行任务**
靠**任务拆解 + 循环 + 记忆 + 工具调用**：将大任务拆成可执行子步骤，逐步执行并把中间结果存入记忆/外部存储（避免上下文溢出），通过反思循环（执行→观察→修正）推进，必要时人工介入确认关键步骤；用持久化状态支持断点续跑。

**6 / 了解 Skills 吗**

**10&11 / 全局锁（GIL）及作用**

**12 / 线程是不是越多越好**

**13 / 请求搜狐网下载页面所有 jpg**
> 面试建议：直接说用 **BeautifulSoup/lxml 解析 `<img>` 标签 + `urljoin` 补全 URL** 更稳——纯正则只能抓双引号里的绝对 `.jpg`，会漏掉相对路径、`//` 协议相对路径、带 `?query` 的图片、`data-src` 懒加载字段等。下面给推荐写法，正则版仅作"会用正则"的兜底示意。

```python
import os
import requests
from urllib.parse import urljoin
from bs4 import BeautifulSoup

url = "https://www.sohu.com"
resp = requests.get(url, timeout=10)
resp.encoding = resp.apparent_encoding
soup = BeautifulSoup(resp.text, "html.parser")

# 兼容懒加载：同时取 src / data-src，再用 urljoin 补全相对路径与 // 协议路径
imgs = set()
for tag in soup.find_all("img"):
    src = tag.get("src") or tag.get("data-src")
    if src and ".jpg" in src.lower():
        imgs.add(urljoin(url, src))

os.makedirs("imgs", exist_ok=True)
for i, img_url in enumerate(imgs):
    try:
        data = requests.get(img_url, timeout=10).content
        with open(f"imgs/{i}.jpg", "wb") as f:
            f.write(data)
    except Exception as e:
        print(f"下载失败 {img_url}: {e}")
```

## 十、计算机网络类

**1 / HTTP vs HTTPS**
HTTPS = HTTP + TLS/SSL 加密。区别：
- HTTP 明文传输不安全，HTTPS 加密 + 身份认证 + 完整性校验；
- 默认端口 HTTP 80 / HTTPS 443；
- HTTPS 需 CA 证书；
- HTTPS 有握手开销，性能略低但安全。

**2 / HTTP 常用请求方法**
GET（查）、POST（增/提交）、PUT（全量更新）、PATCH（部分更新）、DELETE（删）、HEAD（只取头）、OPTIONS（预检/查询支持的方法）。

**3&4&5 / 长连接 vs 短连接 / 为什么长连接不断开**
- 短连接：每次请求建立并关闭 TCP 连接。
- 长连接：一次建连后复用同一条 TCP 连接处理多次请求（HTTP/1.1 默认 `Connection: keep-alive`）。
- 注意区分三个不同概念，面试常被追问：
  - **HTTP keep-alive**：应用层的"连接复用"，让一条 TCP 连接可承载多个 HTTP 请求/响应，省掉反复三次握手/四次挥手与 TLS 握手开销。它并不能保证连接永不断开——客户端或服务端的**空闲超时**（如 Nginx `keepalive_timeout`）到了仍会主动关闭。
  - **TCP keepalive**：传输层的探活机制（`SO_KEEPALIVE`），默认空闲约 2 小时才发探测包，用于检测对端是否已死、清理僵死连接，并非用来"保活不断开"。
  - **业务心跳**：长连接（如 WebSocket、IM、RPC 长连）由应用自己定时发心跳包，目的是探测连接可用性、防止中间设备（NAT/LB）因空闲回收连接。HTTP/1.1 的长连接本身并不依赖业务心跳。
- 一句话：HTTP/1.1 长连接的核心收益是**连接复用**而非"靠心跳永不断开"，空闲超时仍会回收；代价是占用服务端连接资源，需设合理空闲超时。

**6&7 / TCP vs UDP / 适用场景**
| | TCP | UDP |
|---|---|---|
| 连接 | 面向连接(三次握手) | 无连接 |
| 可靠性 | 可靠(确认/重传/排序) | 不可靠 |
| 速度 | 较慢 | 快、开销小 |
| 顺序 | 保证 | 不保证 |
| 场景 | Web、文件传输、支付等要求可靠 | 视频直播、语音、游戏、DNS 等要求低延迟 |

## 十三、算法题 / 编程题类

**1 / 找出第二大的数**
> 注意先和面试官确认题意：是要**第二大的不同值**（[5,5,4]→4），还是**排序后第二个位置的值**（[5,5,4]→5）。两种解法不同。

求第二大的不同值（去重语义）：
```python
def second_largest(nums):
    first = second = float("-inf")
    for n in nums:
        if n > first:
            first, second = n, first
        elif first > n > second:   # 严格小于 first，跳过与最大值相等的重复值
            second = n
    return second if second != float("-inf") else None
```
若按"排序后第二位"算（不去重），把 `elif first > n > second` 改为 `elif n > second`，则 [5,5,4] 返回 5。一次遍历，O(n)。

**2 / 快排原理与复杂度**
分治：选基准 pivot，将比它小的放左、大的放右（partition），再递归处理左右子数组。
- 平均时间 O(n log n)，最坏 O(n²)（已有序且每次选端点为基准），不稳定排序。
- 空间复杂度看实现：**原地 partition 版**额外空间 O(log n)（仅递归栈）；下面这版用列表推导生成 left/mid/right，每层都额外创建新列表，整体额外空间约 O(n)，胜在简洁易记、适合面试白板。
```python
def quick_sort(a):                 # 简洁版（非原地，额外空间 O(n)）
    if len(a) <= 1:
        return a
    pivot = a[len(a)//2]
    left = [x for x in a if x < pivot]
    mid = [x for x in a if x == pivot]
    right = [x for x in a if x > pivot]
    return quick_sort(left) + mid + quick_sort(right)
```

**3 / 冒泡排序原理与复杂度**
相邻两两比较，大的往后冒，每轮确定一个最大值。时间 O(n²)（最好 O(n)，加 flag 提前结束），空间 O(1)，稳定排序。
```python
def bubble_sort(a):
    n = len(a)
    for i in range(n - 1):
        swapped = False
        for j in range(n - 1 - i):
            if a[j] > a[j + 1]:
                a[j], a[j + 1] = a[j + 1], a[j]
                swapped = True
        if not swapped:
            break
    return a
```

**4 / 纯数字字符串的最长连续递增子串**
> 说明：原题示例 `输入 2356789 输出 567` 存在矛盾——若"连续"指相邻数字依次 +1，则 `2356789` 中最长的连续递增子串应为 `56789`（长度 5），而非 `567`。下面按"相邻字符依次 +1 的最长子串"给出正确解法。
```python
def longest_consecutive(s):
    if not s:               # 处理空字符串，避免 s[0] 越界
        return ""
    best = cur = s[0]
    for i in range(1, len(s)):
        if int(s[i]) == int(s[i - 1]) + 1:
            cur += s[i]
        else:
            cur = s[i]
        if len(cur) > len(best):
            best = cur
    return best

print(longest_consecutive("2356789"))  # 56789
```

**5 / 两数之和（判断是否存在）**
```python
def two_sum_exists(nums, target):
    seen = set()
    for n in nums:
        if target - n in seen:
            return True
        seen.add(n)
    return False
```
哈希一次遍历，O(n)。若要返回下标，把 set 换成 dict 存值→下标。

**6 / 用 A 替换字符串 B**
Python：`s.replace("B", "A")`；命令行：`sed -i 's/B/A/g' file`。（同八 9）

**7 / 红黑树**
自平衡二叉查找树，通过节点红/黑着色 + 5 条性质，保证从根到叶最长路径不超过最短路径的 2 倍，近似平衡。
- 优点：插入/删除/查找均为 O(log n)，且**调整代价比 AVL 小**（AVL 更严格平衡、旋转更多）。
- 缺点：实现复杂；查找略逊于严格平衡的 AVL。
- 应用：Java TreeMap/TreeSet、C++ STL map/set、Linux 内核调度等。

**8 / 哈希表作用与优缺点**
通过哈希函数把 key 映射到桶，实现平均 O(1) 的查找/插入/删除。
- 优点：快、查找高效。
- 缺点：有哈希冲突（用链地址法/开放寻址解决）；无序；最坏情况退化到 O(n)；扩容/rehash 有开销；空间换时间。

## 十四、逻辑题 / 智力题类

**1 / 握手问题**
常见版本：n 个人两两握手共多少次 = C(n,2) = n(n-1)/2（每两人一次，不重复）。若题为"每人与其他人各握一次手，总次数"同上。

**2 / 4 个罐子，一个药丸重 1g，电子秤最少几次称出**
> 前提（题目须给定）：每罐有足够多药丸，正常药丸每颗重 m 克，**被污染那一罐的每颗药丸重 m+1 克**（即每颗多 1g）。

**1 次**（用可读出具体重量的电子秤称重；若题面明确是两臂天平，要先和面试官澄清条件）。从第 1、2、3、4 罐分别取 1、2、3、4 颗放在一起称，记"实际总重 − 全部正常时理论总重 (1+2+3+4)·m"之差为 d 克。因为只有被污染罐的药丸每颗多 1g，多出的克数恰等于从该罐取的颗数，所以 **d = 被污染罐的编号**（d=1→1 号，d=2→2 号，d=3→3 号，d=4→4 号）。一次称重即可定位。

**3 / 7 分钟和 11 分钟沙漏量出 15 分钟**
同时倒转两个沙漏并开始计时：
- 第 7 分钟：7 分钟漏完，立即翻转 7 分钟沙漏；
- 第 11 分钟：11 分钟漏完，此时 7 分钟沙漏已走 4 分钟（还剩 3 分钟）；
- 立即翻转 7 分钟沙漏，让那 4 分钟的沙再漏回来，需 4 分钟；
- 11 + 4 = **15 分钟**。

**4 / 三开关三灯，一次查看灯，确定对应关系**
利用灯泡发热：
1. 打开开关 1，等几分钟，再关掉；
