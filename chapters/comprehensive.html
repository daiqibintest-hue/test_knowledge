Q001

什么情况下你会阻塞上线？

来源：测开面试_通用版_含答案.md
参考答案

四类硬性阻塞：

  • ①P0/P1 缺陷未关闭（如金额错误、数据泄露、AI 输出违法内容）
  • ②冒烟用例不通过
  • ③性能不达标（响应时间、错误率、TPS 突破基线）
  • ④高风险变更未灰度。软性阻塞需评审：自动化通过率<90%、未达到代码覆盖率门禁等。作为负责人我有一票否决权，但会用数据和风险评估而不是个人判断。
Q002

严重程度和优先级有什么区别？

来源：测开面试_通用版_含答案.md
参考答案
  • 严重程度（Severity）：技术维度，描述 Bug 对系统的破坏力，分 Blocker/Critical/Major/Minor/Trivial；
  • 优先级（Priority）：业务维度，描述修复的紧迫性，分 P0/P1/P2/P3。

两者可能不一致：比如登录按钮文案错别字是 Trivial 但若是产品发布会用，优先级会被定为 P0；后台日志拼写错误是 Major 但优先级 P3。

Q004

GET 和 POST 的区别是什么？

来源：测开面试_通用版_含答案.md
参考答案
  • 语义：GET 幂等只读，POST 非幂等可能改服务端状态；
  • 参数位置：GET 放 URL，POST 放 Body；
  • 长度限制：GET 受 URL 长度限制（不是 HTTP 协议规定，而是浏览器/服务器实现限制，常见几 KB），POST 放 Body 无此限制；
  • 缓存：GET 可缓存，POST 默认不缓存；
  • 安全：GET 参数会出现在日志和浏览器历史，敏感数据用 POST。

但 RESTful 设计中关键是语义而非技术差异。

Q005

PUT 和 PATCH 有什么区别？

来源：测开面试_通用版_含答案.md
参考答案
  • PUT：全量替换资源，请求体包含资源完整字段，幂等；
  • PATCH：增量更新，只传变更字段，非严格幂等（如增加计数器）。

比如更新工单状态，如果只改 status 字段用 PATCH；如果整个工单对象替换用 PUT。

Q006

Cookie、Session、Token、JWT 有什么区别？

来源：测开面试_通用版_含答案.md
参考答案
  • Cookie：浏览器存储，由服务端 Set-Cookie 下发，自动随请求携带；
  • Session：服务端存储会话状态，依赖 Cookie 传递 sessionId；
  • Token：泛指身份令牌，无状态，客户端主动放入 Header；
  • JWT：Token 的一种实现，由 Header.Payload.Signature 三段组成，自包含用户信息，可被服务端无状态校验。

我们项目用 JWT + Redis 黑名单，既享受无状态又能主动注销。

Q008

场景 7：测试时间被压缩

来源：测开面试_通用版_含答案.md
场景问题

原计划 5 天测试，研发延期后只剩 1 天，明天必须上线。怎么做？

参考答案

不慌，按风险优先级压缩。七步：

  1. 风险评估：
    • 列出本期变更范围（新功能/修改/Bug 修复）；
    • 评估影响（用户量/核心程度/线上风险）；
    • 量化"如果不测，可能出什么问题"。
  2. 范围裁剪：
    • 必跑：核心路径冒烟 + 接口自动化全量（约 30 分钟）+ AI 评估核心集；
    • 选跑：本期变更涉及的回归用例（约 30 个）；
    • 跳过：与变更无关的回归（用接口自动化覆盖）。
  3. 自动化兜底：让 Jenkins 全力跑回归集，1 小时内出结果。
  4. 资源协调：拉团队全员 1 天集中突击核心用例。
  5. 风险升级：明确写邮件给项目经理：「在 1 天测试时间内，仅能覆盖核心路径，遗留风险 X、Y、Z」，让上层书面决策。
  6. 灰度兜底：要求上线后先 10% 灰度 24 小时，监控核心指标，无异常再全量。
  7. 回滚预案：准备回滚脚本和数据回滚方案，可 30 分钟内回滚。

上线后：补跑剩余用例 + 加强监控告警 + 客户反馈快速通道。

绝不"硬扛压力沉默上线"，决策权和风险都让管理层看到。

Q020

场景 5：通知分类准确率 95%，但业务仍然投诉

来源：测开面试_通用版_含答案.md
场景问题

通知自动分类准确率达到 95%，但业务仍然投诉很多。怎么看？

参考答案

准确率高但投诉多，说明指标失真。可能原因：

  1. 类别不均衡：占比 80% 的"通用通知"类全对，但占比 5% 的"高风险运营触达"类只对 50%。整体准确率虽然 95%，但小类灾难。
  2. 指标维度错：只看 Accuracy 不看 Recall。运营触达类漏发等于丢钱，必须看每个小类 Recall。
  3. 业务代价：5% 的错误集中在高价值场景（如错把"严肃运营触达"分到"温馨提醒"）。
  4. 测试集与真实分布不一致：测试集都是历史样本，实际线上 query 分布不同。

修复方案：

  • 改用 F1 + 各类别独立指标，不再只看 Accuracy；
  • 按业务代价加权：错分代价高的类别权重更大；
  • 重要类别（运营触达/合规）单独看 Recall 必须 ≥ 95%；
  • 测试集按线上真实分布抽样；
  • 误分类样本人工 Review 找模式，针对性加训练数据；
  • 关键类别加规则兜底（如金额 > 5 万必走人工审核）。
Q021

"线上问题复现率与闭环率 98%+"——指标怎么算？

来源：测开面试_通用版_含答案.md
参考答案
  • 复现率 = 测试环境成功复现的线上问题 / 线上问题总数。
  • 闭环率 = 已修复并验证通过的线上问题 / 线上问题总数。
  • 统计周期：季度。
  • 数据来源：Jira 工单状态导出。
  • 98%+ 意味着季度内 50 个线上问题中只有 1 个未闭环（一般是非紧急问题延期）。
Q022

"自动化效率提升 90%"——提升前后分别是多少？

来源：测开面试_通用版_含答案.md
参考答案
  • 优化前：每个版本人工回归 8 小时（1 个测试 1 天），自动化覆盖率 30%。
  • 优化后：自动化跑 30 分钟（pytest-xdist 并发 + 接口覆盖率 80%），人工只复核失败用例（约 1 小时）。
  • 提升计算：8 小时降到 1.5 小时 ≈ 81% 提升，加上夜间无人值守跑的间接效益约 90%。

⚠️ 别咬死 90%：这是含间接收益的约数。被追问就退到硬数据——"机器执行从 8 小时压到 30 分钟"（下一题那条最铁），90% 当方向性描述就行。

Q027

"带 3 人团队"——如何分配任务和控制进度？

来源：测开面试_通用版_含答案.md
参考答案

分配：

  • A（5 年经验）：核心模块 + AI 测试 + 自动化框架；
  • B（3 年）：接口自动化 + 性能测试；
  • C（2 年）：功能测试 + 自动化学习。

进度控制：

  • 每日站会 15 分钟（昨天/今天/阻塞）；
  • 每周看板回顾（任务完成率、缺陷数）；
  • 双周一对一深度沟通（成长/困难）；
  • 关键节点 buffer 20%；
  • 延期 1 天必须升级讨论。

结果口径：不要说"小团队替代大团队"这种容易被质疑的话。更稳的是：通过分工、用例分级、接口自动化和风险前置，缓解了人力压力；成员能力成长可以举具体技能变化，但不硬报离职率、替代人数这类难证明数字。

Q028

语句覆盖和分支覆盖的区别？举例。

来源：测开面试_通用版_含答案.md
参考答案

语句覆盖只要求"每行代码执行到"，分支覆盖要求"每个判断的 true/false 都走到"。例：

def f(a, b):
    if a > 0 and b > 0:
        x = 1
    return x

用例 (a=1, b=1) 能让所有语句执行（语句覆盖 100%），但 if 的 false 分支没走到，分支覆盖只有 50%，而且 (a=-1) 时 x 未定义的 Bug 被漏掉。所以分支覆盖比语句覆盖强——这也是为什么只看"行覆盖率"会有盲区。

Q029

list / tuple / set / dict

来源：14-补充题.md
参考答案

list 有序可变；tuple 有序不可变（可做 dict key）；set 无序、元素唯一、用于去重；dict 键值对、键唯一、查找 O(1)。

3. 装饰器：在不改原函数代码的前提下增强功能的语法糖。测试框架里大量用：@pytest.fixture、@pytest.mark.parametrize、@allure.step，以及自定义重试/计时装饰器。

4. 生成器 vs 迭代器：迭代器是实现了 __iter__/__next__ 的对象；生成器是用 yield 简化创建迭代器的方式，惰性求值、省内存，适合处理大数据集。

5. 多线程 vs 多进程：多线程共享内存、适合 IO 密集（接口请求、文件读写）；多进程各自独立内存、能利用多核、适合 CPU 密集。接口自动化并发用多线程即可。

6. GIL：全局解释器锁，同一时刻 CPython 只允许一个线程执行字节码，导致多线程不能真正并行计算。所以 CPU 密集用多进程，IO 密集多线程仍有效（IO 等待时会释放 GIL）。

7. 异常捕获：try/except/else/finally，except 按异常类型分别处理，finally 做资源释放（关连接、关文件）。要捕获具体异常而非裸 except。

8. *args vs kwargs：*args 收集位置参数成 tuple；kwargs 收集关键字参数成 dict。请求封装里 **kwargs 透传 params/json/headers 非常常用。

Q030

Python 怎么做单元测试和 Mock？

来源：测开面试_通用版_含答案.md
参考答案
  • ①框架：unittest（标准库）或 pytest（更简洁，见第五部分）
  • ②Mock：用 unittest.mock 把外部依赖（第三方接口、数据库、时间）替换成可控的假对象，隔离被测逻辑：
from unittest.mock import patch

@patch('module.requests.get')
def test_query(mock_get):
    mock_get.return_value.json.return_value = {'code': 0}
    assert query() == 0   # 不真正发请求

要点：patch 的是"被测模块里引用的路径"，不是定义处；Mock 让单测不依赖外部环境、可重复。

Q033

全局锁（GIL）及作用

来源：测开面经.md
参考答案

GIL（全局解释器锁）是 CPython 解释器的机制，保证同一时刻只有一个线程执行 Python 字节码，简化内存管理与 C 扩展线程安全。代价：CPU 密集任务无法靠多线程并行（多线程只在 IO 等待时切换）。CPU 密集用多进程绕过 GIL；IO 密集用多线程/协程仍有效。（注：3.13 起有可选的 no-GIL 实验特性。）

Q034

线程是不是越多越好

来源：测开面经.md
参考答案

不是。线程过多带来上下文切换开销、内存占用、锁竞争，反而降低性能；受 GIL 限制 CPU 密集型多线程无收益。应按任务类型（IO 密集/CPU 密集）和资源设定合理线程数。

Q035

长连接 vs 短连接 / 为什么长连接不断开

来源：测开面经.md
参考答案
  • 短连接：每次请求建立并关闭 TCP 连接。
  • 长连接：一次建连后复用同一条 TCP 连接处理多次请求（HTTP/1.1 默认 Connection: keep-alive）。
  • 注意区分三个不同概念，面试常被追问：
    • HTTP keep-alive：应用层的"连接复用"，让一条 TCP 连接可承载多个 HTTP 请求/响应，省掉反复三次握手/四次挥手与 TLS 握手开销。它并不能保证连接永不断开——客户端或服务端的空闲超时（如 Nginx keepalive_timeout）到了仍会主动关闭。
    • TCP keepalive：传输层的探活机制（SO_KEEPALIVE），默认空闲约 2 小时才发探测包，用于检测对端是否已死、清理僵死连接，并非用来"保活不断开"。
    • 业务心跳：长连接（如 WebSocket、IM、RPC 长连）由应用自己定时发心跳包，目的是探测连接可用性、防止中间设备（NAT/LB）因空闲回收连接。HTTP/1.1 的长连接本身并不依赖业务心跳。
  • 一句话：HTTP/1.1 长连接的核心收益是连接复用而非"靠心跳永不断开"，空闲超时仍会回收；代价是占用服务端连接资源，需设合理空闲超时。

Q042

Cookie / Session / Token

来源：14-补充题.md
参考答案

Cookie 存客户端；Session 存服务端、靠 Cookie 里的 sessionId 关联；Token 适合分布式和前后端分离——其中 JWT 是自包含的（用户信息编码在 token 里、服务端无需存储即可校验），而 opaque token（不透明令牌）本身无意义，仍依赖服务端存储/查询 来校验有效性。

5. HTTPS 原理：HTTP + TLS，通过 非对称加密协商对称密钥，再用对称密钥加密传输，配合证书校验服务端身份，保证机密性和完整性。

6. 接口幂等性：同一请求执行一次和多次结果一致。下单、支付要做幂等（防重复扣款），常用唯一请求号/token 去重。

7. 如何测接口重复提交：用同一幂等号/请求并发或连续提交多次，断言只生成一条记录、不重复扣减。

8. 如何测接口限流：短时间高频请求，断言超过阈值后返回 429/限流响应，且恢复后能正常访问。

9. 如何测签名和时间戳：篡改签名应被拒；过期时间戳应被拒；重放同一请求应被拒（防重放）。

Q044

inner / left / right join

来源：14-补充题.md
参考答案

inner 只返回两表匹配的行；left 返回左表全部 + 右表匹配（右无则 NULL）；right 反之。

2. group by vs having：group by 分组；having 对 分组后 的结果过滤（where 是分组前过滤，且不能用聚合函数）。

3. 什么是索引：加速查询的数据结构（多为 B+树），用空间和写入性能换查询性能。

4. 哪些情况索引失效：对索引列用函数/运算、隐式类型转换、like '%xx' 前导模糊、or 连接非索引列、不满足最左前缀（联合索引）。

5. 什么是事务：一组要么全成功要么全失败的操作单元。

6. ACID：原子性（不可分割）、一致性（前后数据合法）、隔离性（并发互不干扰）、持久性（提交后永久生效）。

Q045

脏读 / 不可重复读 / 幻读

来源：14-补充题.md
参考答案

脏读=读到别的事务未提交的数据；不可重复读=同一行两次读值不同（被别的事务改了）；幻读=同一查询两次返回行数不同（被别的事务增删了）。

8. 如何用 explain 分析 SQL：看 type（访问类型，全表 ALL 最差）、key（是否用上索引）、rows（扫描行数）、Extra（是否 Using filesort / temporary），据此优化。

9. 什么是数据库锁：并发控制机制，有行锁/表锁、共享锁（读）/排他锁（写）、乐观锁/悲观锁。

10. 如何定位慢 SQL：开慢查询日志找出慢语句 → explain 分析执行计划 → 看是否缺索引/全表扫描/大量排序 → 优化索引或改写 SQL。
