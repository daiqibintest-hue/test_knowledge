2. inner join、left join、right join 有什么区别？

测开面试_通用版_含答案.md
参考答案
  • INNER JOIN：两表都匹配才返回；
  • LEFT JOIN：左表全保留，右表无匹配则字段为 NULL；
  • RIGHT JOIN：右表全保留。

测试时常用 LEFT JOIN 找"主表有但关联表没有"的脏数据。

#268

3. group by 和 having 怎么用？

测开面试_通用版_含答案.md
参考答案

GROUP BY 按字段分组聚合，HAVING 对分组后的结果过滤（WHERE 不能用聚合函数）。例：找重复工单号 SELECT order_no, COUNT(*) FROM orders GROUP BY order_no HAVING COUNT(*) > 1。

#270

5. count(*)、count(1)、count(字段) 有什么区别？

测开面试_通用版_含答案.md
参考答案
  • count(\*)：统计所有行（含 NULL）；
  • count(1)：等价于 count(\*)，性能在 InnoDB 中几乎无差异；
  • count(字段)：统计该字段非 NULL 的行数。

测试覆盖率统计要选对，不然漏算。

#271

6. 索引是什么？

测开面试_通用版_含答案.md
参考答案

索引是加速查询的数据结构，MySQL InnoDB 默认 B+ 树。聚簇索引（主键）数据和索引在一起，二级索引存的是主键值需要回表。代价是写性能下降和占空间。我会用 EXPLAIN 看是否走索引。

8. 事务的 ACID 是什么？

测开面试_通用版_含答案.md
参考答案
  • A 原子性：要么全成功要么全回滚；
  • C 一致性：事务前后数据满足业务规则；
  • I 隔离性：并发事务互不干扰；
  • D 持久性：提交后即使宕机也不丢。
#274

9. 脏读、不可重复读、幻读是什么？

测开面试_通用版_含答案.md
参考答案
  • 脏读：读到其他事务未提交的数据；
  • 不可重复读：同一事务两次读同一行结果不同（其他事务 UPDATE 了）；
  • 幻读：同一事务两次范围查询结果数不同（其他事务 INSERT/DELETE 了）。

18. 服务器 CPU、内存、数据库指标如何？

测开面试_通用版_含答案.md
参考答案

怎么拿到的（被问最多的就是这句）：应用 CPU/内存看 Grafana-Prometheus 大盘，GC 看 GC 日志 + jstat -gcutil；DB 的 CPU/连接数/QPS 看 Grafana 或 SHOW GLOBAL STATUS，慢查询看慢日志；Redis 命中率看 INFO stats（keyspace_hits/misses）；连接池占用看 HikariCP 监控。

当时环境下的样例口径：应用 CPU 约 60-70%（4 核 8G）、堆 4G、Young GC 有波动但 Full GC 不能频繁出现；DB CPU 约 50%、连接数约 80/200、QPS 约 5000、慢查询要为 0 或能解释；Redis 命中率保持在较高水平。面试时要强调：这些数以压测报告和监控截图为准，不同接口、机器配置、数据量会不一样。

7&8&9 / 关系型 vs 非关系型数据库

参考答案
维度关系型(MySQL/PostgreSQL)非关系型(Redis/MongoDB/ES)
结构表+行+列，固定 schema文档/KV/列族/图，灵活

10 / 存储过程

测开面经.md
参考答案

预先编译并存储在数据库中的一组 SQL 语句集合，可带参数、含逻辑控制，通过 CALL 调用。优点：减少网络交互、便于复用与封装、可统一权限控制；缺点：可维护性/可移植性差，业务逻辑下沉到 DB 不利于扩展。 > 注意别把"性能好"说绝对：存储过程主要靠减少应用与 DB 间的网络往返在批量/多步操作上占优，单条简单 SQL 不一定比应用层更快；现代架构也更倾向把业务逻辑放在应用层而非 DB。

#311

Q15：接口返回成功但数据库没数据，如何定位

04-数据库与数据校验.md
参考答案

这是一道经典排查题，思路是 从外到内、逐段确认：

1. 先看响应还是日志：先确认接口 真的 返回成功——不只看 HTTP 200，要看 业务码 是否为成功（可能 200 但 code 非 0）。再结合应用日志。

2. 确认请求参数正确：是不是参数本身就没带对（比如 userId 传错），后端"成功"地处理了一个空操作。

3. 确认是否写到别的库/表：分库分表场景，可能写进了另一个分片，查错库了。

4. 是否异步写入：很多写操作走 MQ 异步落库，接口返回成功只代表消息发出去了，查太早自然没有 → 需要等待/轮询。

5. 是否写缓存没落库 / 消息丢失：先写 Redis 后异步落库，中间消息丢了。

6. 是否事务回滚：接口内后续步骤异常触发回滚，但响应已先返回成功（代码 bug）。

7. 是否主从延迟：写主库、查从库，主从同步有延迟 → 刚写完查从库查不到。

8. 轮询等待：针对异步，设计 带超时的重试查询（每隔 N 秒查一次，最多查 M 次），给异步留时间，避免误报。

9. 业务代码问题 vs 脚本问题：脚本查错库/查询条件写错 → 脚本问题；参数对、时机对、库对，确实没写 → 业务 bug。

10. 用 traceId 定位全链路：拿接口返回或日志里的 traceId，串起网关→服务→MQ→DB 的完整调用链，精确定位卡在哪一环。

  • 慢 SQL：慢查询日志 + explain（看 type=ALL 全表、key=NULL 没走索引、rows 巨大）。show processlist 看是否大量 Sending data 或锁等待。

    • > ⚠️ 关键取舍：连接池/线程池调大 不是免费的——把应用的池调大，压力会直接传导到 DB，要先确认 DB 扛得住，否则只是把瓶颈从应用挪到 DB。

  • TPS 平了、但 RT 也平、资源也都不满 → 典型的 "假饱和"：被某个池或 限流 卡住了，去查线程池/连接池/限流配置。

    Q29.5：向量数据库中存储什么？

    测试开发与AI_Agent面试题汇总.md
    参考答案

    答题要点：这道题可以从主问题「Q29：RAG 的完整流程」里拆开回答，重点说清楚结论、落地位置和边界。

    4. embedding 是什么：把文本转成 高维向量，语义相近的文本向量距离也近，是语义检索的基础。

    5. 向量数据库里存什么：向量 + 原文 chunk + 元数据（来源、标题、更新时间），检索命中向量后取回原文。

    6. 余弦相似度解决什么：衡量两个向量的 语义相似程度，用来从向量库里找和问题最相关的内容。

    8. rerank 为什么必要：向量召回是"粗排"，可能把相关性一般的也召回来；rerank（精排） 用更强的模型对候选重新打分排序，把最相关的排前面，显著提升质量。
